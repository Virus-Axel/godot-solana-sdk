4. CONCLUSIONS
=============================================================================

  AC-2 PASS. The `-assumenosideeffects` rule in `consumer-rules.pro`:

      -assumenosideeffects class com.godotengine.godot_solana_sdk.mwa.util.SdkLog {
          public static void v(java.lang.String, java.lang.String, kotlin.jvm.functions.Function0);
          public static void d(java.lang.String, java.lang.String, kotlin.jvm.functions.Function0);
      }

  successfully strips `SdkLog.v` and `SdkLog.d` call sites -- including the
  lambda arguments -- in the release AAR. The bytecode disassembly of
  `R8Sentinel.exerciseLogs` (§3) is the definitive proof: `getstatic+pop`
  replaces `invokestatic SdkLog.v / SdkLog.d`, and the VERBOSE/DEBUG lambda
  singletons are never instantiated because their `<clinit>` omits the
  `putstatic INSTANCE` step. The INFO lambda (`i` -- intentionally retained
  per the rule) goes through the normal `getstatic INSTANCE; invokestatic`
  path and executes at runtime.

  The grep-based verification in the original story (expected `VERBOSE=0,
  DEBUG=0, INFO=1`) turns out to measure the wrong thing: R8 does NOT remove
  the `.class` files of the stripped lambdas, so the string literals remain
  in their constant pools as unreachable dead code. The bytecode check above
  is strictly stronger than the grep.

  Ongoing enforcement: ci/verify_r8_strip.sh mechanizes the bytecode
  check as a CI gate (Gate 2a: literal-FQN grep for invokestatic to
  unminified SdkLog.v / SdkLog.d; Gate 2b: name-agnostic invokestatic
  count threshold <= 3). Gate 2a's reliability depends on
  proguard-library-self.pro's `-dontobfuscate` directive preserving
  SdkLog's compiled name in the library's own release AAR; Gate 2b
  catches the scenario where that directive is removed.
