# Story 1-6: MwaClient Kotlin Interface + FakeMwaClient + testdata/mwa-fixtures

## User Story
As an SDK contributor, I want a Kotlin `MwaClient` interface wrapping clientlib-ktx with a `FakeMwaClient` reading JSON fixtures, so that the Kotlin plugin is unit-testable without Android device and Fake Wallet APK behavior is reproducible.

## Acceptance Criteria
*(verbatim from `docs/plan.md` §Story 1-6 — do not rephrase)*

- **AC-1 (interface):** *Given* `MwaClient` interface declares suspend funs for each of the 7 MWA ops + lifecycle returning `Result<T, MwaError>`, *when* `MwaClientImpl` (wrapping `com.solanamobile.mobilewalletadapter.clientlib.MobileWalletAdapter`) is instantiated, *then* each method invokes `clientlib.transact { ... }` with the canonical identity/cluster config. *(AC prose references groupId `com.solanamobile`; actual JVM package is `com.solana.mobilewalletadapter.clientlib` — see D-2, Verified Interfaces §MobileWalletAdapter.)*
- **AC-2 (FakeMwaClient):** *Given* `FakeMwaClient` reads `testdata/mwa-fixtures/{op}_{scenario}.json`, *when* a test configures the fake with a scenario name, *then* the next op returns the canned response (success or typed `MwaError`) without any network.
- **AC-3 (fixture format):** *Given* `testdata/mwa-fixtures/` contains 20+ JSON files, *when* `tools/validate_fixtures.py` runs in CI, *then* all files validate against the documented fixture schema; PR-blocking.
- **AC-4 (error case):** *Given* a test configures `FakeMwaClient` with a non-existent scenario name, *when* the op runs, *then* an assertion error fires immediately with message `"no fixture for scenario '<name>'"`.

## Architecture Guardrails

### DD-19 [LOCKED] — MwaClient/MwaClientImpl/FakeMwaClient split
`MwaClient` abstraction consumed by plugin (Epic 2+). `MwaClientImpl` wraps clientlib-ktx. `FakeMwaClient` drives fixture-based T1 unit tests and T4 contract tier. Rationale: AC-D-4 enables JVM unit tests without Android device.

### DD-28 [LOCKED] — Shared fixture set drives both fakes
Same `testdata/mwa-fixtures/*.json` files drive `FakeMwaClient` AND the emulated Fake Wallet APK (T3). T4 contract tier runs identical suite against both; divergence is PR-blocking.

### DD-33 + NFR-10 — 0 hand-rolled protocol
`MwaClientImpl` delegates all protocol/WebSocket/ECDH work to clientlib-ktx. Any raw WebSocket, JSON-RPC framing, or ECDH code under `mwa/client/` is a violation.

- **F: `MwaClient` (interface)** — suspend funs returning `MwaResult<T>`.
- **G: `MwaClientImpl` (concrete)** — wraps `MobileWalletAdapter`, owns per-call `ConnectionIdentity` + blockchain.

### §10.3.1 — Fake/Real contract parity
T4 contract tier runs identical suite against `FakeMwaClient` and a thin adapter replaying recorded clientlib-ktx traffic.

### 7 MWA ops + lifecycle
`authorize`, `reauthorize`, `deauthorize`, `disconnect`, `signMessages`, `signTransactions`, `signAndSendTransactions` + `close()`. Per clientlib-ktx 2.0.3 canonical pattern (`transact(sender){…}` per call), `ActivityResultSender` is ephemeral — passed as first param, never stored. `close()` is a no-op for both impls (no shared state — see Design Decision "Per-call adapter" below).

## Design Decisions (resolved during pre-implementation revision)

1. **Per-call adapter construction.** Each `MwaClientImpl` method constructs a fresh `MobileWalletAdapter(identity)` inside its own body. No adapter field on `MwaClientImpl`, no shared state between calls, no concurrency race on `authToken`. Matches scaffold `MyComposable.kt:30,67,118` per-call pattern.
2. **authToken is opaque.** `AuthResult.authToken: SecretString` wraps the **UTF-8 bytes of the base64 string** the wallet/fixture returned. `MwaClientImpl` assigns via `adapter.authToken = String(authToken.reveal(), Charsets.UTF_8)` — identity transform, no decode/re-encode, no base64 encoder variant to choose.
3. **corrId is internal.** Both impls generate `val corrId = UUID.randomUUID().toString().take(8)` at method entry. No interface parameter. Tests assert format regex `^[0-9a-f]{8}$`, not value.
4. **chainId/cluster consistency.** `authorize`/`reauthorize` require both; validated via `canonicalChainIdFor(cluster)` helper (`devnet`↔`solana:devnet`, `mainnet-beta`↔`solana:mainnet`, `testnet`↔`solana:testnet`). Mismatch → `MwaResult.Failure(MwaError.ProtocolError)` **without** constructing adapter or calling clientlib. `signAndSendTransactions` keeps `cluster`-only (chainId implicit from prior authorize).
5. **public_key encoding is base64** (not base58). FakeMwaClient decodes via `java.util.Base64.getDecoder().decode(...)`. Base58 is wallet-UI convention; fixtures are test data.
6. **Test fixture directory via Gradle system property.** `build.gradle.kts` sets `systemProperty("mwa.fixtures.dir", rootDir.parentFile.resolve("testdata/mwa-fixtures").absolutePath)` on all unit tests. Tests read via `File(System.getProperty("mwa.fixtures.dir"))`. Survives CWD drift and Gradle subproject-CWD convention (`android/`).

## Fixture schema (content of `testdata/mwa-fixtures/SCHEMA.md`)

```jsonc
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["schema_version", "op", "scenario", "expected_input", "response"],
  "properties": {
    "schema_version": { "const": 1 },
    "op": { "enum": ["authorize", "reauthorize", "deauthorize", "disconnect",
                      "sign_messages", "sign_transactions", "sign_and_send"] },
    "scenario": { "type": "string", "pattern": "^[a-z0-9][a-z0-9_]*$" },
    "expected_input": { "type": "object" },
    "response": {
      "oneOf": [
        { "required": ["type", "payload"],
          "properties": {
            "type": { "const": "success" },
            "payload": { "type": "object" }
          }
        },
        { "required": ["type", "code"],
          "properties": {
            "type": { "const": "error" },
            "code": { "enum": ["USER_CANCELED", "WALLET_REJECTED", "NO_MWA_WALLET_INSTALLED",
                               "WALLET_UNINSTALLED", "TOKEN_EXPIRED", "RPC_FAILED",
                               "NETWORK_OFFLINE", "TIMEOUT", "NOT_CONNECTED",
                               "UNSUPPORTED_PLATFORM", "PROTOCOL_ERROR", "STORAGE_CORRUPT",
                               "REAUTH_REQUIRED"] }
          }
        }
      ]
    }
  },
  "additionalProperties": false
}
```

**Filename parser (LOCKED — see T5):** longest-prefix match against op enum `[sign_and_send, sign_transactions, sign_messages, reauthorize, deauthorize, disconnect, authorize]`. Scenario = chars after `{op}_`. Self-consistency: `filename.op == json.op && filename.scenario == json.scenario`.

**Scenario naming vs response.code:** Scenario names describe the situational cause; `response.code` is the protocol-level error returned. These may differ intentionally — e.g., `reauthorize_cluster_mismatch` returns `code: "NOT_CONNECTED"` (per 2-2 AC-4 cache-key forced-re-authorize semantics).

**Example (`authorize_success.json`):**
```json

  "scenario": "success",

                     "cluster":"devnet","chain_id":"solana:devnet"},
  "response": {"type":"success",
               "payload": {"auth_token":"dGVzdC1hdXRoLXRva2VuLWJhc2U2NA==",
                           "public_key":"QAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=",
                           "account_label":"Test Wallet","wallet_uri_base":"",
                           "wallet_package":"com.solanamobile.fakewallet",
                           "cluster":"devnet","chain_id":"solana:devnet"}}
}
```
