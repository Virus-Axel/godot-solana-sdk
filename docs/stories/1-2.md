# Story 1-2: SecretString + SdkLog + Scaffold Cleanup

## User Story

As an SDK maintainer, I want a redacted-by-default secret wrapper, a release-stripped logger with lambda-safe signatures, and a cleaned-up Android plugin scaffold, so that token leakage is impossible by construction and the codebase follows the host repo's conventions.

## Acceptance Criteria

The four acceptance criteria below are the authoritative, verbatim ACs for Story 1-2 (from `docs/plan.md` §Story 1-2).

- **AC-1 (happy) [amended A-7]:** Given `SecretString` Kotlin class (wrapping `ByteArray`) and C++ POD-like `SecretString` wrapping `godot::PackedByteArray`, when either is stringified via `toString()` (Kotlin) / `to_string()` (C++), then the literal string `"<redacted>"` is returned; explicit `.reveal(): ByteArray` / `.reveal_bytes()` methods are the only way to access contents. (`operator<<` dropped per A-7 — DD-26 forbids the `<ostream>` include required to provide it; plan.md's original phrasing is historical.)

- **AC-2 (SdkLog lambda):** Given `SdkLog.v/d/i/w/e` accept `(tag: String, corrId: String, msgFactory: () -> String)` signatures, when a release build runs with R8 `-assumenosideeffects` on `SdkLog.v/d`, then both the call AND the lambda body are stripped (argument evaluation does NOT run); verified by decompiling release AAR.

- **AC-3 (scaffold cleanup):** Given the existing `android/plugin/` scaffold has compileSdk=33, minSdk=21, `println()` calls, module-global `var myResult`/`myAction`/`myConnectedKey`/etc., and no ktlint config, when this story merges, then compileSdk=34, minSdk=23, `println` replaced with `SdkLog.d`, module globals replaced by a scoped `MwaSessionState` class, ktlint config added and `./gradlew ktlintCheck` green.

- **AC-4 (CI grep):** Given CI runs a grep lint, when any source file calls `Log.*(authToken)`, `Log.*(publicKey)`, `Log.*(SecretString.reveal())`, or `SdkLog.[vdiwe]\(.*\$\{` (string interpolation without lambda), then the build fails with a named violation line.

## Architecture Guardrails

### DD-14 [LOCKED] — `SecretString` wrapper

From `docs/architecture.md:1166` (amended A-7): "`SecretString` wrapper type in both Kotlin and C++ with `toString()` (Kotlin) / `to_string()` (C++) → `"<redacted>"`; tokens never unwrap into `String`-typed fields. (See A-7 — `operator<<` dropped per DD-26.) Rationale: AC-D-8, AC-NFR-SEC-6; defense in depth against accidental `Log.*` leakage."

From §2.2 component K (`docs/architecture.md:83`): "Wrapper whose `toString()` returns `"<redacted>"`; prevents accidental token leakage to `Log.*` or stack traces | `fun value(): ByteArray` (caller opts-in)".

**Implementation drift from the architecture text:** the plan fixes the accessor name as `reveal()` / `reveal_bytes()` (not `value()`). The plan is authoritative for this story because `reveal()` reads as a deliberate, auditable opt-in; every consumer call-site is grep-able for `SecretString.reveal()` which AC-4's CI grep leverages as one of its four banned patterns (AC-D-8). The architecture doc's `value()` phrasing is descriptive, not load-bearing. **Use `reveal()` / `reveal_bytes()`.**

### DD-26 [LOCKED] — C++ `SecretString` uses only `godot::` types

From `docs/architecture.md:802` (amended A-7): "`SecretString` on the C++ side is a POD-like wrapper holding engine-allocated `PackedByteArray` storage, with `to_string()` returning the literal `"<redacted>"` (see A-7 — `operator<<` dropped because it would require `<ostream>` and violate DD-26). No STL internals."

From `docs/architecture.md:798-800` (§7.1.1 ABI Hygiene): "Public C++ headers of the GDExtension boundary use **only** `godot::` types — `godot::Variant`, `godot::String`, `godot::Dictionary`, `godot::PackedByteArray`, `godot::Ref<T>`. No `std::string`, `std::vector`, `std::unique_ptr`, or other STL across the GDExtension ABI."

→ C++ `SecretString` ships in `src/mwa/include/secret_string.hpp`. The header must reference only `godot::` types. The header-lint CI job (scheduled to land in Story 1-4) will later scan `src/mwa/include/` for STL leaks; this story must not plant any.

### DD-30 [LOCKED] — `SdkLog` signatures

From `docs/architecture.md:1203`: "`SdkLog.v/d/i/w/e` signatures take `() -> String` lambdas; CI lint bans string-interpolated calls. Rationale: architect review #9 (R8 argument-eval foot-gun)."

From §5.5 Logging Signatures (`docs/architecture.md:649-656`):

```kotlin
SdkLog.d(TAG, corrId) { "token = $token" }  // argument evaluation stripped by R8 in release
// NOT: SdkLog.d(TAG, corrId, "token = $token")  // banned by CI lint
```

"A CI grep rejects `SdkLog\.(v|d|i|w|e)\([^{]*\$\{` (string interpolation without lambda). This prevents the classic R8 foot-gun where `-assumenosideeffects` strips the call but argument evaluation still runs."

### §8.5 — R8 rules (authoritative excerpt, amended A-6)

From `docs/architecture.md:1023-1030` (amended A-6) — the `-assumenosideeffects` + `-keep` rules that this story must ship in `android/plugin/consumer-rules.pro`:

```
# Strip verbose/debug logs in release (package corrected — see A-6)
-assumenosideeffects class com.godotengine.godot_solana_sdk.mwa.util.SdkLog {
    public static void v(...);
    public static void d(...);
}

# Keep Story 1-2 Task 6 R8-strip evidence sentinel from DCE (see A-6)
-keep class com.godotengine.godot_solana_sdk.mwa.util.R8Sentinel { *; }
```

Two correctness dependencies to match the rule exactly:

1. **Package path (A-6):** `SdkLog` must live at `com.godotengine.godot_solana_sdk.mwa.util.SdkLog`. ProGuard/R8 class matchers in `-assumenosideeffects` are exact strings — the pre-amendment rule `...mwa.SdkLog` (no `util.`) silently matches nothing and R8 does not strip anything. Story plants `SdkLog.kt` at `android/plugin/src/main/kotlin/com/godotengine/godot_solana_sdk/mwa/util/SdkLog.kt` (A-5 subpackage convention) — this matches the amended rule.
2. **Method descriptor:** Rule targets `public static void v(...)` / `d(...)`. `SdkLog` must be `object SdkLog` AND every member carries `@JvmStatic` — without `@JvmStatic`, Kotlin emits `SdkLog.INSTANCE.v(...)` (instance dispatch on the singleton) which doesn't match `public static void v(...)`. Plan wording: `object SdkLog`.

The `-keep` on `R8Sentinel` is non-negotiable for Task 6 evidence: `R8Sentinel.exerciseLogs()` is never called from production code, so R8's default DCE pass removes the class wholesale. Without the keep, the decompile grep returns `VERBOSE=0, DEBUG=0, INFO=0` and Task 6 cannot distinguish "R8 correctly stripped v/d" from "R8 DCE'd everything." The keep rule preserves `R8Sentinel.class` in the AAR so the `INFO=1` discriminator exists.

## Verified Interfaces

This story **defines** three new primitives. No prior implementation exists for any of them in `godot-solana-sdk` (scanned on 2026-04-17). All entries are `DEFINES-NEW`; the interface-audit step in `/implement` skips audit for these on first use.

### Kotlin `SecretString` (DEFINES-NEW)
- **Source (to be created):** `android/plugin/src/main/kotlin/com/godotengine/godot_solana_sdk/mwa/util/SecretString.kt`
- **Signature:**
  ```kotlin
  package com.godotengine.godot_solana_sdk.mwa.util

  class SecretString(private val bytes: ByteArray) {
      fun reveal(): ByteArray = bytes.copyOf()  // defensive copy — caller cannot mutate internal state
      override fun toString(): String = "<redacted>"
      override fun equals(other: Any?): Boolean = other is SecretString && bytes.contentEquals(other.bytes)
      override fun hashCode(): Int = bytes.contentHashCode()
  }
  ```
- **Plan match:** DEFINES-NEW — using plan contract + architecture DD-14 guardrails.
- **Accessor name:** `reveal()` (plan-authoritative; see DD-14 discussion above).
- **Defensive copy rationale:** without `copyOf()`, a caller can hold a reference to the internal `bytes` array and mutate it post-construction, breaking the immutability invariant that `toString()` depends on (and leaking across equality checks in caches).

### C++ `SecretString` (DEFINES-NEW)
- **Source (to be created):** `src/mwa/include/secret_string.hpp` (+ `src/mwa/secret_string.cpp` if needed; header-only preferred since no non-trivial logic)
- **Signature:**
  ```cpp
  // src/mwa/include/secret_string.hpp
  #ifndef GODOT_SOLANA_SDK_MWA_SECRET_STRING_HPP
  #define GODOT_SOLANA_SDK_MWA_SECRET_STRING_HPP

  #include <godot_cpp/variant/packed_byte_array.hpp>
  #include <godot_cpp/variant/string.hpp>

  namespace godot_solana_sdk::mwa {

  class SecretString {
  public:
      explicit SecretString(godot::PackedByteArray bytes) : bytes_(bytes) {}
      godot::PackedByteArray reveal_bytes() const { return bytes_; }   // Godot PackedByteArray is CoW — a copy here is cheap
      godot::String to_string() const { return godot::String("<redacted>"); }

  private:
      godot::PackedByteArray bytes_;
  };

  }  // namespace godot_solana_sdk::mwa

  // godot::String concatenation convenience (engine types only, DD-26-compliant). Never exposes bytes_.
  inline godot::String operator+(const godot::String& prefix, const godot_solana_sdk::mwa::SecretString& s) {
      return prefix + s.to_string();
  }

  #endif  // GODOT_SOLANA_SDK_MWA_SECRET_STRING_HPP
  ```
- **Plan match:** DEFINES-NEW — using plan contract (amended AC-1 per A-7) + DD-26 (godot:: only) guardrails.
- **Namespace:** `godot_solana_sdk::mwa` — matches existing `src/` convention for the SDK; the header-lint CI (future Story 1-4) will not complain.
- **`operator<<` intentionally absent (A-7):** Amendment A-7 drops `operator<<` from AC-1. `std::ostream& operator<<` would require `<ostream>` and drag STL into the public header, violating DD-26; Godot's `godot::String` has no chainable `operator<<` idiom. `to_string()` is the sole stringification path. The GoogleTest assertion for AC-1 uses `.to_string()` directly: `EXPECT_EQ(s.to_string(), godot::String("<redacted>"))`. Downstream token-handling stories (2-1+) that concatenate `SecretString` into log strings use the `operator+(godot::String, SecretString)` overload above.

### Kotlin `SdkLog` (DEFINES-NEW)
- **Source (to be created):** `android/plugin/src/main/kotlin/com/godotengine/godot_solana_sdk/mwa/util/SdkLog.kt`
- **Signature:**
  ```kotlin
  package com.godotengine.godot_solana_sdk.mwa.util

  import android.util.Log

  object SdkLog {
      // In test/JVM builds (where android.util.Log is stubbed to no-op), this wrapper still evaluates the
      // lambda only when the level is enabled. In release Android, R8 -assumenosideeffects strips v/d
      // call sites + lambda bodies entirely (verified in Task 6 via AAR decompile).

      @JvmStatic fun v(tag: String, corrId: String, msg: () -> String) { Log.v(tag, "[corrId=$corrId] ${msg()}") }
      @JvmStatic fun d(tag: String, corrId: String, msg: () -> String) { Log.d(tag, "[corrId=$corrId] ${msg()}") }
      @JvmStatic fun i(tag: String, corrId: String, msg: () -> String) { Log.i(tag, "[corrId=$corrId] ${msg()}") }
      @JvmStatic fun w(tag: String, corrId: String, msg: () -> String) { Log.w(tag, "[corrId=$corrId] ${msg()}") }
      @JvmStatic fun e(tag: String, corrId: String, msg: () -> String) { Log.e(tag, "[corrId=$corrId] ${msg()}") }
  }
  ```
- **Plan match:** DEFINES-NEW — plan contract + DD-30 + §8.5 R8 rule alignment.
- **Why `@JvmStatic`:** the R8 `-assumenosideeffects` rule in §8.5 is written against `public static void v(...)` / `d(...)` method descriptors. Kotlin `object` members compile to instance methods on the singleton by default; `@JvmStatic` generates additional static-dispatch methods that match `public static void v(...)` byte-for-byte. Without `@JvmStatic`, R8 cannot match the rule. **Verified during Task 6**: if the decompiled release AAR still shows `SdkLog.INSTANCE.v(...)` invocations, `@JvmStatic` is missing or on the wrong members.
- **Log-line format:** `"[corrId=<corrId>] <message>"` — matches §7.2 Correlation IDs (`docs/architecture.md:810`): "Logged by `SdkLog` at each layer: `tag="MWA", format="[corrId=xxxx] message"`."

- **Signature:** single class with private mutable fields per the table under "§2.2 Component I — `MwaSessionState`" above.

- **Not a seam:** `MwaSessionState` is internal plugin state consumed only by `GDExtensionAndroidPlugin` (Kotlin) — no Story 1-4+ consumer lists this class as an interface contract. ### Why selective `@JvmStatic` AND the `util.` package must both match the R8 rule

Task 3's `SdkLog.kt` ships two correctness dependencies that the R8 rule is allergic to — both must be right or release builds silently retain verbose/debug log call sites. The diagnostic symptom is identical for both bugs (`MWA_R8_SENTINEL_VERBOSE_*` strings appear in the decompiled release class), so during Task 6 review the order of investigation matters:

1. **Package path matches the rule (A-6).** The amended rule targets `com.godotengine.godot_solana_sdk.mwa.util.SdkLog`. If `SdkLog.kt` accidentally lands at `com.godotengine.godot_solana_sdk.mwa.SdkLog` (no `util.`) or any other subpackage, ProGuard/R8's exact-name matcher finds nothing and strips nothing. Check first because the fix is in the file path / package declaration, not in bytecode dispatch. Grep proof: `find android/plugin/src/main/kotlin -name SdkLog.kt` must return exactly `.../com/godotengine/godot_solana_sdk/mwa/util/SdkLog.kt`.
2. **Method descriptor matches `public static void v(...)`.** `@JvmStatic` on every `v/d/i/w/e` member. Without it, Kotlin compiles `object SdkLog { fun v(...) }` to `SdkLog.INSTANCE.v(...)` (instance method on the singleton) and R8 cannot match. Check second because the fix is a Kotlin annotation.

Pre-A-6, this section listed `@JvmStatic` as the primary cause. Post-A-6, the package mismatch is #1 and `@JvmStatic` is #2 — a broken package makes every `@JvmStatic` ceremony invisible to R8.

### Why `println` → `SdkLog.d` (and not `Log.v` per spec wording)

Spec AC-D-12 (`docs/spec.md:163`) says: "`println`→`Log.v(TAG, …)`". Architecture DD-30 + plan §Story 1-2 AC-3 say `SdkLog.d`. **Architecture is authoritative** when it postdates spec with a LOCKED decision. DD-30 was added in architect review #9 (post-spec) and explicitly replaces raw `Log.*` with `SdkLog.*` for the R8 strip + corrId reasons. Use `SdkLog.d`.

## Risks

### R1 — R8 `-assumenosideeffects` doesn't strip the lambda body
**Likelihood:** Medium. **Impact:** AC-2 fail (block merge).
**Mitigation:** Task 6's sentinel-string procedure is the sole authoritative check. If sentinels still appear in the decompiled AAR, investigate causes in this order (post-A-6 ordering — package mismatch outranks `@JvmStatic` because it's the easier silent failure):

1. **Package mismatch (A-6).** `SdkLog.kt` must be at `com.godotengine.godot_solana_sdk.mwa.util.SdkLog` — matching the amended rule `-assumenosideeffects class com.godotengine.godot_solana_sdk.mwa.util.SdkLog`. A drift to `...mwa.SdkLog` (no `util.`), `...mwa.log.SdkLog`, or any other subpackage silently matches nothing. Verify with `grep -n 'package com.godotengine.godot_solana_sdk.mwa' android/plugin/src/main/kotlin/**/SdkLog.kt` — must show `package com.godotengine.godot_solana_sdk.mwa.util`.
2. **`R8Sentinel` DCE'd (A-6).** If `INFO=0` in the decompile grep, the `-keep class …util.R8Sentinel { *; }` rule is missing from `consumer-rules.pro` — R8 eliminated the whole sentinel class. Fix = add the keep line. Without it, `VERBOSE=0, DEBUG=0, INFO=0` reports green by count but proves nothing about the strip.
3. **`@JvmStatic` missing on `SdkLog.v/d`** — Kotlin dispatch goes through `SdkLog.INSTANCE.v(...)` which doesn't match R8's `public static void v(...)` rule. Fix = add `@JvmStatic` to every `v/d/i/w/e` member.
4. **Caller bypasses the lambda signature.** `SdkLog.v(tag, corrId, "literal")` with a concrete `String` arg instead of a `() -> String` lambda — this is actually NOT a problem for strip correctness (R8 strips the call regardless) but it IS a problem for NFR-1 because the concrete `String` was constructed before the call and may persist in logs. Task 7's grep #4 catches this pattern at the source (banned pattern: `SdkLog\.[vdiwe]\([^{]*\$\{`).
