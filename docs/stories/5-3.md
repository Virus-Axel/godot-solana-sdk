# Story 5-3: Lifecycle Observer + mwa_cancelled_lifecycle + Rotation Test

## User Story
*(verbatim from `docs/plan.md:392` — do not rephrase)*

As a Godot game developer, I want in-flight MWA ops to emit a deterministic cancellation signal when Android destroys my Activity (e.g., on device rotation), so that my UI doesn't hang waiting for a signal that will never arrive.

## Context

- **Story 4-2 is the FIRST `mwa_cancelled_lifecycle` emitter; Story 5-3 is the SECOND.** Story 4-2 (commit `f91f682b`) shipped the cancel-in-flight loop in `mwaForgetAll` at `GDExtensionAndroidPlugin.kt:1405-1488` — `inflightMap.snapshot()` enumerates `(requestId → sourceMethod)` slots, then `for ((reqId, srcMethod) in snapshot) { if (inflightMap.tryTerminate(reqId)) { nativeBridge.postMwaCancelledLifecycle(buildCancelledLifecycleJson(reqId, srcMethod, "forget_all_invoked").toString()) } else { diagnostics.incrementLateResult() } }` — with the CAS-on-`tryTerminate` claim guarding against double-terminal-signal emission. Story 5-3's lifecycle observer reuses **the EXACT SAME emit pattern** but with `reason: "activity_destroyed"` (per AC-1 spec text at `plan.md:401`) instead of `"forget_all_invoked"`. **No new helper, no new signal, no new payload schema** — just a second invocation site for `inflightMap.snapshot() + tryTerminate-CAS + buildCancelledLifecycleJson + postMwaCancelledLifecycle`.

## Acceptance Criteria
*(verbatim from `docs/plan.md:401-403` — do not rephrase)*

- **AC-1 (happy):** *Given* `MwaAndroidPlugin` registers `DefaultLifecycleObserver` on the Godot `Activity`, *when* the Activity emits `Lifecycle.Event.ON_DESTROY` with an in-flight `connect` op, *then* the op's coroutine is cancelled AND `mwa_cancelled_lifecycle{request_id, source_method:"connect", reason:"activity_destroyed"}` fires (terminal signal, per DD-23).
  - **Evidence path:** Kotlin unit `MwaLifecycleObserverTest.AC-1 ON_DESTROY with in-flight connect emits mwa_cancelled_lifecycle{reason:"activity_destroyed"}` (T1) + `MwaAndroidPluginSignAndSendTest.AC-4c breadcrumb cleared on lifecycle cancellation` (T2 un-disables; closes CR-3-3-B).
- **AC-2 (rotation instrumented test):** *Given* an `androidTest` that calls `MWA.connect(...)` and triggers a configuration change (rotation) mid-flight via `ActivityScenario.recreate()`, *when* the test runs, *then* the test observes `mwa_cancelled_lifecycle` within 1 s AND no further signal fires for that `request_id`.
  - **Evidence path:** androidTest `AC2RotationCancelsInflightE2ETest.kt` (T3) using `ActivityScenario.launch(...) + scenario.recreate()` + asserting `nativeBridge.postMwaCancelledLifecycle` was invoked exactly once and no `postConnectCompleted` / `postMwaError` / `postMwaTimeout` followed for the same `request_id` within a bounded wait window.
- **AC-3 (no double terminal):** *Given* a connect already completed successfully before rotation, *when* rotation fires ON_DESTROY, *then* NO `mwa_cancelled_lifecycle` is emitted for the already-terminated `request_id`.
  - **Evidence path:** Kotlin unit `MwaLifecycleObserverTest.AC-3 already-terminated requestId not re-emitted on ON_DESTROY` (T1) — pre-register an op then `tryTerminate` it, then trigger the observer's onDestroy; assert `postMwaCancelledLifecycle` is NEVER invoked (CAS-on-`tryTerminate` returns `false` per DD-5-3-3 inheritance).

## Architecture Guardrails

5 LOCKED design decisions covering: lifecycle observer file/class layout, registration timing, no-double-terminal CAS reuse, reason literal value, and breadcrumb-cleanup wiring scope.

- **DD-5-3-1 (LOCKED) — Lifecycle observer is a SEPARATE file `MwaLifecycleObserver.kt`, NOT inlined into `GDExtensionAndroidPlugin.kt`.** Rationale: (a) `GDExtensionAndroidPlugin.kt` is already 2484 LOC; inlining would push it over the 2500 LOC code-smell threshold (per global rules `code-standards.md` §3 file >800 lines is a code smell). (b) The observer's responsibility is independent — it owns "Activity lifecycle → cancel in-flight" while the plugin owns "JNI dispatch + op coordination." Separation respects single-responsibility. (c) The observer needs **no `@UsedByGodot` reflection surface** (it's invoked by Android's lifecycle dispatcher, not Godot's GDScript-side method-name resolution) so A-10's `@UsedByGodot`-keep rule does NOT apply; an explicit minimal `-keepclassmembers` rule for `onDestroy` is added to `consumer-rules.pro` as part of T2 to defend against R8 inlining.
  - **File path:** `android/plugin/src/main/kotlin/com/godotengine/godot_solana_sdk/mwa/plugin/MwaLifecycleObserver.kt` (matches `InflightMap.kt` + `NativeBridge.kt` package layout — Story 4-2 inheritance).
  - **Class shape:**
    ```kotlin
    internal class MwaLifecycleObserver(
        private val inflightMap: InflightMap,
        private val nativeBridge: NativeBridge,
        private val diagnostics: MwaDiagnostics,
        private val cleanupBreadcrumb: (requestId: String) -> Unit,
        private val cancelInFlight: () -> Unit,
    ) : DefaultLifecycleObserver {
        override fun onDestroy(owner: LifecycleOwner) {
            // Step 1: snapshot in-flight, claim each via tryTerminate-CAS, emit
            //         mwa_cancelled_lifecycle per slot with reason="activity_destroyed".
            // Step 2: invoke cleanupBreadcrumb(requestId) per slot (no-op for
            //         non-sign_and_send ops via existing helper's try/catch).
            // Step 3: cancelInFlight() — caller-provided lambda that cancels
            //         the plugin's coroutine scope so suspending op bodies abort.
        }
    }
    ```
  - **Rejected alternative:** inline as `private inner class` inside `GDExtensionAndroidPlugin` — couples observer lifetime to plugin instance, blocks unit-tier mocking via constructor injection, and pushes the file over the size threshold.

- **DD-5-3-2 (LOCKED) — Registration timing: at plugin construction (init block / GodotPlugin onMainCreate hook), unregister on plugin teardown via DefaultLifecycleObserver.onDestroy (NO explicit removeObserver).** Rationale:
  1. **`DefaultLifecycleObserver` is auto-removed on `ON_DESTROY`** per `androidx.lifecycle:lifecycle-runtime` contract — once `onDestroy` fires, the observer is unregistered from the lifecycle. We do NOT need to call `lifecycle.removeObserver(observer)` in plugin teardown — the same `ON_DESTROY` event that triggers our cancel-loop ALSO unregisters the observer from the (now-destroyed) Activity's lifecycle.
  2. **Registration site:** in `GDExtensionAndroidPlugin.init { ... }` (the existing init block at line 72-78 already runs after `instance = this`). After `instance = this`, the observer is constructed and registered via `(godot.getActivity() as? LifecycleOwner)?.lifecycle?.addObserver(observer)`. If `getActivity()` returns null (plugin loaded outside Activity context — possible during Robolectric or non-standard test setup), the registration is a no-op + `SdkLog.w(TAG, "5-3") { "lifecycle_observer_register_skipped: activity unavailable" }` is logged.
  3. **Re-registration on Activity recreation (AC-2 rotation case):** when `ActivityScenario.recreate()` fires `ON_DESTROY` on the OLD Activity, the observer detaches with it. The NEW Activity's lifecycle has NO observer. **DD-5-3-2 explicitly accepts this** — the in-flight slots from BEFORE rotation are correctly cancelled on the OLD Activity's `ON_DESTROY`; ops started AFTER rotation belong to the NEW Activity's lifecycle and need a fresh observer registration. Per the GodotPlugin lifecycle, the plugin instance survives Activity recreation (it's tied to the Godot engine, not the Android Activity); a new in-flight op on the NEW Activity registers a new observer via the same init-block-equivalent code path. **CR-5-3-A tracks** the "fresh registration on Activity recreation" question — this story does NOT add cross-recreation observer continuity; it accepts the per-Activity-instance scope and verifies AC-2 within a SINGLE recreate cycle (the OLD Activity's ON_DESTROY is what AC-2 asserts on).
  - **Rejected alternative:** register on `onMainResume` / unregister on `onMainPause` (the GodotPlugin lifecycle hooks at `org.godotengine.godot.plugin.GodotPlugin`) — too coarse: rotation triggers pause→destroy→create→resume, which would unregister + re-register the observer multiple times mid-rotation; the destroy event would be observed by an unregistered observer on the OLD Activity. Lifecycle-API-native registration (via `lifecycle.addObserver`) handles the lifecycle-event delivery directly without proxying through GodotPlugin pause/resume hooks.
  - **Rejected alternative:** use `ProcessLifecycleOwner.get().lifecycle` — that's process-level, not Activity-level; rotation does NOT fire process-level ON_DESTROY. Wrong scope.

- **DD-5-3-3 (LOCKED) — AC-3 no-double-terminal: reuse Story 4-2 CAS-on-`tryTerminate` pattern verbatim.** The observer's onDestroy body is a near-mechanical copy of `mwaForgetAll`'s cancel-loop (GDExtensionAndroidPlugin.kt:1408-1421):
  ```kotlin
  override fun onDestroy(owner: LifecycleOwner) {
      val snapshot = inflightMap.snapshot()
      for ((reqId, srcMethod) in snapshot) {
          if (inflightMap.tryTerminate(reqId)) {
              nativeBridge.postMwaCancelledLifecycle(
                  buildCancelledLifecycleJson(reqId, srcMethod, "activity_destroyed").toString(),
              )
              cleanupBreadcrumb(reqId)
          } else {
              diagnostics.incrementLateResult()
          }
      }
      cancelInFlight()  // cancels plugin's coroutine scope; suspending op bodies see CancellationException + abort
  }
  ```
  - `tryTerminate` is atomic (`ConcurrentHashMap.remove` returns null if absent) — already-terminated requestIds (e.g., a `connect_completed` already fired before rotation) return `false`, the emit is skipped, and `diagnostics.incrementLateResult()` increments. **AC-3 is satisfied automatically** by the inherited shape — no new code path.
  - The `buildCancelledLifecycleJson(requestId, sourceMethod, reason)` helper at `GDExtensionAndroidPlugin.kt:1505` is `private` today; T2 widens it to `internal` so `MwaLifecycleObserver` (in the same package via the kotlin/ source path) can call it. This is the ONLY visibility change to the plugin file.
  - **Rejected alternative:** copy-paste the JSONObject construction inline into the observer — duplicates the 3-key-payload contract and creates a divergence vector if A-12 ever adds a 4th key. Single-source-of-truth via the existing helper.

- **DD-5-3-4 (LOCKED) — `reason` literal is `"activity_destroyed"`** (exact string per AC-1 spec text at `plan.md:401`). NOT `"lifecycle_teardown"` (Story 4-2's kdoc speculation at GDExtensionAndroidPlugin.kt:1497-1498 is **superseded** — Story 5-3 owns the literal and AC-1 dictates it). Story 4-2's `reason: "forget_all_invoked"` literal at GDExtensionAndroidPlugin.kt:1416 is unchanged.
  - The `reason` field is a **fixed-vocabulary enum string** per A-12 + Story 4-2's kdoc (line 1497-1500). Future emitters MUST file an amendment if they introduce a new reason literal. Today's vocabulary: `{"forget_all_invoked", "activity_destroyed"}` (2 values).
  - **GDScript consumer guidance** (lands in MWA.gd kdoc only — no code change for 5-3): switch on `payload["reason"]` to differentiate the cancellation source. Game UIs typically show different prompts for "you logged out" (`forget_all_invoked`) vs "screen rotated mid-op, please retry" (`activity_destroyed`).

- **DD-5-3-5 (LOCKED) — `cleanupBreadcrumb(requestId)` is invoked for EVERY snapshotted requestId, NOT just sign_and_send slots.** Rationale: the existing `cleanupBreadcrumb` helper at `GDExtensionAndroidPlugin.kt:1797` is a try/catch around `removePendingSubmission(requestId)`. For non-sign_and_send ops, `removePendingSubmission` is a no-op against an absent key (the breadcrumb entry never existed) — the `try` block silently succeeds. Filtering by `op_type == "sign_and_send"` requires re-reading `InflightEntry.sourceMethod` and pattern-matching on `"sign_and_send"` — adds a branch + ties the observer to the SigningOp.SIGN_AND_SEND.sourceMethod literal. The unconditional invocation is cheaper and idempotent.
  - **AC-4c (CR-3-3-B test) closure path:** the un-disabled test at `MwaAndroidPluginSignAndSendTest.kt:462` pre-seeds a breadcrumb via `storedPending["sas-ac4c"] = ...`, registers an in-flight `sign_and_send` slot, fires the observer's onDestroy, asserts `postMwaCancelledLifecycle` was invoked once AND `store.removePendingSubmission("sas-ac4c")` was invoked exactly once AND `storedPending["sas-ac4c"] == null`.
  - **`cleanupBreadcrumb` runs on the `Dispatchers.Default` thread of the observer's onDestroy invocation.** The observer's `onDestroy` is called by the Android main thread (Activity lifecycle dispatch); `removePendingSubmission` writes to `EncryptedSharedPreferences.edit().remove(key).apply()` (`apply` is async) — safe on main thread per the existing helper's design. No dispatcher hop required.
  - **Rejected alternative:** filter by `op_type` and only invoke `cleanupBreadcrumb` for sign_and_send slots — adds branch complexity for zero observable benefit (the helper is a no-op for absent keys). KISS.

## Concerns

- **CR-5-3-A (LOW, cross-Activity-instance observer continuity):** the lifecycle observer registered in the plugin's init-block is auto-removed on `ON_DESTROY` per the `androidx.lifecycle` contract. Plugin instance survives Activity recreation (it's tied to the Godot engine, not the Android Activity); the NEW post-recreate Activity has NO observer. AC-2 verifies the OLD Activity's `ON_DESTROY` only — in-flight ops started on the NEW Activity post-rotation would not be observed. **Acceptable today** because (a) AC-1 + AC-2 + AC-3 do not require post-recreate continuity, (b) the typical game-side mitigation is to NOT start ops mid-rotation. Mitigation candidate: add an `onMainResume()` GodotPlugin hook that re-invokes `registerLifecycleObserver()` per-Activity. Out of 5-3 scope; tracked for Story 5-4 / 5-6 triage.
- **CR-5-3-B (LOW, ActivityScenario.recreate timing on slow CI emulators):** the AC-2 androidTest asserts within a 1s budget post-`recreate()`. macos-arm64 CI runners (emulator-on-rosetta) occasionally exceed 1s for `ON_DESTROY` dispatch. Mitigation deferred to Story 5-6's flake-baseline triage; T3's `awaitCondition` honors the 1s spec budget today. If CR-5-3-B materializes as a flake, the fix is to widen the budget to 2-3s (still within reasonable lifecycle-dispatch latency expectations) AND file an amendment if the AC-2 spec text needs updating.
- **CR-5-3-C (LOW, observer leak risk on plugin re-init):** if Godot re-instantiates the plugin (uncommon but possible under test runners that reload the class — see GDExtensionAndroidPlugin.kt:74-76 `Last-writer wins on re-init`), the OLD plugin instance's observer is still registered on its bound Activity's lifecycle. The OLD plugin's `inflightMap` is unchanged after re-init (Companion `instance` is updated, but the OLD plugin's collaborators are still GC-rooted via the registered observer). **Mitigation:** the plugin's init-block could call `lifecycle.removeObserver(prevObserver)` on re-init — but this requires retaining a handle to the previous observer + an Activity reference, both of which contradict the LifecycleOwner-auto-cleanup design. Acceptable today because plugin re-init is a test-only path; production callers always have one plugin instance per process. Tracked for retrospect / forward planning.
