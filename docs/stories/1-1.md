# Story 1-1: Error-Codes Manifest + 3-Language Codegen + Generator Tests

## User Story

As an SDK contributor, I want a single source of truth for the 13 MWA error codes that regenerates Kotlin + C++ + GDScript constants, so that enum drift across layers is impossible.

## Acceptance Criteria

The three acceptance criteria below are the authoritative, verbatim ACs for Story 1-1.

- **AC-1 (happy):** Given `tools/error-codes.yaml` defines 13 codes with `cpp_enum`, `kotlin_enum`, `recoverable`, `retry_hint`, `default_user_message`, when `python tools/gen_error_codes.py` runs, then `android/plugin/src/generated/.../MwaError.kt` contains a sealed class with 13 subclasses, `src/generated/mwa_error_codes.hpp` contains `enum class MwaErrorCode` with 13 members + `code_name()` function, and `addons/SolanaSDK/mwa_error_codes.gd` contains 13 string constants; all 3 files regenerate byte-identically on repeat invocation.

- **AC-2 (drift check):** Given the generated files are committed and `error-codes.yaml` is modified, when the CI `codegen_drift_check` job runs, then the job fails the build with a diff showing the drift.

- **AC-3 (generator tests):** Given the generator Python script, when `pytest tools/test_gen_error_codes.py` runs, then 3 golden-file fixtures assert byte-identical output; property-based tests round-trip random valid manifests and cross-verify `recoverable`/`retry_hint` across all 3 artifacts; malformed YAML produces a specific error message; test suite is PR-blocking independent of drift check.

## Architecture Guardrails

### DD-13 [LOCKED] — Single Source of Truth

Single `tools/error-codes.yaml` manifest → codegen for Kotlin enum + C++ enum + GDScript constants. Rationale: AC-A-3, AC-D-2; single source of truth; CI drift check enforces that committed artifacts match the manifest.

### DD-33 [LOCKED] — Generator Test Independence

Error-codes generator ships with golden-file + property tests + failure tests; these tests are PR-blocking **independent** of the drift-check job. Rationale: drift check catches committed drift but not generator bugs. Two distinct CI jobs must both pass.

### §3.3 — Error Taxonomy Manifest (error-codes.yaml) — v1 Schema

```yaml
schema_version: 1
codes:
  - name: <UPPER_SNAKE, unique>
    cpp_enum: <identifier used in C++ enum class MwaErrorCode>
    kotlin_enum: <PascalCase identifier used as sealed-class subclass name>
    recoverable: <bool>
    retry_hint: <one of: retry_now | install_wallet | connect | none>
    default_user_message: <string, human-readable>
```

The 13 authoritative entries (must appear in this exact order):

| # | name | cpp_enum | kotlin_enum | recoverable | retry_hint | default_user_message |
|---|------|----------|-------------|-------------|------------|----------------------|
| 1 | USER_CANCELED | USER_CANCELED | UserCanceled | true | retry_now | "You cancelled the wallet prompt." |
| 2 | WALLET_REJECTED | WALLET_REJECTED | WalletRejected | true | retry_now | "The wallet rejected the request." |
| 3 | NO_MWA_WALLET_INSTALLED | NO_MWA_WALLET_INSTALLED | NoMwaWalletInstalled | true | install_wallet | "No Solana wallet installed. Install one to continue." |
| 4 | WALLET_UNINSTALLED | WALLET_UNINSTALLED | WalletUninstalled | true | connect | "The previously connected wallet is no longer installed." |
| 5 | TOKEN_EXPIRED | TOKEN_EXPIRED | TokenExpired | true | connect | "Your wallet session has expired. Please reconnect." |
| 6 | RPC_FAILED | RPC_FAILED | RpcFailed | true | retry_now | "The Solana network rejected the transaction." |
| 7 | NETWORK_OFFLINE | NETWORK_OFFLINE | NetworkOffline | true | retry_now | "No internet connection." |
| 8 | TIMEOUT | TIMEOUT | Timeout | true | retry_now | "The operation timed out." |
| 9 | NOT_CONNECTED | NOT_CONNECTED | NotConnected | true | connect | "Not connected to a wallet. Call MWA.connect() first." |
| 10 | UNSUPPORTED_PLATFORM | UNSUPPORTED_PLATFORM | UnsupportedPlatform | false | none | "MWA is only available on Android." |
| 11 | PROTOCOL_ERROR | PROTOCOL_ERROR | ProtocolError | false | connect | "Wallet communication failed." |
| 12 | STORAGE_CORRUPT | STORAGE_CORRUPT | StorageCorrupt | true | connect | "Secure storage is unavailable. Please reconnect." |
| 13 | REAUTH_REQUIRED | REAUTH_REQUIRED | ReauthRequired | true | connect | "Your session needs renewal." |

**Schema invariants (validator must enforce):**
- `schema_version == 1`
- `codes` is a list of length 13 (for v1 manifest; generator accepts any N ≥ 1 for test manifests)
- Every code has all six keys: `name`, `cpp_enum`, `kotlin_enum`, `recoverable`, `retry_hint`, `default_user_message`
- `name`, `cpp_enum` are UPPER_SNAKE_CASE ASCII identifiers (`^[A-Z][A-Z0-9_]*$`)
- `kotlin_enum` is PascalCase ASCII (`^[A-Z][A-Za-z0-9]*$`)
- `recoverable` is a Python `bool` (not "true"/"false" strings)
- `retry_hint` ∈ `{retry_now, install_wallet, connect, none}`
- `default_user_message` is a non-empty string
- All `name` values are unique within the manifest; same for `cpp_enum`, `kotlin_enum`

### Codegen Output Targets (§3.3)

Three artifacts written by `tools/gen_error_codes.py`:

1. **C++ header** — `src/generated/mwa_error_codes.hpp`
   - Include guard: `GODOT_SOLANA_SDK_MWA_ERROR_CODES_HPP` (matches existing `tools/json_to_header.py` convention).
   - Top comment: `// This file is generated, do not edit.` (matches existing tools).
   - `enum class MwaErrorCode : int` with 13 members in manifest order, starting at 0.
   - `const char* code_name(MwaErrorCode e)` inline function returning the UPPER_SNAKE `name` string.

2. **Kotlin sealed class** — `android/plugin/src/generated/kotlin/com/limechain/mwa/generated/MwaError.kt`
   - Package: `com.limechain.mwa.generated`.
   - Top comment: `// This file is generated, do not edit.`
   - `sealed class MwaError(val code: String, val recoverable: Boolean, val retryHint: String, val defaultMessage: String)` with 13 `object` subclasses (one per `kotlin_enum`).

3. **GDScript constants** — `addons/SolanaSDK/mwa_error_codes.gd`
   - Top comment: `# This file is generated, do not edit.`
   - `class_name MwaErrorCodes` with 13 `const MWA_ERR_<name> := "<name>"` entries, plus a `const ALL_CODES := [...]` array in manifest order.

### §10.3.2 — Codegen Generator Test Strategy (AC-D-23)

Test file `tools/test_gen_error_codes.py` contains three test groups:

1. **Golden-file tests (3 fixtures):** three minimal YAML inputs (1, 2, 13 codes) under `tools/testdata/`; each has expected `.hpp`, `.kt`, `.gd` goldens asserted byte-for-byte. Re-running the generator on unchanged input produces byte-identical output (idempotence).

2. **Property tests (Hypothesis):** strategy generates random valid manifests of length 1–20; emitted artifacts are parsed back (regex-based extraction sufficient; full parser not required) and `recoverable` / `retry_hint` values are cross-verified for agreement across all 3 artifacts.

3. **Failure tests:** malformed YAML (missing keys, invalid `retry_hint`, duplicate `name`, non-bool `recoverable`, schema_version ≠ 1, empty `codes`) each raise `ManifestValidationError` with a specific message identifying the offending code and field.

**Determinism rules (idempotence requirement from AC-1):**
- No timestamps in output.
- No absolute paths in output.
- No generator-version strings in output.
- UTF-8 encoding, LF line endings, trailing newline.
- Dictionary iteration uses manifest order only (never Python dict key order).

- `android/plugin/src/generated/kotlin/…` — new path; Android plugin module does not yet exist (Story 1-2 creates it). This story only writes the generated Kotlin file; Story 1-2 will add the Gradle module that consumes it.

## Pre-Implementation Notes

### Why TDD Full for Task 4 / Task 2

The generator emits code consumed by three languages; a bug would produce subtly-wrong enum values that compile cleanly but diverge at runtime (exactly the drift DD-13 is designed to prevent). Golden-file tests pin down the exact byte layout before the generator is written, which forces the template design to be driven by observable output, not by implementation convenience. Hypothesis property tests catch classes of bugs (ordering, escaping, unicode in `default_user_message`) that 3 goldens cannot enumerate.

### Test approach (per-task)

| Task | Test-first? | Rationale |
|------|-------------|-----------|
| Task 1 | No | Data file + trivial validator; schema covered indirectly by Task 4 failure tests. |
| Task 2 | **Yes (Full TDD)** | Generator must pass golden tests that are authored in Task 4 before this task starts. |
| Task 3 | No | Mechanical — run generator, commit output, verify `--check` passes. |
| Task 4 | **Yes (Full TDD)** | The pytest suite IS the test code; it is authored first, committed red, then Task 2 turns it green. |
| Task 5 | No | CI YAML; validated by running workflows on a throwaway PR or with `act` locally. |

### Expected commit count: 5 (one per task)

1. `feat(codegen): add MWA error-codes manifest + schema validator (1-1 Task 1)`
2. `test(codegen): add golden + property + failure pytest suite (red) (1-1 Task 4)` — note Task 4 commits before Task 2 due to TDD ordering.
3. `feat(codegen): implement Python generator with Jinja2 templates (1-1 Task 2)` — flips Task 4's suite green.
4. `feat(codegen): commit generated Kotlin/C++/GDScript artifacts + linguist markers (1-1 Task 3)`
5. `ci(codegen): add drift-check and generator-tests workflows (1-1 Task 5)`

Each commit runs the full verification loop from `.claude/rules/code-standards.md` §2 (format → lint → build → test → security). For Python-only commits the relevant chain is `pytest tools/ -q` + `python tools/gen_error_codes.py --check`; C++/Kotlin/GDScript verification kicks in once Stories 1-2+ build pipelines land.

### Story size justification

M (4-6 tasks). Single-language implementation (Python) with three output languages whose contents are mechanical translations of a shared manifest. No new runtime code, no network, no device interaction. Test volume is the dominant effort (~15+ pytest cases).

Before first commit of this story: `git tag pre-mwa-1-1`. After Gate 5 on the story: `git tag post-mwa-1-1`. Every commit stages only the files owned by its task (no `git add .`).
