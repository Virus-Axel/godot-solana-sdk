# Story 2-2: reauthorize + TOKEN_EXPIRED + 10-Cycle Silent-Reauth Test

## User Story
As a Godot game developer, I want to call `MWA.reauthorize()` on app relaunch and have the session silently restore without any wallet UI, so that returning players skip the chooser.

## Context
- **Wave 5 sibling of Story 2-3 with stronger surface area.** 2-3 (disconnect, retention) closed Gate 5 on 2026-04-24 (`post-mwa-android-2-3`); the SecureTokenStore retention invariant + DD-2-3-5 CacheKey re-derivability from caller-args are now MECHANICALLY proven on real `EncryptedSharedPreferences` (Story 2-3 T5 `AC1DisconnectRetainsTokenTest` freshKey roundtrip, `AC1DisconnectRetainsTokenTest.kt:78-94` evidence). Story 2-2's silent-reauth lookup path is the FIRST production consumer of that proof — `mwaReauthorize` will rebuild the same `CacheKey(cluster, chainId, walletPackage, identityUri)` from the caller's fresh args and call `store.getToken(rebuiltKey)` to retrieve the retained CacheRecord.
- **The conceptual cliff between disconnect (2-3, RETAIN) ↔ reauthorize-token-expired (2-2 AC-3, DELETE) ↔ deauthorize (4-1, DELETE-AND-REVOKE).** AC-3 is the FIRST production caller of `SecureTokenStore.deleteToken(key)` (signature at `SecureTokenStore.kt:73-75`, file hash `dd56afe63f6b3058`). Misclassifying AC-3 as "retain like disconnect" leaves a stale-but-unusable CacheRecord on disk that re-fires the same `TOKEN_EXPIRED` on next reauth. Misclassifying as "revoke like 4-1" reaches across the JNI seam to the wallet that just told us the token is dead — protocol-incorrect and pointless. AC-3 is the GRACEFUL WIPE path: client-side delete, NO wallet round-trip. See `docs/stories/2-3.md#§8.3 SecureTokenStore` for the explicit retention contract that does NOT apply here.
- **AC-4 cluster-mismatch detection is FREE per DD-27 + DD-2-2-7 3-tuple filter.** Per arch §3.1 + Story 2-1 T2's `CacheKey` length-prefixed canonicalization (`CacheKey.kt:17`, file hash `5a76693885ef1d68`), the 4-tuple `(cluster, chainId, walletPackage, identityUri)` produces a different `toHash()` for `cluster="devnet"` vs `cluster="mainnet"`. **However**, `walletPackage` is NOT known to the caller of `MWA.reauthorize()` — neither GDScript nor the JNI `(reqId, identityJson, cluster, chainId, timeoutMs)` shape carries it (see DD-2-2-7 below for the full source-of-truth analysis). The plugin therefore CANNOT call `store.getToken(fullKey)` directly at the reauth call site. Instead, per DD-2-2-7, it iterates `store.listAllKeys()` and filters by the 3-tuple `(cluster, chainId, identityUri)` derived from caller args; a caller who connected on devnet then config-swapped to mainnet finds ZERO matching records under the mainnet 3-tuple → AC-4 still collapses to AC-2's cache-miss → `mwa_error{NOT_CONNECTED}` branch, just via the filter step rather than the direct 4-tuple `toHash()`. **DD-2-2-1 [LOCKED]** below makes this explicit: the implementation does NOT add a separate "compare cached cluster against active cluster" branch; the 3-tuple filter does it.
- **AC-5 10-cycle real-Phantom release-gate is NOT CI-runnable today.** GitHub Actions has no FTL hooks for Phantom-on-emulator wallet automation, and Phantom-on-emulator support is wallet-version-fragile. T6 ships a documented manual procedure (`docs/release-gates/2-2-10-cycle-reauth.md`) NOT a CI test. **DD-2-2-6 [LOCKED]** below scope-fences this.
- **AC-6 fingerprint stability is mathematical.** `AuthTokenFingerprint.compute(authToken.reveal(), salt)` is pure HKDF-SHA256 (Story 2-1 T3, AC-7 closure). Per-install salt is stored under Keystore alias `godot-sdk-mwa-fingerprint-salt-v1` (DD-9 from Story 2-1) and is NEVER rotated by reauthorize (only `forget_all` rotates per Story 4-2 scope). Token bytes come from the SAME retained `CacheRecord`. Therefore: `fingerprint(token, salt)` returns byte-identical output before and after reauthorize. The unit test asserts the function-level invariant; the androidTest asserts the `MWA.get_auth_token_fingerprint()` getter round-trip.

## Architecture Guardrails

### §4.2 UC-3 reauthorize flow [LOCKED — inlined from arch.md]
> Ref: `docs/architecture.md#4 Workflows` — UC-3 reauthorize-success / reauthorize-token-expired alt-flows.

Reauthorize is a CLIENT-DRIVEN silent-restore op: the plugin loads the `CacheRecord` from `SecureTokenStore` keyed by the caller-rebuilt `CacheKey`, passes the cached `authToken` (SecretString) to `MwaClient.reauthorize(sender, identity, authToken, cluster, chainId)`, and on success writes the (potentially-rotated) returned auth_token back into the store + updates `MwaSessionState`. NO wallet UI surfaces (NFR-5). On `MwaResult.Failure(TokenExpired)`: graceful client-side wipe of the SecureTokenStore record + MwaSessionState auth-only fields + emit `mwa_error{TOKEN_EXPIRED}`. On cache miss (no record under the rebuilt key): emit `mwa_error{NOT_CONNECTED}` immediately — do NOT call `MwaClient.reauthorize()`.

### DD-2-2-1 [LOCKED] — AC-4 cluster-mismatch collapses to AC-2 cache-miss via DD-2-2-7's 3-tuple filter; NO separate branch
The `CacheKey` 4-tuple `(cluster, chainId, walletPackage, identityUri)` produces a DIFFERENT `toHash()` for differing cluster strings (DD-27). Combined with DD-2-2-7's `listAllKeys()`-then-filter discovery (the plugin doesn't know `walletPackage` at the reauth call site), a caller who connected on devnet then config-swapped to mainnet finds ZERO records matching the 3-tuple `(cluster="mainnet", chainId="solana:mainnet", identityUri=<caller>)` → no record loaded → AC-4 collapses to AC-2's NOT_CONNECTED branch.

**The implementation does NOT add a separate "compare cached cluster against active cluster" branch anywhere in `mwaReauthorize`.** If the implementer is tempted to add a `if (cachedRecord.cluster != activeCluster) emitFailure(NOT_CONNECTED)` check, that is a **Rule 2 deviation** requiring an amendment — the 3-tuple filter from DD-2-2-7 already distinguishes the two via DD-27. T1's AC-4 unit test asserts both: (a) the NOT_CONNECTED outcome, AND (b) that the code path to the failure is the filter-empty path (`listAllKeys()` filtered by the mainnet 3-tuple yields zero candidates), NOT a downstream "cached.cluster != requested.cluster" comparison. The static guard for this is in `ci/grep_bans.sh` pattern-9 (added in T2; see DD-2-2-K resolution below) — NOT in unit tests (per concern C-2-2-K).

### DD-2-2-2 [LOCKED] — AC-3 graceful-wipe ordering: state-wipe → disk-delete → emit-error
On `MwaResult.Failure(TokenExpired)` from `MwaClient.reauthorize`, the wipe-and-emit sequence MUST execute in this exact order, gated by a single `inflightMap.tryTerminate(requestId)` CAS:

1. **CAS (`tryTerminate`)** — win the terminal-signal slot. On false, log `late_result` + return without wiping or emitting.
2. **Wipe in-memory state** — `sessionState.clearOnLogout()` (preserves identity fields per Story 2-3 §8.3 contract).
3. **Wipe on-disk record** — `store.deleteToken(key)`. **First production caller of `SecureTokenStore.deleteToken`.**
4. **Emit terminal error** — `nativeBridge.postMwaError(buildErrorJson(..., MwaError.TokenExpired, sourceMethod = "reauthorize", ...))`.

**Why this order:**
- **Disk-after-memory** — convention uniformity with Story 2-3 DD-2-3-2's wipe-then-emit pattern + readability. Note: the entire `handleTokenExpired` block runs inside `withContext(mainDispatcher) { ... }`, so the four steps are observably atomic from any Godot-main reader on the same dispatcher; there is no concurrent reader on a different thread that could observe a torn (in-memory=true, disk=gone) pair under the current design. The explicit ordering keeps the handler readable AND future-proofs the code against a refactor that hops dispatchers (which WOULD reintroduce the torn-state hazard if the steps were reordered).
- **Emit-after-both-wipes** — convention uniformity + readability: the emission is the LAST step so the handler reads top-to-bottom as "claim → wipe → wipe → emit." Same dispatcher-serialization argument applies: under current design the caller's signal handler cannot interleave between the wipes and the emit (it runs on the same main dispatcher after the post returns); the ordering protects future refactors.
- **CAS-first** — gates the entire wipe-and-emit triple atomically, parallel to Story 2-3 DD-2-3-2's CAS-first-then-wipe-then-emit pattern. Any CAS-loss → no wipe + no emit (defense-in-depth; uniformity with disconnect/authorize is the primary motivator since CAS-loss on this path is not realistically reachable today).

T1 asserts the lexical order via grep on the `mwaReauthorize` body: `tryTerminate` precedes `clearOnLogout` precedes `deleteToken` precedes `postMwaError`.

### DD-2-2-3 [LOCKED] — Watchdog parity with authorize: `withTimeoutOrNull(effective)`, effective = min(60_000, opts.timeout_ms)
> Ref: `docs/architecture.md#7.3 Watchdog & Terminal-Signal Invariant` — DD-23 effective-watchdog formula.

Reauthorize uses `withTimeoutOrNull(effectiveMs)` with the SAME watchdog logic as Story 2-1's `mwaAuthorize`: `effective = min(INTERNAL_DEFAULT 60s, opts.timeout_ms)`. AC-1's ≤3s SLO is a **soft bound** on `reauthorize_completed` end-to-end latency under happy-path conditions; the watchdog's **hard bound** is the explicit caller timeout (defaulting to 60s).

**Method-return vs signal-fire timing:**
- `mwaReauthorize(requestId, ...)` itself returns within 1 frame (≤16ms) — it just `scope.launch`es and returns.
- `reauthorize_completed` fires when the coroutine completes — bounded by `withTimeoutOrNull(effectiveMs)`.
- AC-1's "≤3s on Godot main" is the signal-fire bound for the silent-happy fixture path.
- AC-2's "within 1 frame" is the cache-miss return — synchronous from the dispatcher hop, no `MwaClient.reauthorize` call (no suspension point).

The watchdog identity: `effective = min(60_000L, opts.timeoutMs ?: 60_000L)` — reuse Story 2-1's existing `effectiveWatchdog(timeoutMs)` helper at `GDExtensionAndroidPlugin.kt:116` (file hash `5a76693885ef1d68`). Do NOT introduce a sibling helper.

### DD-2-2-5 [LOCKED] — AC-6 fingerprint preservation is mathematical, not semantic
> Ref: `docs/architecture.md#DD-34` — fingerprint stability invariant.

Per DD-34: `AuthTokenFingerprint.compute(ikm = authTokenBytes, salt = perInstallSalt)` is HKDF-SHA256, output `[0..4].toHexString()` → 8 lowercase hex chars. The per-install salt is preserved across reauthorize (NEVER rotated by reauth — only `forget_all` rotates per Story 4-2). The token bytes come from the SAME retained CacheRecord (or — on success path — the wallet's possibly-rotated returned token, which AC-1 says must equal the previous public_key but does NOT mandate equal token bytes).

**Critical clarification:** `MwaClient.reauthorize` MAY return a refreshed `authToken` (clientlib-ktx implementation detail). If so, `CacheRecord.authToken` updates → fingerprint may change. **AC-6 says fingerprint is the SAME pre and post.** Per the FakeMwaClient `reauthorize_success.json` fixture (verified at `testdata/mwa-fixtures/reauthorize_success.json:11+18` — the `expected_input.auth_token` and `response.payload.auth_token` are byte-identical: `dGVzdC1hdXRoLXRva2VuLXJlYXV0aA==`), the test path uses an unchanged token. Real-Phantom AC-5 release-gate observes the same fingerprint stability empirically.

If a future MwaClient implementation rotates tokens on reauthorize, the AC-6 contract must be revisited (file an amendment). For 2-2 scope: `MwaClient.reauthorize` returns the same `authToken` bytes the caller passed in (ground truth: clientlib-ktx 2.0.3 + Phantom production behavior + FakeMwaClient fixture parity).

**Rotation-detection contract.** Even though the FakeMwaClient happy-path fixture returns byte-identical token bytes, `MwaClient.reauthorize` MAY in production return a refreshed/rotated `authToken` (clientlib-ktx implementation detail). The implementation MUST defensively handle this:
- Compare `auth.authToken.reveal()` (returned bytes) against `cached.authToken.reveal()` (pre-call bytes) byte-for-byte: `val isRotated = !auth.authToken.reveal().contentEquals(cached.authToken.reveal())`.
- If rotated: (a) update `CacheRecord.authToken` to the new bytes via `store.putToken(key, cached.copy(authToken = auth.authToken, lastUsedAtMs = clock()))`, (b) recompute `authTokenFingerprint` from the new bytes + the SAME per-install salt (the salt does NOT rotate on reauth), (c) log a non-secret event marker `SdkLog.w(TAG, requestId) { "auth_token_rotated_by_wallet" }` (the marker logs the EVENT only — bytes/fingerprint never appear in the log args), (d) update `sessionState.setAuthTokenFingerprint(newFingerprint)`, (e) emit `reauthorize_completed` with the NEW fingerprint in the payload.
- AC-6's "same fingerprint pre/post" invariant ONLY holds when the wallet does NOT rotate. If the wallet rotates, the new fingerprint is the CORRECT value (the cached token would be stale). T1 + T5 fixtures use byte-identical tokens (FakeMwaClient `reauthorize_success.json`), so AC-6's strict equality assertion holds in test scope.
- T6 release-gate manual procedure (real Phantom) treats a fingerprint change between cycles as a Phantom-rotation EVENT — record in runs/<date>.md but do NOT auto-fail the gate. Two or more rotations in a single 10-cycle run would suggest a wallet bug — escalate manually.
- A future story (Story 4-x or amendment) may revisit if rotation behavior becomes a contract issue; for 2-2 scope, "rotated" is a non-failure case.

T1 unit test asserts `AuthTokenFingerprint.compute(token, salt)` returns the same bytes given the same inputs — function-level math, not protocol round-trip. T5 androidTest asserts `MWA.get_auth_token_fingerprint()` returns the same string before and after.

### DD-2-2-7 [LOCKED] — `walletPackage` discovery via `store.listAllKeys()` 3-tuple filter (concern C-2-2-A resolution)
**The problem.** `mwaReauthorize` needs the full 4-tuple `CacheKey(cluster, chainId, walletPackage, identityUri)` to call `store.getToken(key)` / `store.deleteToken(key)`. Caller-provided args carry only THREE of the four fields:
- `cluster` and `chainId` come from the JNI argshape `(reqId, identityJson, cluster, chainId, timeoutMs)`.
- `identityUri` comes from `parseIdentity(identityJson).identityUri`.
- `walletPackage` does **NOT** travel through any caller-side surface — neither GDScript callers (`MWA.reauthorize(opts)`) nor the JNI shape carries it. It is captured only at AUTHORIZE time (Story 2-1 T4 `handleSuccess` line 541-550) where it lives on `auth.walletPackage` from the `MwaClient.authorize` result and is written to `CacheRecord.walletPackage` via `putToken`. `MwaSessionState` does NOT hold it (no `walletPackage` field in `MwaSessionState.kt:46-66` — frozen for 2-2 per Verified Interfaces).

**Three options considered (all evaluated 2026-04-26):**
1. **Iterate `store.listAllKeys()` and filter by `(cluster, chainId, identityUri)`** — preserves the MwaSessionState freeze AND the FakeMwaClient/SecureTokenStore freeze. O(N) where N is the number of stored CacheRecords (in practice N ≤ small — typically 1, occasionally 2-3 if the user has connected to multiple wallets on the same game/cluster).
2. Add a `walletPackage` getter+setter to `MwaSessionState` — requires unfreezing MwaSessionState.kt (Rule 3 amendment) and writing to it from `handleSuccess` (Story 2-1 surface change).
3. Pass `walletPackage` as an explicit `mwaReauthorize` JNI argument — requires changing the JNI argshape (Rule 3 amendment + C++ JNI signature change + GDScript caller-side knowledge that doesn't exist; the user doesn't know which wallet they connected to).

**Choose option 1.** Rationale: zero surface changes outside `mwaReauthorize` itself; preserves all 2-2 freezes; the listAllKeys cardinality is naturally bounded by user behavior; the filter is a 3-line `filter { it.cluster == cluster && it.chainId == chainId && it.identityUri == identityUri }`.

**Tie-break on multi-match.** If 2+ keys match the 3-tuple (e.g., the user connected with Phantom AND Solflare to the same game on devnet), pick the most-recently-used: `keys.maxByOrNull { store.getToken(it)?.lastUsedAtMs ?: 0L }`. Rationale: most-recent-wallet matches user expectation (they're trying to silent-reauth into the wallet they last used). The tie-break is deterministic — never two records with byte-identical timestamps in practice (Story 2-1 T4 `handleSuccess` updates `lastUsedAtMs = clock()` before `putToken`).

**T2 pseudocode update.**
```kotlin
val store = storeProvider(requireContext())
val candidates = store.listAllKeys().filter {
    it.cluster == cluster && it.chainId == chainId && it.identityUri == parsed.identityUri
}
if (candidates.isEmpty()) {
    emitFailureReauth(requestId, MwaError.NotConnected,
        "no cached token under (cluster=$cluster, chainId=$chainId, identityUri=${parsed.identityUri})")
    return@withContext
}
val key = candidates.maxByOrNull { store.getToken(it)?.lastUsedAtMs ?: 0L } ?: candidates.first()
val cached = store.getToken(key)
if (cached == null) {
    // Race: key was in the listAllKeys snapshot but record was deleted between
    // listAllKeys and getToken. Treat as cache-miss (concurrent disconnect / wipe).
    emitFailureReauth(requestId, MwaError.NotConnected,
        "cache record vanished between listAllKeys and getToken (race)")
    return@withContext
}
// ... proceed with cached.authToken to MwaClient.reauthorize, using `key` for any subsequent deleteToken
```

                          val result = withTimeoutOrNull(effectiveMs) {

                              is MwaResult.Failure -> {

                                      handleTokenExpired(requestId, key, store)  // DD-2-2-2 graceful wipe
                                  } else {
                                      emitFailureReauth(requestId, result.error, null)
                                  }
                              }
                          }
                      } finally {
                          runCatching { client.close() }
                      }
                  }
              } catch (ce: CancellationException) { throw ce }
              catch (ex: Throwable) { /* same crash-handling pattern as mwaAuthorize, source_method=reauthorize */ }
          }
      }

              SdkLog.w(TAG, requestId) { "auth_token_rotated_by_wallet" }  // event marker, NO bytes
          }

          // ... remaining sessionState updates (publicKey, walletLabel, etc.) per Story 2-1 handleSuccess
          if (!inflightMap.tryTerminate(requestId)) {
              diagnostics.incrementLateResult(); SdkLog.w(...) { "late_result outcome=reauthorize" }; return
          }

              buildSuccessJson(requestId, newFingerprint, publicKeyBase58, walletLabel, walletIconUri, cluster).toString())
      }

      }
