# Story 4-3: Keystore Corruption Recovery + Manifest Hardening + R8 Rules + Dep Lockfile + Reproducible Build

## User Story
As an SDK auditor, I want the plugin to (a) recover without crashing when Android Keystore entries are invalidated by a lockscreen reset, (b) ship with hardened manifest + R8 rules that consumer apps cannot silently weaken, and (c) have reproducible builds with verified dependencies, so that the plugin passes a mobile security audit.

## Architecture Guardrails

### LOCKED Design Decisions (story-local)

- **DD-4-3-1 [LOCKED] — `StorageCorruptException` catches emit `reauth_required{reason:"keystore_corrupt"}`, NOT `mwa_error{StorageCorrupt}`.** AC-1 verbatim: "*reauth_required{reason:"keystore_corrupt"} fires*." The existing catches at `GDExtensionAndroidPlugin.kt:720` (reauthorize) and `:1328` (authorize) currently emit `emitFailureReauth(...MwaError.StorageCorrupt...)` and `emitFailure(...MwaError.StorageCorrupt...)` respectively — both fire `mwa_error`. Story 4-3 T2 PIVOTS both call sites to the new helper `emitReauthRequiredKeystoreCorrupt(requestId, sourceMethod)` which calls `nativeBridge.postReauthRequired(buildReauthRequiredJson(reason="keystore_corrupt", ...))`. The legacy `MwaError.StorageCorrupt` enum entry STAYS in the codegen output (used as the `cause` field of the reauth_required payload to preserve A-14 10-key continuity for tooling that already keys off `code=STORAGE_CORRUPT`); see DD-4-3-1.b for payload shape. Rationale: AC-1 is unambiguous about the signal channel (`reauth_required` not `mwa_error`); the existing emit sites were a Story 2-1 hold-over that pre-dated the AC-1 wording.
  - **DD-4-3-1.a — fail-closed plugin-boundary wrapper.** Add a wrapper helper `withStorageOrReauthRequired(requestId, sourceMethod) { block }` that wraps EVERY first-touch SecureTokenStore call site (including the cache-lookup early in authorize/reauthorize and the `getOrCreatePerInstallSalt` call) so `StorageCorruptException` is caught uniformly. Currently the catches only exist on the success-write paths; corruption surfacing during cache READ propagates as an uncaught `Throwable` → wrapped as `MwaError.ProtocolError`. Failing CLOSED to `reauth_required` is the AC-1 contract.
  - **DD-4-3-1.b — `reauth_required` payload shape (1-Dict per A-12).** Per arch §3.2 `reauth_required` is in the 1-param error/lifecycle family (A-12 amendment §"4 error/lifecycle family"). The canonical payload Dictionary shape for `reason="keystore_corrupt"` is:
    ```
    {
      "reason": "keystore_corrupt",            // string, lowercase_snake — only value defined this story
      "request_id": "<8-hex>" | "",            // empty when corruption fires outside an op (no request to correlate)
      "source_method": "authorize" | "reauthorize" | "<other>" | "",  // op that triggered the corrupt-recovery branch; "" when uninvolved
      "developer_details": "EncryptedSharedPreferences corrupt: <ExceptionClassName>",
      "cause": MwaError.StorageCorrupt.code     // resolves to "STORAGE_CORRUPT" via codegen — consumers MAY key off this for diagnostics; see C-4-3-G (use codegen reference, not string literal, to drift-protect against future code-string changes)
    }
    ```
    The `reason` key is the CONTRACT discriminator — consumers `match payload.reason: "keystore_corrupt": …`. Reserved future values: `"session_expired"`, `"token_rotated"` (NOT in scope this story). Document in §Dev Notes for downstream MWA.gd facade work.

- **DD-4-3-2 [LOCKED] — `NativeBridge.postReauthRequired(reauthDictJson: String)` added; `postReauthRequiredNative` reverse-fun keep rule mirrored.** Mirror the 8 existing `postX` methods on `NativeBridge`. The variable name `reauthDictJson` MUST be added to `ci/grep_bans.sh` pattern-8 (the SdkLog/Log.* token-leak guard) so future maintainers cannot interpolate the payload into a log line. The seam method takes ONLY the JSON blob (1-param family per A-12; `request_id` is embedded inside the Dictionary at the `request_id` field, NOT a separate arg — same shape as `postMwaError`/`postMwaTimeout`). Reverse external fun `postReauthRequiredNative` is ALREADY on the Companion (line 311); no JNI changes required.

- **DD-4-3-3 [LOCKED] — `backup_rules.xml` AND `data_extraction_rules.xml` co-ship; both exclude the prefs file.** Arch §8.4 specifies `backup_rules.xml` (legacy AutoBackup, `<full-backup-content>`) but Android 12+ (API 31+) deprecated this in favor of `data_extraction_rules.xml` (`<data-extraction-rules>` with separate `cloud-backup` and `device-transfer` sections). Since the plugin's `targetSdk = 34`, BOTH files MUST ship — `backup_rules.xml` for API 23-30 fallback, `data_extraction_rules.xml` for API 31+. Both reference the same prefs file path `godot-sdk-mwa-tokens-v1.xml`. The `AndroidManifest.xml` declares `android:dataExtractionRules="@xml/data_extraction_rules"` ALONGSIDE `android:fullBackupContent="@xml/backup_rules"` — mutually exclusive on a per-API basis but coexisting on disk. Arch §8.4 already calls out `android:dataExtractionRules` line 987.

- **DD-4-3-4 [LOCKED] — `dependencyLocking { lockAllConfigurations() }` AND `gradle/verification-metadata.xml` BOTH committed; lockfile generated via `--write-locks`, verification metadata via `--write-verification-metadata sha256,pgp` (PGP-best-effort; see DD-4-3-4.a).** AGP-friendly Gradle dependency-verification (https://docs.gradle.org/current/userguide/dependency_verification.html) needs both: the lockfile pins resolved versions; the verification metadata pins per-artifact SHA256 + PGP signatures. Use SHA-256 (NOT SHA-1, which is deprecated). Lockfile lives at `android/gradle.lockfile` (single-file mode); verification metadata at `android/gradle/verification-metadata.xml`. Both regenerated via documented maintainer command (see §Dev Notes "lockfile regeneration").
  - **DD-4-3-4.a — PGP best-effort, SHA-256 mandatory.** Some transitive Maven artifacts (notably older `org.json:json` releases) have NO PGP signature on Maven Central. Set `<verify-signatures>false</verify-signatures>` at the verification-metadata level OR enumerate per-component `<not-required>` markers; pick the latter for finer-grained audit posture. SHA-256 verification stays mandatory across all 200+ transitive jars. CI fails if SHA mismatches; PGP gaps are warned not blocked.

- **DD-4-3-5 [LOCKED] — Reproducible-build CI runs on `ubuntu-latest` (NOT macos-latest); Gradle build cache disabled; AAR canonicalized before SHA256 diff.** Three deviations from the AC-6 prose:
  1. **`ubuntu-latest` over `macos-latest`** — every existing CI workflow that needs to run Gradle (instrumented_tests.yml, build.yml) is on `ubuntu-latest` per the CI-fixup-round-3 D-F5 lesson (instrumented_tests.yml:32-39). macOS runners cost 10x and have arm64-vs-x86_64 emulator pain (D-F4); ubuntu-latest is the established baseline. The plan/spec wording "macos-latest" is a hold-over from the spec's Phantom-instrumented preference — for THIS reproducible-build job the OS is irrelevant as long as both runners agree.
  2. **`org.gradle.caching=false` + `org.gradle.parallel=false` + `--no-build-cache`** — a warm Gradle build cache or differently-ordered task graph can produce different jar timestamps even with deterministic inputs. Force-disable to widen the determinism floor.
  3. **AAR canonicalization step** — Gradle 8.x AAR output embeds `META-INF/MANIFEST.MF` with `Created-By: Gradle <ver>` AND `Build-Jdk` metadata. Even on identical Gradle/JDK versions the timestamp on `MANIFEST.MF` and the `lastModifiedTime` on inner zip entries is non-deterministic. `tools/canonicalize-aar.sh` (T6) re-zips the AAR with `0` timestamps via `find ... -exec touch -t 197001010000 {} \;` then `zip -X` (no extra-fields) — the established `strip-nondeterminism`-equivalent hand-rolled. SHA256 of the canonicalized AAR is the diff target.

private fun buildReauthRequiredKeystoreCorruptJson(
    requestId: String,
    sourceMethod: String,
    ex: StorageCorruptException,
): String = JSONObject()
    .put("reason", "keystore_corrupt")
    .put("request_id", requestId)
    .put("source_method", sourceMethod)
    .put("developer_details", "EncryptedSharedPreferences corrupt: ${ex.cause?.javaClass?.simpleName}")
    .put("cause", MwaError.StorageCorrupt.code)  // codegen-sourced (resolves to "STORAGE_CORRUPT") per C-4-3-G — single source of truth, drift-resistant if codegen ever changes the literal
    .toString()

private suspend fun emitReauthRequiredKeystoreCorrupt(
    requestId: String,
    sourceMethod: String,
    ex: StorageCorruptException,
) {
    if (!inflightMap.tryTerminate(requestId)) {
        diagnostics.incrementLateResult()
        SdkLog.w(TAG, requestId) { "late_result outcome=reauth_required source=$sourceMethod" }
        return
    }
    SdkLog.w(TAG, requestId) {
        "reauth_required reason=keystore_corrupt source=$sourceMethod cause=${ex.cause?.javaClass?.simpleName}"
    }
    nativeBridge.postReauthRequired(buildReauthRequiredKeystoreCorruptJson(requestId, sourceMethod, ex))
}
```

### Gotchas
- **Tink shaded `InvalidProtocolBufferException` is NOT on the AAR's public compile classpath.** AC-1 prose names the exception, but `SecureTokenStore.kt:56` catches the JDK parent `IOException` (D-T1-4 — accepted at Story 2-1 Gate 5). Tests use `IOException` not the shaded class. Do NOT add a `tink` import.
- **R8 release-variant authorize E2E (T6 portion of AC-5)** — AGP 7.4.1's bundled R8 + Kotlin 1.8 sealed-class output requires `jvmTarget = "11"` per `build.gradle.kts:99-112` D-F1. Do NOT bump kotlinOptions.jvmTarget without coordinated AGP/Kotlin upgrade (see CR-5 in concerns).
- **Manifest-merge `xmlns:tools`** — required at the `<manifest>` ROOT for `tools:replace="android:allowBackup"` to be parsed; without it, AGP's manifest merger silently ignores the attribute. T3 must add `xmlns:tools="http://schemas.android.com/tools"` to the manifest root tag.
- **`ComposeWalletActivity android:exported="true"` decision** — current value is `true` but the Activity has no inbound `<intent-filter>`, so it is unreachable from outside the plugin. AC-3 allows EITHER removal of `exported="true"` OR adding a justification + filter. Implementer should default to removal (simpler, AC-3-strict) unless a downstream caller requires the exported surface (no current caller does).
- **Gradle `dependencyLocking` + lock-file mode quirks** — single-file mode (default in Gradle 8.x) writes ONE `gradle.lockfile`; per-configuration mode writes one per `Configuration`. Stay with single-file. The lockfile MUST be committed; `.gitignore` rules in `android/` should NOT exclude it.
