# Story 1-3: ISigner in godot-solana-sdk Core + Transaction Refactor + Compat Shim

## User Story

As an SDK consumer, I want `Transaction.sign(keypair)` to keep working after MWA lands, so that my existing godot-solana-sdk integration doesn't break.

## Acceptance Criteria

The five acceptance criteria below are the authoritative, verbatim ACs from `docs/plan.md` §Story 1-3 (lines 75-79). AC-6 is added by this story-creation pass to lock the synchronous-emission contract surfaced by adversarial review (Blocker 1) — it constrains the implementation of AC-3 and is verified by Task 4's smoke test.

- **AC-1 (interface defined):** *Given* `src/isigner.hpp` is added to core, *when* it compiles, *then* `ISigner : public godot::RefCounted` declares `virtual bool is_connected() const`, `virtual godot::String get_public_key() const`, `virtual void sign_messages(PackedByteArray, PackedInt32Array, String request_id)`, `virtual void sign_transactions(PackedByteArray, PackedInt32Array, String request_id)` with signals `sign_completed(String, Array)` and `sign_failed(String, String, String)`.

- **AC-2 (Transaction refactor):** *Given* `Transaction` node previously held a concrete signer reference, *when* the refactor merges, *then* `Transaction` depends only on `Ref<ISigner>` and the existing desktop `WalletAdapter` class `implements ISigner` via a thin bridging impl.

- **AC-3 (compat shim):** *Given* existing consumer code calls `Transaction.sign(keypair)`, *when* the code runs against the post-refactor SDK, *then* it works unchanged (wrapped internally by `LocalKeypairSigner : ISigner`) AND emits a one-time deprecation warning `"Transaction.sign(Keypair) is deprecated; use ISigner. See docs/migrations/isigner.md. Removed in v1.2."` via Godot `push_warning()`.

- **AC-4 (migration doc):** *Given* `docs/migrations/isigner.md` is authored, *when* a consumer reads it, *then* they find: what changed, a migration diff (before/after GDScript), the deprecation timeline (v1.1 warn → v1.2 remove), and an FAQ covering the common 3 use cases.

- **AC-5 (error case):** *Given* a post-v1.2 build (flag simulated via compile define), *when* `Transaction.sign(keypair)` is called, *then* it errors with `ERR_METHOD_NOT_FOUND` and a clear message pointing at `ISigner`.

- **AC-6 (synchronous-emission compat — added 2026-04-20 by adversarial review, see "Emission timing for compat shim" in Dev Notes):** *Given* GDScript caller pattern `tx.set_signers([keypair]); tx.sign()` (the pre-refactor synchronous behavior), *when* `tx.sign()` returns, *then* `tx.signatures[0]` is non-empty (populated synchronously inside the call frame). The `LocalKeypairSigner` MUST emit `sign_completed` synchronously via direct `emit_signal(...)` (NOT `call_deferred`) so the legacy synchronous consumer pattern is preserved verbatim.

## Architecture Guardrails

### DD-11 [LOCKED] — `ISigner` core abstraction

From `docs/architecture.md:1158`: "`ISigner` abstract C++ interface added to `godot-solana-sdk` core; `Transaction` depends on `ISigner`; desktop `WalletAdapter` and Android `MobileWalletAdapter` both implement it."

→ The interface lives in **core** (`src/isigner.hpp`), not under `src/mwa/`. No reverse dependency from core to MWA — implementations register themselves into the core seam. This means MWA's future `MobileWalletAdapter` (Story 2-1) will `#include "isigner.hpp"` from core, and the dependency arrow points MWA → core only.

### DD-25 [LOCKED] — Backward-compat shim with deprecation timeline

From `docs/architecture.md:1201`: "`ISigner` contract inlined in §2.3.1 with full C++ signature + threading contract; backward-compat shim retains `Transaction::sign(keypair)` for one minor with deprecation warning; removed v1.2. `docs/migrations/isigner.md` + upstream signoff required."

→ Two non-negotiables flow from this:
1. The compat shim MUST emit a deprecation warning on first use (one-time per process — not per call) referencing `docs/migrations/isigner.md` and the v1.2 removal milestone.
2. `docs/migrations/isigner.md` is a hard requirement, not optional documentation. AC-4 verifies it exists with the four required sections (what changed / migration diff / timeline / 3-use-case FAQ).

**Upstream signoff:** mentioned in DD-25 as `upstream-signoff.md` commit reference. Out of scope for this story's mechanical implementation — flag for the user as a release-time gate (the merge-back to `Virus-Axel/godot-solana-sdk` upstream cannot happen without it). Track via concern, do not block this story.

### DD-22 [LOCKED] — Single thread-marshalling rule (scoped to cross-thread Kotlin→GDScript)

From `docs/architecture.md:1198`: "Single thread-marshalling rule: all signals cross to Godot main via `call_deferred` in the C++ `GodotMainDispatcher`; no Kotlin-direct `emitSignal` for user-facing signals."

→ **Scope clarification (load-bearing for Task 4 / AC-6):** DD-22's `call_deferred` mandate is explicitly scoped per `docs/architecture.md:780-794` (§7.1) to **Kotlin → GDScript** signal marshalling, where the Kotlin coroutine resumes on an arbitrary thread and the C++ JNI callback fires off-main. The §2.3.1 contract for `ISigner` says only "results arrive via signals on the Godot main thread" — it does NOT mandate `call_deferred` as the mechanism. For implementations that already run on Godot main (the `LocalKeypairSigner` compat shim invoked synchronously from `Transaction::sign_at_index` on the calling thread, which IS the Godot main thread for any GDScript-driven code path), direct `emit_signal(...)` is fully compliant with the §2.3.1 contract.

**Story 2-1's `MobileWalletAdapter` requires `call_deferred`** (Android JNI callbacks land on coroutine IO threads). **Story 1-3's `LocalKeypairSigner` does NOT** — `Keypair::sign_message` runs synchronously on the calling thread, no thread hop happens, and the §2.3.1 "Godot main thread" guarantee is satisfied trivially.

This is the resolution to adversarial-review Blocker 1 — see "Emission timing for compat shim" in Dev Notes for the full rationale.

### §2.3.1 — `ISigner` Contract (LOCKED — verbatim header, with A-11 amendment for `GDCLASS_CUSTOM`)

From `docs/architecture.md:128-166` — the load-bearing C++ signature. The header below is the **exact contract** Story 1-3 ships at `src/isigner.hpp`. Note one amendment from this story-creation pass:

**A-11 (filed by this story, see `docs/amendments.md#A-11`):** the `GDCLASS(ISigner, godot::RefCounted)` macro in the architecture verbatim is amended to `GDCLASS_CUSTOM(ISigner, godot::RefCounted)` for codebase-wide registration consistency. All three verified existing classes (`Transaction`, `WalletAdapter`, `Keypair`) use `GDCLASS_CUSTOM`. Using plain `GDCLASS` for the parent of child classes that all use `GDCLASS_CUSTOM` risks silent vtable / metadata divergence. Rationale recorded in A-11.

```cpp
// src/isigner.hpp (LOCKED contract per §2.3.1, with A-11 amendment for GDCLASS_CUSTOM)
#pragma once
#include <godot_cpp/classes/ref_counted.hpp>
#include <godot_cpp/variant/packed_byte_array.hpp>
#include <godot_cpp/variant/string.hpp>

namespace godot_solana_sdk {

class ISigner : public godot::RefCounted {
    GDCLASS_CUSTOM(ISigner, godot::RefCounted);  // Amended per A-11 (was GDCLASS in §2.3.1)

protected:
    static void _bind_methods();

public:
    // Synchronous state queries (≤1 ms, no I/O)
    virtual bool is_connected() const = 0;
    virtual godot::String get_public_key() const = 0;  // base58, "" if not connected

    // Async operations — return void, emit signals on Godot main thread
    // Signals defined on ISigner itself: sign_completed(request_id, signatures),
    //                                     sign_failed(request_id, error_code, message)
    virtual void sign_messages(const godot::PackedByteArray & messages_concat,
                                const godot::PackedInt32Array & lengths,
                                const godot::String & request_id) = 0;
    virtual void sign_transactions(const godot::PackedByteArray & txs_concat,
                                    const godot::PackedInt32Array & lengths,
                                    const godot::String & request_id) = 0;
};

}  // namespace godot_solana_sdk
```

**Threading contract (load-bearing — see DD-22 scope clarification above):** all four virtual methods return immediately (no blocking I/O on the calling thread); results arrive via `sign_completed` / `sign_failed` signals on Godot main thread. `is_connected()` and `get_public_key()` are synchronous reads (≤1 ms, no I/O). `LocalKeypairSigner` (this story) emits **synchronously** via direct `emit_signal(...)` because the underlying `Keypair::sign_message` runs synchronously on the calling thread (which IS Godot main for GDScript-driven sign() calls). Story 2-1's `MobileWalletAdapter` will use `call_deferred` via `GodotMainDispatcher` because Android JNI callbacks land off-main.

**Lifetime ownership:** `Ref<ISigner>` (engine-managed via `RefCounted`). Callers store `Ref<ISigner>` fields, not raw pointers. The compat-shim `LocalKeypairSigner` is itself a `RefCounted` and is constructed on-the-fly inside `Transaction::sign_at_index`'s adapter path; it lives as long as the `Transaction` keeps it in `signers`. With synchronous emission (AC-6), the local `Ref<LocalKeypairSigner>` in `sign_at_index` is valid through the entire emit chain — see Concern 4 resolution below.

**Signal payloads** (defined in `_bind_methods()` via `ADD_SIGNAL`):
- `sign_completed(request_id: String, signatures: Array)` — emitted on success; `signatures` is `Array[PackedByteArray]` matching the count of inputs.
- `sign_failed(request_id: String, error_code: String, message: String)` — emitted on failure; `error_code` is the string form from the `MwaError` taxonomy. **Note (CR-14, see `docs/concerns.md#CR-14`):** `LocalKeypairSigner` reuses `MwaErrorCode::NOT_CONNECTED` for its non-MWA error reporting. Acceptable namespace overlap for v1.1; v1.2 cleanup may rename to `SignerErrorCode`. Add inline `// TODO(CR-14)` comment at the use site.

### Architectural reality check — `WalletAdapter : Node`, `Keypair : Resource`, `ISigner : RefCounted`

This story straddles three Godot base-class hierarchies that DO NOT inherit from each other:

| Class | Base | Notes |
|-------|------|-------|
| `WalletAdapter` | `godot::Node` (verified, see `include/wallet_adapter/wallet_adapter.hpp:27`) | Not a `RefCounted` — held as raw pointer in scene tree |
| `Keypair` | `godot::Resource` (verified, see `include/keypair.hpp:27`) | `Resource` IS a `RefCounted` (Godot inheritance), so `Ref<Keypair>` is valid |
| `ISigner` | `godot::RefCounted` (LOCKED §2.3.1) | New |

**Why this matters for AC-2:** `WalletAdapter` cannot `: public ISigner` directly because Godot's GDExtension does not support multiple inheritance through two `GDCLASS_CUSTOM` chains rooted at different engine bases (`Node` vs `RefCounted`). The plan AC-2 wording — "WalletAdapter class **implements ISigner via a thin bridging impl**" — anticipates this. **Task 2 implements the bridge as a separate `WalletAdapterSigner : ISigner` adapter class** that holds a non-owning pointer to a scene-tree-owned `WalletAdapter` and forwards `sign_messages` / `sign_transactions` calls into the existing `WalletAdapter::sign_message(...)` API.

**Why this matters for AC-3:** `Keypair` IS a `Resource` (so `Ref<Keypair>` works). `LocalKeypairSigner : public ISigner` holds a `Ref<Keypair>` cleanly — no bridge needed, just composition.

### Existing Transaction signer model (verified — pre-refactor baseline)

`Transaction::sign()` currently iterates a `signers: Array` of `Variant`s. The active dispatch in `Transaction::sign_at_index(uint32_t index)` (verified at `src/transaction/transaction.cpp:176-205`) uses **runtime type checks** to branch:

1. `Keypair::is_keypair(signers[index])` → call `Keypair::sign_message(serialize_message())` synchronously; immediately set `signatures[index]` and emit `signer_state_changed`.
2. `Keypair::is_compatible_type(signers[index])` → convert via `Keypair::new_from_variant(...)` then sign synchronously.
3. `signers[index].has_method("sign_message")` → cast to `WalletAdapter*`, connect `message_signed` / `signing_failed` signals, call `WalletAdapter::sign_message(serialize(), index)` (async).
4. Else → `ERR_PRINT_ONCE_ED("Unknown signer type.")`.

`Transaction::sign()` (verified at `transaction.cpp:620`) takes **no parameters** today — it consumes a pre-populated `signers` Array set by the caller via `set_signers(Array)`. The compat path in AC-3 (`Transaction.sign(keypair)`) is GDScript-side ergonomic shorthand: callers do `tx.set_signers([keypair]); tx.sign()`. The deprecation warning fires inside `sign_at_index` when branch (1) or (2) is taken (a raw `Keypair` in `signers`).

**Critical pre-refactor invariant (preserved by AC-6):** branch (1) and (2) populate `signatures[index]` synchronously — by the time `sign()` returns, `tx.signatures[0]` is non-empty. AC-6 makes this an explicit acceptance criterion. The post-refactor `LocalKeypairSigner` MUST preserve this invariant via synchronous `emit_signal`.

> Ref: `src/transaction/transaction.cpp#sign_at_index` — current type-dispatch logic that the refactor replaces with `Ref<ISigner>` calls.
> Ref: `src/wallet_adapter/wallet_adapter.cpp#_bind_methods` — current `message_signed` / `signing_failed` signals the bridge will subscribe to.

## Dev Notes

### Emission timing for compat shim — DD-22 scope clarification (load-bearing)

This subsection documents the architectural rationale for `LocalKeypairSigner`'s synchronous `emit_signal` (per AC-6 / Concern from adversarial review Blocker 1). It is the load-bearing decision the implementer must understand before writing Task 4.

**The architectural contract surface:**

1. **§2.3.1 (`docs/architecture.md:128-166`)** says: "methods return immediately; results arrive via signals on the **Godot main thread**." It does NOT mandate `call_deferred` as the mechanism. The guarantee is "main thread"; any mechanism that satisfies that guarantee is compliant.

2. **DD-22 [LOCKED] (`docs/architecture.md:1198`)** says: "Single thread-marshalling rule: all signals cross to Godot main via `call_deferred` in the C++ `GodotMainDispatcher`; no Kotlin-direct `emitSignal` for user-facing signals."

3. **§7.1 (`docs/architecture.md:780-794`)** scopes DD-22's `call_deferred` rule explicitly: "Kotlin callbacks land on Android main thread or clientlib-ktx coroutine IO; `emit_signal` can only run on Godot main thread. **Single rule (no fallback path):** *Every* signal to GDScript crosses to the Godot main thread via `call_deferred` in the C++ `MobileWalletAdapter` node. **Kotlin** never emits signals directly to GDScript."

DD-22's `call_deferred` mandate is therefore SCOPED to the **cross-thread Kotlin → C++ JNI → GDScript** path. It exists to handle the case where the originating callback fires off-main and must be marshalled to main. It does NOT apply to implementations that are already on Godot main.

**The legacy consumer contract surface:**

Pre-refactor `Transaction::sign()` is synchronous. The dispatch in `Transaction::sign_at_index` (verified `transaction.cpp:176-205`):
- Branch (1): `Keypair::sign_message(serialize_message())` runs synchronously and immediately assigns `signatures[index] = sig`.
- By the time `tx.sign()` returns control to the GDScript caller, `tx.signatures[0]` is populated.

Existing GDScript code relies on this:
```gdscript
tx.set_signers([kp])
tx.sign()
var sig := tx.signatures[0]   # works — sig is populated
```

**The trap if we used `call_deferred`:**

If `LocalKeypairSigner` used `call_deferred("emit_signal", "sign_completed", ...)`, the signal would fire NEXT frame, AFTER `tx.sign()` returns to the caller. `tx.signatures[0]` would be EMPTY at the line above. Every existing consumer would silently break: no error, no warning, just `tx.signatures[0] == PackedByteArray()` until the next frame.

**The resolution (locked by AC-6):**

`LocalKeypairSigner::sign_messages` emits **synchronously** via direct `emit_signal(...)` (NOT `call_deferred`). The signal handler runs INSIDE the call frame of `sign_messages`, which runs INSIDE `sign_at_index`, which runs INSIDE `sign()`. By the time `sign()` returns, `signatures[index]` is populated — preserving the pre-refactor synchronous semantics verbatim.

**Compliance with §2.3.1's "Godot main thread" guarantee:** trivially satisfied — the entire call chain runs on the calling thread, which IS Godot main for any GDScript-driven `tx.sign()` invocation. No thread hop ever happens for `LocalKeypairSigner`. The §2.3.1 contract is honored without `call_deferred`.

**Story 2-1's `MobileWalletAdapter` is different.** Android JNI callbacks land on coroutine IO threads (off-main). `MobileWalletAdapter` MUST use `call_deferred` via `GodotMainDispatcher` to satisfy DD-22's cross-thread marshalling rule. Story 1-3's `LocalKeypairSigner` doesn't need it because no thread hop happens.

**Implementer guard — load-bearing AC-6 smoke test:**

Before declaring Task 4 complete, run a GDScript smoke test that does:
```gdscript
tx.set_signers([kp])
var err := tx.sign()
assert(err == OK)
assert(tx.signatures.size() == 1)
assert(tx.signatures[0].size() == 64)  # SIGNATURE_LENGTH
```
If `tx.signatures[0]` is empty (size 0) → the implementation used `call_deferred` and broke AC-6. Re-implement with synchronous `emit_signal`.
