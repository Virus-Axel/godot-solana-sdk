# Story 4-1: deauthorize + finally-block Semantics + Cache Wipe Independent of Remote

## User Story
As a Godot game developer, I want `MWA.deauthorize()` to ALWAYS clear the local token even if the wallet is unreachable, so that a returned-and-stolen device never retains a stale connected state.

## Context
- **Wave-5 sibling of Story 2-3 (disconnect) — same skeleton, different invariant.** Where 2-3's `mwaDisconnect` performs a CLIENT-SIDE auth-only wipe and PRESERVES the SecureTokenStore CacheRecord (so 2-2 reauthorize can still find it), 4-1's `mwaDeauthorize` performs a CLIENT-SIDE wipe **plus** a best-effort REMOTE revoke **plus** an UNCONDITIONAL `SecureTokenStore.deleteToken(key)`. The two stories differ in three places only: (1) `mwaDeauthorize` adds a `try { remote_revoke } catch { warn } finally { wipe }` envelope, (2) `mwaDeauthorize` deletes the on-disk token, (3) the `deauthorize_completed` payload carries `{remote_revoke_succeeded, local_cache_cleared, warning}` instead of `{source_method: "disconnect"}`. Everything else — InflightMap CAS-first, mainDispatcher hop, JNI bridge wiring, terminal-signal invariant, mwa_error fallback for catastrophic local failures — is inherited verbatim from 2-3.
- **AC-NFR-SEC-5 [LOCKED]: local cache MUST be cleared regardless of remote outcome.** This is the security-critical invariant the entire story exists to enforce. The threat model is a returned/stolen device: if `deauthorize` were to short-circuit on a network error, the device retains a working `auth_token` that the next person who picks up the phone can use to silently `reauthorize` and resume the session. The `finally` block enforces this mechanically; AC-5's structural CI test enforces it grammatically (so a future maintainer cannot inadvertently move the wipe out of the `finally` and have CI catch it).
- **All Wave 1-4 stories closed. Wave 5 progress: 2-1, 2-3, 2-2, 3-1 done; 4-1 starting (this story); 4-3 backlog.** Kotlin unit baseline: **213/213 passing** (refreshed 2026-04-27 post-3-1). Pytest: 83/83 green. Total: 213. AndroidTest baseline: 12 ACs (post-3-1). No security-sensitive tests baselined as failing.
- **Current `deauthorize` wiring is a stub on the Kotlin side.** `GDExtensionAndroidPlugin.Companion.mwaDeauthorizeFromJni(reqId)` (line 173-175) currently calls `instance?.emitNotImplemented(reqId, "deauthorize") ?: emitInstanceNullError(reqId, "deauthorize")`. Story 4-1 replaces this with `instance?.mwaDeauthorize(reqId) ?: emitInstanceNullError(reqId, "deauthorize")`, mirroring the disconnect pattern (`mwaDisconnectFromJni` at :168-169). The C++ binding (`MobileWalletAdapter::deauthorize` at `mobile_wallet_adapter.cpp:174-182`), the JNI dispatcher (`MwaAndroidBridgeJni::deauthorize` at `mwa_android_bridge_jni.cpp:486-493`), and the `deauthorize_completed` signal declaration (`mobile_wallet_adapter.cpp:92` + `MWA.gd:30,87`) are ALREADY IN PLACE from Story 2-1 T7 + the original scaffolding. Story 4-1 needs only: (a) the Kotlin `mwaDeauthorize` body, (b) `NativeBridge.postDeauthorizeCompleted` interface method + override + companion `external fun`, (c) the C++ `JNIEXPORT postDeauthorizeCompletedNative` mirror function, (d) the `MWA.gd func deauthorize() -> void` facade method.

## Acceptance Criteria
*(verbatim from `docs/plan.md` lines 297-301 — do not rephrase)*

- **AC-1 (happy remote + local):** *Given* connected session, wallet online, *when* `MWA.deauthorize()` is called, *then* within ≤3 s `deauthorize_completed{remote_revoke_succeeded: true, local_cache_cleared: true, warning: ""}` fires; `SecureTokenStore.getToken(key)` returns null; `is_connected()==false`.
  - **Evidence path:** Kotlin unit `MwaAndroidPluginDeauthorizeTest.`\``deauthorize happy path emits both flags true and deletes token`\` (T1) + androidTest `AC1DeauthorizeHappyTest` (T5).

- **AC-2 (network-off):** *Given* network disabled, *when* `MWA.deauthorize()` is called, *then* `deauthorize_completed{remote_revoke_succeeded: false, local_cache_cleared: true, warning: "remote_unreachable"}` fires; token STILL removed from SecureTokenStore; androidTest covers this path with `setNetworkEnabled(false)`.
  - **Evidence path:** Kotlin unit `MwaAndroidPluginDeauthorizeTest.`\``deauthorize with remote-revoke failure still wipes token and emits remote_unreachable warning`\` (T1, FakeMwaClient `deauthorize_remote_unreachable.json` fixture forcing `MwaResult.Failure`) + androidTest `AC2DeauthorizeNetworkOffTest` (T5).

- **AC-3 (wallet uninstalled):** *Given* wallet app uninstalled between connect and deauth, *when* `MWA.deauthorize()` is called, *then* same shape as AC-2 (local wipe always).
  - **Evidence path:** Kotlin unit `MwaAndroidPluginDeauthorizeTest.`\``deauthorize with WalletUninstalled classification still wipes token and emits remote_unreachable warning`\` (T1, FakeMwaClient `deauthorize_wallet_uninstalled.json` fixture which encodes error code `WALLET_UNINSTALLED` → `MwaError.WalletUninstalled` → `MwaResult.Failure`). WalletUninstalled is the `MwaResult.Failure` outcome class clientlib-ktx surfaces when `ActivityNotFoundException` propagates from `adapter.transact { ... }`. The deauthorize layer treats ANY `MwaResult.Failure` as "remote unreachable", so AC-2 and AC-3 share one code path; the test split exists to prove the classification doesn't accidentally upgrade WalletUninstalled to a non-recoverable error.

- **AC-4 (idempotent):** *Given* already deauthorized, *when* called again, *then* `deauthorize_completed{remote_revoke_succeeded: true|false, local_cache_cleared: true}` fires (no error).
  - **Idempotence semantics:** The second call has no `auth_token` in `MwaSessionState` (wiped by first call's `finally`) and no `CacheRecord` in `SecureTokenStore` (deleted by first call's `finally`). The Kotlin path therefore SKIPS the remote revoke (no token to revoke; remote_revoke_succeeded:true vacuously) and emits `deauthorize_completed{remote_revoke_succeeded: true, local_cache_cleared: true, warning: ""}`. Calling `SecureTokenStore.deleteToken(key)` on a missing key is a no-op (verified via grep of `SecureTokenStore.kt:73-76` — it calls `prefs.edit().remove(...).apply()` which is idempotent on Android `SharedPreferences`).
  - **Evidence path:** Kotlin unit `MwaAndroidPluginDeauthorizeTest.`\``second deauthorize after wipe emits remote_revoke_succeeded=true vacuously`\` (T1).

- **AC-5 (finally-block enforcement):** *Given* Kotlin code in `MwaAndroidPlugin.deauthorize`, *when* reviewed, *then* `SecureTokenStore.deleteToken()` + `MwaSessionState.clear()` are inside a `finally` block; a CI structure test (AST or source regex) verifies this.
  - **Evidence path:** A new Kotlin unit test class `DeauthorizeStructuralTest` (T6) — runs as part of `./gradlew test`, NOT a separate tier — reads the **source file** `GDExtensionAndroidPlugin.kt` from disk via `Files.readString(Paths.get("src/main/java/.../GDExtensionAndroidPlugin.kt"))`, locates the `fun mwaDeauthorize(...)` body via brace-balanced parsing, and asserts the body contains exactly one `finally {` lexical token and that BOTH `secureTokenStore.deleteToken(` AND `sessionState.clear` appear textually inside that finally-block's brace range. Regex-based (no PSI/AST dependency added). Implementation reference: `addons/SolanaSDK/mwa/MWA.gd` similar source-file-as-test pattern is rare in this codebase — closest precedent is `ci/grep_bans.sh` (CI-level grep) but here we want a Kotlin-runnable test that fails on `./gradlew test`, NOT just CI shell. ## Architecture Guardrails

### §4.5 Workflow E — Deauthorize [LOCKED — inlined verbatim from `docs/architecture.md:533-580`]

```
MWA.deauthorize()
   │
   ▼
Kotlin plugin:
   try {
     clientlib.revoke(auth_token)  // remote best-effort
     remote_succeeded = true
   } catch (e) {
     remote_succeeded = false
     SdkLog.w(...)
   } finally {
     SecureTokenStore.deleteToken(currentKey())  // MUST always run
     MwaSessionState.clear()
   }
   emitSignal("deauthorize_completed", {
     remote_revoke_succeeded: remote_succeeded,
     local_cache_cleared: true,
     warning: if (!remote_succeeded) "remote_unreachable" else ""
   })
```

**Invariant**: AC-NFR-SEC-5 — local cache MUST be cleared regardless of remote outcome. `finally` block enforces this; unit test simulates network-off.

### `deauthorize_completed` signal payload [LOCKED — `arch.md:262, :669`]
```
{request_id, remote_revoke_succeeded, local_cache_cleared, warning}
```
The 4-key result Dictionary is wrapped in the **A-12 2-param** `(request_id: String, result: Dictionary)` signal arity:
- arg 0 (String): `request_id`
- arg 1 (Dictionary): `{request_id, remote_revoke_succeeded, local_cache_cleared, warning}`

The result Dictionary INCLUDES `request_id` as a top-level key (mirroring connect/reauthorize/disconnect/sign_messages payloads — every `*_completed` result dict echoes the request_id internally for symmetry; see `arch.md:669`). This duplicates the arg-0 String, which is intentional and consistent across the family.

### A-12 2-param `*_completed` arity [LOCKED — `amendments.md:189`]
> "7 `*_completed` family — 2 params: `PropertyInfo(Variant::STRING, "request_id")` THEN `PropertyInfo(Variant::DICTIONARY, "result")`. Applies to: `connect_completed`, `reauthorize_completed`, `disconnect_completed`, `deauthorize_completed`, `sign_messages_completed`, `sign_transactions_completed`, `sign_and_send_completed`."

`deauthorize_completed` is in the list. The result Dictionary's 4-key shape is NOT the 6-key auth-success shape (DD-2-2-4) and NOT the 2-key sign-success shape (DD-3-1-5). It is a unique 4-key shape per `arch.md:669`.

### A-14 10-key error envelope [LOCKED]
For mwa_error emissions on the deauthorize path (catastrophic local failures only — see DD-4-1-3): `{request_id, code, message, user_message, layer, source_method: "deauthorize", correlation_id, retryable, cause, developer_details}`. `source_method` MUST be the literal string `"deauthorize"` (parallel to `"disconnect"`, `"reauthorize"`, `"sign_messages"`).

### DD-15 [LOCKED — `arch.md:1172`]
Single `mwa_error` signal (NOT per-operation error signals). The deauthorize path uses `mwa_error` only for catastrophic local failures (e.g. `clearOnLogout` itself crashing). Remote revoke failure does NOT route to `mwa_error` — it routes to `deauthorize_completed{remote_revoke_succeeded: false, warning: "remote_unreachable"}` per the AC-2 contract.

### DD-4-1-1 [LOCKED] — `try { remote } catch { warn } finally { wipe }` is the canonical structure; remote failure is a SUCCESSFUL deauthorize
The deauthorize path emits **`deauthorize_completed`** (not `mwa_error`) on remote-revoke failure. Remote failure is treated as a "successful" deauthorize from the user's perspective — the local state is consistent (wiped); the user can re-authorize against any wallet again. The `warning: "remote_unreachable"` field signals to the application that the remote-side `auth_token` may still be valid (and could be silently rotated by the wallet on next encounter), but for the local device the deauthorize is complete and binding. This is consistent with the spec.md FR-5 contract: deauthorize is a CLIENT-SIDE finality operation; the remote revoke is best-effort.

### DD-4-1-2 [LOCKED] — Local wipe = wipe-all-cache-records-for-identity (per DD-4-1-6) + `MwaSessionState.clear()`, in that order
1. **On-disk wipe via DD-4-1-6 listAllKeys filter** — for each `CacheKey` in `secureTokenStore.listAllKeys()` whose `identityUri` matches `sessionState.getIdentityUri()` (a String 4-tuple field), call `secureTokenStore.deleteToken(matchingKey)`. The on-disk record is the higher-stakes wipe (it survives process death; `MwaSessionState` does not). If the process were to crash between step 1 and step 2, the on-disk token(s) are gone and `MwaSessionState` will be re-initialized empty on next start. The reverse ordering would leave a window where the in-memory state is wiped but the on-disk token remains — a security regression.
2. `sessionState.clear()` second — full clear (NOT `clearOnLogout()`). Verified extant: `MwaSessionState.kt:241 fun clear()` exists alongside `:270 fun clearOnLogout()`. `clearOnLogout()` is the disconnect-path wipe that PRESERVES caller identity for silent reauth (Story 2-3 DD-2-3-1); the deauthorize path wipes EVERYTHING because the user's intent is "this device is no longer trusted with this wallet." Identity fields will be re-populated on the next `MWA.connect()` call.

### DD-4-1-3 [LOCKED] — `try` block wraps ONLY the remote revoke; wipe-crashed path goes to `mwa_error`; flag-based post-finally branch (no return from finally)
The `try { ... } catch { ... } finally { ... }` envelope wraps the remote-revoke call ONLY, not the wipe. Rationale: if the wipe itself crashes (e.g. `secureTokenStore.deleteToken` throws on a corrupted EncryptedSharedPreferences), the local wipe is INCOMPLETE — emitting `deauthorize_completed{local_cache_cleared: true}` would be a lie.

The wipe sits inside the `finally` block but is itself wrapped in a SECOND `try { wipe } catch (ex: Throwable) { mwa_error + set wipeCrashed flag }` (mirroring the disconnect pattern at `GDExtensionAndroidPlugin.kt:467-481`). If the wipe crashes, the path emits `mwa_error{code: PROTOCOL_ERROR, source_method: "deauthorize", developer_details: "wipe crashed: ${ex.javaClass.simpleName}"}` and the post-finally branch reads the flag to skip the `deauthorize_completed` emission. The terminal-signal invariant (one of `deauthorize_completed | mwa_error | mwa_timeout | mwa_cancelled_lifecycle` per request) is preserved.

**Pseudocode (the structural pattern T6 enforces):**
```kotlin
var remoteSucceeded = false
var wipeCrashed = false
val authTokenForRevoke = sessionState.getAuthToken()
val identityUri = sessionState.getIdentityUri()

if (authTokenForRevoke == null || identityUri.isEmpty()) {
    // DD-4-1-4 idempotence path — vacuous success
    remoteSucceeded = true
} else {
    try {
        mwaClient.deauthorize(sender, identity, authTokenForRevoke)  // suspending
        remoteSucceeded = true
    } catch (e: Throwable) {
        remoteSucceeded = false
        SdkLog.w(TAG, requestId) { "remote revoke failed: ${e.javaClass.simpleName}" }
    } finally {
        try {
            // DD-4-1-6 multi-key wipe
            secureTokenStore.listAllKeys()
                .filter { it.identityUri == identityUri }
                .forEach { secureTokenStore.deleteToken(it) }
            sessionState.clear()
        } catch (ex: Throwable) {
            wipeCrashed = true
            SdkLog.e(TAG, requestId) { "mwaDeauthorize wipe crashed: ${ex.javaClass.simpleName}" }
            nativeBridge.postMwaError(buildErrorJson(
                requestId = requestId,
                error = MwaError.ProtocolError,
                developerDetails = "wipe crashed: ${ex.javaClass.simpleName}",
                layer = "kotlin",
                cause = ex.javaClass.simpleName,
                sourceMethod = "deauthorize",
            ).toString())
        }
    }
}

// Post-finally branch — flag-based, NO `return@withContext` from inside finally.
// This avoids Kotlin's finally-swallow-in-flight-exception gotcha (see C-4-1-F): if
// the outer catch's SdkLog.w were to throw (defensively possible), an early return
// from inside the inner finally would silently consume the SdkLog throw. The flag
// pattern is structurally cleaner and T6's source-grep test still passes (deleteToken
// + sessionState.clear are inside the finally-block byte range).
if (!wipeCrashed) {
    val resultJson = JSONObject().apply {
        put("request_id", requestId)
        put("remote_revoke_succeeded", remoteSucceeded)
        put("local_cache_cleared", true)
        put("warning", if (!remoteSucceeded) "remote_unreachable" else "")
    }.toString()
    nativeBridge.postDeauthorizeCompleted(requestId, resultJson)
}
```

The CAS register/tryTerminate from 2-3 wraps this entire `withContext(mainDispatcher)` block (omitted from pseudocode for clarity).

### DD-4-1-4 [LOCKED] — Idempotence path: skip remote revoke if no token OR no identityUri in MwaSessionState
The first step of `mwaDeauthorize` (after InflightMap register + CAS) is `val authTokenForRevoke: SecretString? = sessionState.getAuthToken()` AND `val identityUri: String = sessionState.getIdentityUri()`. If `authTokenForRevoke == null` OR `identityUri.isEmpty()`, the path SKIPS the remote-revoke try/catch entirely (sets `remoteSucceeded = true` vacuously per AC-4) and proceeds directly to the `finally`-wipe (which is also a no-op when identityUri is empty: `listAllKeys().filter { ... }` returns empty list, no deleteToken calls; sessionState.clear() is idempotent). Rationale:
- No token: remote call would fail with a clientlib validation error — that's a synthetic "remote_unreachable" warning that misleads the user. Vacuous-success correctly reports "there was nothing to revoke".
- No identityUri: the listAllKeys filter cannot match anything, so the multi-key wipe is empty. This is the cold-start-deauthorize-without-prior-connect-or-reauth case (not a typical flow but defensively handled per DD-4-1-6).

### DD-4-1-5 [LOCKED] — `MwaClient.deauthorize` signature: 3 args via `MwaClient.kt:68`
```kotlin
suspend fun deauthorize(sender: ActivityResultSender, identity: ConnectionIdentity, authToken: SecretString): MwaResult<Unit>
```
The `MwaClient` interface already exposes `deauthorize` (verified: `MwaClient.kt:68`, `FakeMwaClient.kt:92-93`, `MwaClientImpl.kt:77-93`). The plugin call site reuses the established Story 3-1 pattern of reconstructing `ConnectionIdentity` from `MwaSessionState` preserved-across-disconnect identity fields (DD-3-1-8 inheritance). NO new clientlib seam is added.

### DD-4-1-6 [LOCKED] — `currentKey` is found via `store.listAllKeys()` filter on `identityUri`; multi-key wipe by design
**Problem:** `CacheKey(cluster: String, chainId: String, walletPackage: String, identityUri: String)` requires four caller-side fields. Of these:
- `cluster: String` — NOT in `MwaSessionState`. `MwaSessionState.clusterName: String` is display-only and NOT a `CacheKey` input; `MwaSessionState.cluster: Int` is a different domain (an enum-int, not the human-readable cluster name).
- `chainId: String` — NOT a field of `MwaSessionState` at all.
- `walletPackage: String` — NOT a field of `MwaSessionState`. Captured as a local val in `handleConnectSuccess` / `handleReauthSuccess` (`auth.walletPackage.orEmpty()`) and embedded in `CacheKey` + `CacheRecord` only — never written back to session state.
- `identityUri: String` — YES, preserved across disconnect via Story 2-3 DD-2-3-5; available as `sessionState.getIdentityUri()`.

`MwaSessionState.getConnectedKey(): ByteArray?` returns the raw 32-byte public key, NOT a `CacheKey` input. There is NO derivation function from public-key bytes to a `CacheKey`.

`mwaDeauthorize(requestId: String)` has NO JNI args for cluster/chainId/identityUri — unlike `mwaReauthorize(requestId, identityJson, cluster, chainId, timeoutMs)` which DOES — so even Story 2-2's 3-tuple filter resolution path doesn't directly apply.

**Resolution:** Filter `secureTokenStore.listAllKeys()` by `it.identityUri == sessionState.getIdentityUri()` (single-field equality, not a 3-tuple AND filter). Delete ALL matching records. Semantics: "deauthorize wipes ALL cached records for this identity, across clusters and across walletPackages." This aligns with the security threat model: "this device is no longer trusted with this wallet." A user who switches clusters / wallet packages with the same identity has implicitly trusted multiple records to the same caller; deauthorize revokes all of them.

**Cold-start guard:** if `sessionState.getIdentityUri().isEmpty()` (no prior connect/reauth in this process), the filter returns empty; remote revoke is also skipped (DD-4-1-4); the deauthorize_completed signal still fires with both flags true vacuously (DD-4-1-4 vacuous-success).

**Multi-record wipe sequencing:** sequential `deleteToken(key)` calls. Each is idempotent individually (`SecureTokenStore.kt:73-76` calls `prefs.edit().remove(...).apply()` which is a no-op on missing keys). If one throws (corrupted prefs), DD-4-1-3's defensive try/catch routes to `mwa_error` and the post-finally branch skips `deauthorize_completed` (terminal-signal invariant preserved).

**Why not extend the JNI shim with cluster/chainId args (option 2)?** Would require updating `MobileWalletAdapter::deauthorize()` C++ binding signature (currently no-arg per `mobile_wallet_adapter.cpp:174-182`), the GDScript facade method, the JNI dispatcher, and the C++ → Kotlin shim. Cross-cutting scope expansion for marginal benefit (pinpoint deletion vs broad wipe) — and broad wipe is the SAFER security default. Rejected.

**Why not maintain a `currentSessionCacheKey: CacheKey?` field on MwaSessionState (option 4)?** Would introduce a new state field with its own invariant (must be populated at connect/reauth, wiped at disconnect/deauth). Cross-story freeze edit on `MwaSessionState.kt`. Rejected.

**Why not hand-roll `clusterIntToString` + `derivedChainId` (option 3)?** Hacky parallel mapping that has to stay in sync with the auth path. Rejected.

## Dev Notes

### Story 2-3's CAS-first contract is inherited verbatim
The `mwaDeauthorize` body must follow 2-3's CAS-first ordering exactly:
1. `inflightMap.register(requestId, clock())` — reserve slot; PROTOCOL_ERROR on duplicate.
2. `inflightMap.tryTerminate(requestId)` — win the terminal-signal CAS BEFORE any state mutation. If lost (defense-in-depth; not expected on deauthorize path since there's no competing watchdog timeout — see "Watchdog scoping decision" below): increment `diagnostics.lateResult`, log, return.
3. **NEW vs 2-3:** capture `authTokenForRevoke = sessionState.getAuthToken()` and `identityUri = sessionState.getIdentityUri()`. If `authTokenForRevoke == null` OR `identityUri.isEmpty()`: enter idempotent path per DD-4-1-4 (set `remoteSucceeded = true` vacuously, jump to finally where the listAllKeys filter is empty and is itself a no-op).
4. **Multi-key wipe per DD-4-1-6:** in the `finally` block, do `secureTokenStore.listAllKeys().filter { it.identityUri == identityUri }.forEach { secureTokenStore.deleteToken(it) }`, then `sessionState.clear()`. Both inside the inner defensive `try { wipe } catch (ex) { wipeCrashed = true; mwa_error }` envelope (DD-4-1-3).
5. **Flag-based post-finally branch (no return-from-finally per C-4-1-F):** after the outer try/catch/finally completes, `if (!wipeCrashed) emit deauthorize_completed`. NEVER use `return@withContext` from inside the inner finally-catch — that would swallow any in-flight exception from the outer catch (e.g., a defensive `SdkLog.w` throw).
6. Build resultJson `{request_id, remote_revoke_succeeded, local_cache_cleared: true, warning}` and `nativeBridge.postDeauthorizeCompleted(requestId, resultJson.toString())`.

### Watchdog scoping decision — NO watchdog on deauthorize path
Unlike `mwaConnect` / `mwaReauthorize` / `mwaSignMessages` which use `effectiveWatchdog(timeoutMs)` + `withTimeout`, `mwaDeauthorize` does NOT install a watchdog. Rationale:
- AC-1 says "within ≤3 s" but this is a soft latency budget for the happy path, NOT a contractual timeout that should fire `mwa_timeout`.
- The only operation that could hang is the remote revoke (`mwaClient.deauthorize(...)`), and the AC-2 contract REQUIRES that any failure of the remote (including hangs) be classified as `remote_unreachable` and emit `deauthorize_completed` (NOT `mwa_timeout`). A watchdog firing `mwa_timeout` would VIOLATE AC-2.
- clientlib-ktx 2.0.3 enforces its own per-call timeout internally (verified: `MwaClientImpl.deauthorize` line 81-93 wraps `adapter.transact(sender) { _ -> deauthorize(tokenStr) }` which has clientlib-ktx's internal session timeout). If clientlib hangs beyond reasonable latency, it surfaces as a `MwaResult.Failure` which the deauthorize path classifies as `remote_unreachable`.
- DD-4-1-1 + AC-2 together imply: "any non-Success outcome from `mwaClient.deauthorize` becomes `remote_revoke_succeeded: false, warning: remote_unreachable`." There is no path on which deauthorize emits `mwa_timeout`.

## Guardrails

1. **Do NOT introduce a `runDeauthorizeOp` helper.** DD-3-1-9 scopes the runOp pattern to signing. Deauthorize is a one-shot path; use the disconnect pattern (inline body) directly.
2. **Do NOT use `MwaSessionState.clearOnLogout()` in `mwaDeauthorize`.** That's the disconnect-path wipe. Deauthorize wipes EVERYTHING via `MwaSessionState.clear()`. DD-4-1-2 enforces this.
3. **Do NOT route remote-revoke failure to `mwa_error`.** AC-2 contract requires `deauthorize_completed{remote_revoke_succeeded:false}`. DD-4-1-1.
4. **Do NOT skip the wipe on remote-revoke failure.** AC-NFR-SEC-5 / AC-2 contract: local wipe ALWAYS runs. DD-4-1-3.
5. **Do NOT emit two terminal signals.** InflightMap tryTerminate-CAS preserves the one-of-{deauthorize_completed | mwa_error} invariant. The wipe-crash defensive path emits `mwa_error` and DOES NOT also emit `deauthorize_completed` — enforced via the `wipeCrashed` flag + post-finally `if (!wipeCrashed)` branch.
6. **Do NOT `return@withContext` from inside the inner finally-catch.** The flag-based pattern is mandatory (DD-4-1-3 / C-4-1-F resolution). Returning from a finally block can swallow in-flight exceptions from the outer catch.
7. **Do NOT install a watchdog timeout on the deauthorize path.** See "Watchdog scoping decision" above. AC-2 requires remote failure to surface as `remote_unreachable`, NOT `mwa_timeout`.
8. **Do NOT construct a `CacheKey` from `MwaSessionState.cluster: Int` or `clusterName: String`.** Per DD-4-1-6, these are NOT valid `CacheKey.cluster: String` inputs. Use the listAllKeys-filter-on-identityUri pattern instead.
