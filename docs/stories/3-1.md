## User Story
As a Godot game developer, I want to call `MWA.sign_messages([bytes1, bytes2])` and receive back an array of signatures, so that my game can prove ownership or sign off-chain data.

### `runSigningOp` signature [plan.md:608 conceptual; impl signature locked below per DD-3-1-3]

plan.md:608 conceptual signature:
```kotlin
suspend fun <T> MwaAndroidPlugin.runSigningOp(
    op: SigningOp,
    requestId: String,
    block: suspend MobileWalletAdapterClient.() -> T,
): Result<T, MwaError>
```

**Impl signature (per DD-3-1-1, DD-3-1-2 narrative, DD-3-1-3 watchdog evolution, Revision 2 C-3-1-W single-wrap shape):**
```kotlin
@VisibleForTesting
internal suspend fun <X> runSigningOp(
    op: SigningOp,
    requestId: String,
    timeoutMs: Long,
    block: suspend MwaClient.() -> MwaResult<X>,
): MwaResult<X>
```

**Single-wrap shape (Revision 2 C-3-1-W fix):** the block returns `MwaResult<X>` (NOT a generic `T`); the helper inspects `MwaResult.Failure` cases internally and routes them through `nativeBridge.postMwaError(buildErrorJson(... error.code ... op.sourceMethod ...))`. The CALL SITE (e.g., `mwaSignMessages`) only pattern-matches on `MwaResult.Success(payload)` to route to `handleSignMessagesSuccess`; all error paths are emitted by the helper. This single-wrap design avoids the v1 double-wrap (`MwaResult<MwaResult<X>>`) which would have left wallet-level Failures (e.g., USER_CANCELED from FakeMwaClient `sign_messages_user_canceled.json`) unrouted at the call site. AC-4's "USER_CANCELED → mwa_error{code=USER_CANCELED}" assertion only passes under the single-wrap shape. Stories 3-2 / 3-3 inherit the single-wrap shape unchanged (they also return `MwaResult<X>` from their respective `MwaClient` calls).

**Visibility (Revision 2 C-3-1-X fix):** the helper is `@VisibleForTesting internal` (NOT `private` as v1 spec stated). Reason: T1 tests #5 and #6 invoke the helper directly across packages (test in `com.godotengine.godot_solana_sdk.mwa.plugin`, helper in `plugin.walletadapterandroid`). Kotlin `private` is file-private; `internal` is module-private; `@VisibleForTesting` is the codebase's standard idiom for cross-package test access (parallel to Story 2-1's `inflightMapForTest()` and `dispatcher_for_testing` patterns). Logged as D-3-1-X (Rule 1 — visibility relaxed for cross-package test access).

**This signature is consumed by Stories 3-2 (sign_transactions) and 3-3 (sign_and_send + breadcrumb).** T2 lands the IMPL signature; the plan.md:608 conceptual signature differs in three Rule-1-deviation ways (logged in `docs/concerns.md` at T7 per the C-3-1-L sub-step):

- **D-3-1-1 (Rule 1 — receiver class naming):** plan.md:608's `MwaAndroidPlugin.` extension-style receiver becomes a `private suspend fun` member of `GDExtensionAndroidPlugin` (the actual production class). The plan.md "MwaAndroidPlugin" is the conceptual class name; the codebase uses `GDExtensionAndroidPlugin`. T2 lands as a `private suspend fun` member with the same access to `inflightMap`, `scope`, `mainDispatcher`, `nativeBridge`, `storeProvider`, `mwaClientFactory`, `senderProvider`, `clock`, `effectiveWatchdog`, `buildErrorJson`, `parseIdentity` as `mwaReauthorize`.
- **D-3-1-2 (Rule 1 — block receiver type):** plan.md:608's `block: suspend MobileWalletAdapterClient.() -> T` (clientlib-ktx receiver) becomes `block: suspend MwaClient.() -> T` (SDK seam receiver). Rationale: `MwaClient` is the test seam — `FakeMwaClient` implements it; `MobileWalletAdapterClient` is internal to clientlib-ktx (only reachable inside `MwaClientImpl`). Using the clientlib-ktx receiver would break test injection of `FakeMwaClient` and force AC-2's unit test through a real-clientlib path. The DD-3-1-2 narrative below LOCKS the receiver as `MwaClient`; T2 pseudocode below reconciles to `signMessages(...)` (5-arg `MwaClient` method) — NOT `signMessagesDetached(...)` which is `MobileWalletAdapterClient`-only.
- **D-3-1-3 (Rule 1 — return type):** plan.md:608's `Result<T, MwaError>` becomes `MwaResult<T>` (existing sealed class with `Success<T>(value, corrId)` / `Failure(error, corrId)` per `MwaClient.kt`). The codebase has no `Result<T, MwaError>` typealias today; introducing one for one helper would be churn. `MwaResult<T>` is the established return type for the `MwaClient` family — uniformity wins.
- **D-3-1-4 (Rule 1 — signature evolution: add `timeoutMs: Long` param):** plan.md:608 omits `timeoutMs`, but DD-3-1-3 mandates the helper passes the caller's `timeoutMs` through `effectiveWatchdog(timeoutMs)`. Three options were considered (see C-3-1-B below — hardcode default, thread-local, evolve signature); evolving the signature is cleanest. Stories 3-2 / 3-3 inherit the evolved signature — they take the same `timeoutMs: Long` from their JNI shim.

All four deviations are Rule 1 (style/naming/type alignment with codebase), NOT Rule 3 (interface contract change) — the conceptual *contract* of `runSigningOp` (CAS, watchdog, error mapping, op-tag) is preserved; only the surface shape evolved to fit the codebase's existing types.

### DD-3-1-1 [LOCKED] — `runSigningOp` is a private member function on `GDExtensionAndroidPlugin`, not a top-level extension
The plan.md signature is `suspend fun <T> MwaAndroidPlugin.runSigningOp(...)` (extension function shape). The codebase uses `private suspend fun ...` member functions for internal helpers (`handleReauthSuccess`, `handleTokenExpired`, `emitFailureReauth`, etc. — see `GDExtensionAndroidPlugin.kt:647,753,780`). DD-3-1-1 aligns with the codebase: `runSigningOp` is a `private suspend fun` member of `GDExtensionAndroidPlugin` with the impl signature pinned in the section above. **Rationale:** consistent with the existing helper convention; no need to expose as an extension since it's internal to the plugin module. Stories 3-2 and 3-3 reference it as a sibling private member.

### DD-3-1-2 [LOCKED] — Block receiver = `MwaClient` (the SDK seam), NOT `MobileWalletAdapterClient`
plan.md:608 declares `block: suspend MobileWalletAdapterClient.() -> T`, but `MobileWalletAdapterClient` is the underlying clientlib-ktx type — only reachable inside `MwaClientImpl` (the real impl). `MwaClient` is the SDK's abstraction seam, implemented by both `MwaClientImpl` (production) AND `FakeMwaClient` (tests). If `runSigningOp`'s block receiver is `MobileWalletAdapterClient`, then:
- `FakeMwaClient` cannot be injected — it implements `MwaClient`, not `MobileWalletAdapterClient`.
- AC-2 / AC-4 / T1 unit tests that drive scenarios via FakeMwaClient cannot compose with `runSigningOp` directly.
- The test seam established by Story 1-6 (DD-19 from architecture.md) breaks.

**DD-3-1-2 LOCKS the receiver as `MwaClient`.** The block lambda uses `MwaClient`-level APIs (`signMessages(sender, identity, authToken, messages, addresses)`, `signTransactions(sender, identity, authToken, transactions)`, `signAndSendTransactions(sender, identity, authToken, transactions, cluster)` per the existing 3-method surface in `MwaClient.kt:96-145`). Per-op-type polymorphism is achieved via the lambda body (the caller writes the op-specific call), NOT via the `SigningOp` enum or block receiver subtype.

T2 pseudocode (post-revision):
```kotlin
runSigningOp(SigningOp.SIGN_MESSAGES, requestId, effectiveMs) {
    // `this` is MwaClient (the abstraction seam — Fake or Impl).
    signMessages(senderProvider(), restoredIdentity, restoredAuthToken, messages, addresses)
}
```

NOT:
```kotlin
// REJECTED — signMessagesDetached is on MobileWalletAdapterClient (clientlib-ktx),
// not on MwaClient. FakeMwaClient cannot serve this receiver type.
runSigningOp(SigningOp.SIGN_MESSAGES, requestId, effectiveMs) {
    signMessagesDetached(messages.toTypedArray(), arrayOf(sessionState.getKey()))
}
```

**Rationale:** preserves the test seam (DD-19) and AC-2 / AC-4 unit-test composability. plan.md:608's `MobileWalletAdapterClient` is conceptual — the codebase's abstraction is `MwaClient`. Logged as D-3-1-2 (Rule 1 — block receiver type alignment).

### DD-3-1-3 [LOCKED] — `runSigningOp` signature evolved to take `timeoutMs: Long`; watchdog passes it through `effectiveWatchdog(timeoutMs)`
plan.md:608's signature has no `timeoutMs` parameter, but DD-3-1-3 mandates `runSigningOp` runs the block inside `withTimeoutOrNull(effectiveMs) { ... }` where `effectiveMs = effectiveWatchdog(timeoutMs)` (matching the auth-path pattern at `GDExtensionAndroidPlugin.kt:570-578`). Three options to resolve the gap:
1. **Add `timeoutMs: Long` to the signature** (selected) — `runSigningOp(op, requestId, timeoutMs, block)`. Each caller (`mwaSignMessages`, `mwaSignTransactions`, `mwaSignAndSend`) takes `timeoutMs` from its JNI shim and passes it through.
2. Hardcode `INTERNAL_WATCHDOG_DEFAULT_MS` — caller's `timeoutMs` is ignored. REJECTED: inconsistent with `mwaAuthorize` / `mwaReauthorize` watchdog behavior; breaks user override capability.
3. Thread-local — REJECTED: ugly, not idiomatic Kotlin coroutines, breaks structured concurrency.

**DD-3-1-3 LOCKS option 1.** The impl signature gains `timeoutMs: Long` as the third parameter (between `requestId: String` and `block: suspend MwaClient.() -> T`). Stories 3-2 and 3-3 inherit the evolved signature unchanged. Rationale: uniformity with the auth path; one source of truth for watchdog clamping behavior; user-provided `opts.timeout_ms` from GDScript propagates through the JNI shim → `runSigningOp` → `withTimeoutOrNull`. Logged as D-3-1-4 (Rule 1 — signature evolution: add `timeoutMs: Long` for DD-3-1-3 watchdog parity).

Do NOT introduce a sign-specific timeout default; if Story 5-2 or future feedback surfaces a need (e.g., hardware-wallet signing taking >120 s), that is an amendment.

### DD-3-1-4 [LOCKED] — `SigningOp` is an enum class, not a sealed class
plan.md:608 references `op: SigningOp` but does NOT define the type. Architecture §11 / §12 has no SigningOp definition. T2 introduces it as a Kotlin `enum class SigningOp { SIGN_MESSAGES, SIGN_TRANSACTIONS, SIGN_AND_SEND }` colocated with `runSigningOp` in `GDExtensionAndroidPlugin.kt` (or a sibling file `SigningOp.kt` in the same package). **Rationale:** the three values are closed and the helper does not carry op-specific payload data inside the SigningOp value (the op-specific data flows via the `block: suspend MwaClient.() -> T` lambda per DD-3-1-2). An enum is sufficient. **Consumed by:** mapping inside `runSigningOp` to the `source_method` string for `mwa_error` envelopes (`"sign_messages"`, `"sign_transactions"`, `"sign_and_send"`) — T2 lands the mapping as `val SigningOp.sourceMethod: String` extension property.

### DD-3-1-5 [LOCKED] — Success-payload builder
`buildSignSuccessJson(requestId: String, signedPayloads: List<ByteArray>): JSONObject` is a NEW small helper colocated with `buildSuccessJson` (file-private at the bottom of `GDExtensionAndroidPlugin.kt`, parallel to the existing `buildSuccessJson` at line 1080). ### DD-3-1-6 [LOCKED] — AC-3 NOT_CONNECTED preflight ordering: BEFORE InflightMap.register
AC-3 says "fires within 1 frame (no wallet UI)". Per Story 2-2 DD-2-2-2 precedent (CAS-first-then-wipe-then-emit) we COULD register first then check connection. **DD-3-1-6 LOCKS the opposite for sign_messages:** the connection check (`sessionState.getAuthToken() == null` OR equivalent) happens BEFORE `inflightMap.register`. Rationale: AC-3's "within 1 frame" + "no wallet UI" requires NO scope.launch — the preflight is purely synchronous (read sessionState on the calling thread, emit error, return). The InflightMap CAS register is reserved for ops that will actually execute a coroutine. A failed preflight that registered would leak an InflightMap slot until watchdog timeout. Pattern: `if (!isConnected()) { nativeBridge.postMwaError(buildErrorJson(... NOT_CONNECTED ...)); return }`. The AC-3 unit test asserts: (a) `mwa_error{NOT_CONNECTED}` fires, (b) `inflightMap.registerCount == 0` (no slot ever taken), (c) `mwaClientFactory()` was never invoked.

**Trade-off acknowledged:** this is asymmetric with mwaReauthorize (which registers BEFORE the cache lookup so even cache-miss errors emit through the CAS-gated path). The asymmetry is justified because reauth's cache-miss is a coroutine-scope decision (parsing identity JSON, async cache lookup) whereas sign_messages's not-connected preflight is a synchronous one-line read.

**Behavioral wrinkle (per C-3-1-J):** A caller who sends two `mwaSignMessages` calls with the SAME `requestId` while disconnected gets `mwa_error{NOT_CONNECTED}` for BOTH calls — NOT a `PROTOCOL_ERROR{cause=duplicate_request_id}` for the second. Reason: the InflightMap CAS register is bypassed by the preflight, so the duplicate-detection mechanism doesn't fire. This is acceptable: the caller's bug surfaces as `NOT_CONNECTED` regardless of duplicate-id, and the corrective action (call `connect()` first) is the same. Tests assert the `NOT_CONNECTED` outcome and `inflightMap.registerCount == 0`; they do NOT assert duplicate-id semantics in the disconnected branch (those semantics only apply when a coroutine is actually launched). The `duplicate_request_id` PROTOCOL_ERROR path is exercised only in the connected branch (T1 test #7).

### DD-3-1-8 [LOCKED] — `ConnectionIdentity` reconstruction via `MwaSessionState` preserved-across-disconnect identity fields (DD-2-3-5 inheritance)
`MwaClient.signMessages(...)` requires a `ConnectionIdentity` (per `MwaClient.kt:96-102`).     identityUri = Uri.parse(sessionState.getIdentityUri()),
    iconUri = Uri.parse(sessionState.getIconUri()),
    identityName = sessionState.getIdentityName(),

These fields are pinned at `connect` time (Story 2-1 T4 `handleSuccess` writes them), preserved across `disconnect` per Story 2-3 DD-2-3-5 (auth-only fields are wiped on logout; identity fields are NOT). ### DD-3-1-9 [LOCKED] — `runSigningOp` responsibility split: helper does CAS+watchdog+errors; call site does identity/cluster/auth-token reads
plan.md:236 AC-1 verbatim text says `runSigningOp` handles "connection-identity restore, cluster selection, auth-token restore, watchdog, and error translation via ErrorTaxonomy". The implementation reality is finer-grained:

| Responsibility | Owner |
|---|---|
| CAS register (InflightMap.register) | `runSigningOp` |
| Watchdog (`effectiveWatchdog(timeoutMs)` + `withTimeoutOrNull`) | `runSigningOp` |
| Client invoke (`block.invoke(client)` inside watchdog) | `runSigningOp` |
| Timeout signal emission (`emitTimeoutSign(requestId, watchdogMs, op)`) | `runSigningOp` |
| Error translation (map `MwaError.*` → `mwa_error{source_method=op.sourceMethod}` envelope, late_result handling, CAS tryTerminate gating) | `runSigningOp` |
| **Connection-identity restore** (`sessionState.getIdentity*()` reads + `ConnectionIdentity` construction) | **call site** (e.g., `mwaSignMessages`) |
| **Cluster selection** (`sessionState.getClusterName()` read) | **call site** |
| **Auth-token restore** (`sessionState.getAuthToken()` read) | **call site** |
| Op-specific client method (`signMessages(...)` vs `signTransactions(...)` vs `signAndSendTransactions(...)`) | **call site** (lambda body) |

The call-site does the sessionState reads and packs the values into the lambda closure; the helper invokes the closure inside its CAS+watchdog+error scaffolding. This split is necessary because (a) the lambda body needs op-specific arguments (`messages` vs `transactions` vs `(transactions, cluster)`) which the helper cannot know about, and (b) `sessionState` access is `mainDispatcher`-scoped and the helper's `withContext(mainDispatcher) { ... }` boundary is the single place where dispatcher hops happen.

**plan.md AC-1 verbatim text is aspirational; the impl reality is the table above.** Stories 3-2 / 3-3's call sites mirror sign_messages's structure (read sessionState, pack lambda, invoke `runSigningOp` with op + requestId + timeoutMs). Logged as a non-deviation (the contract holds: the helper IS the shared pipeline; the SHAPE of "shared" is the table, not a single closure that does everything).

      internal suspend fun <X> runSigningOp(
          op: SigningOp,
          requestId: String,
          timeoutMs: Long,
          block: suspend MwaClient.() -> MwaResult<X>,
      ): MwaResult<X> = withContext(mainDispatcher) {

              nativeBridge.postMwaError(buildErrorJson(requestId, MwaError.ProtocolError,
                  "duplicate requestId on ${op.sourceMethod} — caller must use fresh UUIDs per op",
                  "kotlin", "duplicate_request_id", op.sourceMethod).toString())
              return@withContext MwaResult.Failure(MwaError.ProtocolError, null)

                      client.block()  // DD-3-1-2: block receiver = MwaClient (Fake or Impl).
