# Story 3-3: sign_and_send + Pending-Submission Breadcrumb + pending_submission_found Signal

## User Story
As a Godot game developer, I want to call `MWA.sign_and_send([tx_bytes])` and receive tx signatures after the wallet submits to the cluster, AND be notified on next app launch if my last sign_and_send might have landed after a process death, so that players never silently lose transactions.

## Architecture Guardrails

### DD-3-3-A — Breadcrumb storage API extension on `SecureTokenStore` [LOCKED]

**Decision:** Extend `SecureTokenStore` with a 4-method parallel API for pending submissions, string-keyed under a `pending::` namespace prefix inside the SAME `EncryptedSharedPreferences` instance:

```kotlin
fun putPendingSubmission(requestId: String, breadcrumbDictJson: String)
fun getPendingSubmission(requestId: String): String?  // null if not present
fun listAllPendingSubmissions(): List<Pair<String, String>>  // (requestId, breadcrumbDictJson)
fun removePendingSubmission(requestId: String)
```

Internal storage: each entry keyed under `"pending::" + requestId` (raw string key, NOT hashed via `CacheKey.toHash()` — these are NOT auth tokens; they're recovery breadcrumbs). Values are JSON strings (NOT `CacheRecord` instances).

**Rationale:** Option A from the design space (vs Option B: separate `PendingSubmissionStore` collaborator; Option C: bend `CacheRecord` semantics). Option A wins on:
1. Minimal surface area (no new ctor wiring; no new collaborator injection across the plugin's `@VisibleForTesting` ctor)
2. Parallel to existing `getOrCreatePerInstallSalt` special-case namespace inside the same prefs (precedent for non-CacheRecord entries — `SecureTokenStore.kt:107-129`)
3. Single Tink/Keystore failure surface (corruption affects ALL prefs entries uniformly; one fail-closed `withStorageOrReauthRequired` wrap covers all disk-touch sites)
4. `listAllKeys()` already iterates `prefs.all` (`SecureTokenStore.kt:97`); `listAllPendingSubmissions` mirrors with a `startsWith("pending::")` filter

**Forbidden alternatives:**
- DO NOT introduce a separate `PendingSubmissionStore` class — adds ctor surface + doubles the keystore-corruption blast radius
- DO NOT bend `CacheRecord` semantics with sentinel `walletPackage = "__pending__"` — breaks type contracts and pollutes `listAllKeys()` for the auth-cache consumers

### DD-3-3-B — Breadcrumb write-ahead ordering [LOCKED]

**Decision:** Write-then-call. The breadcrumb is written to disk (via `withStorageOrReauthRequired { putPendingSubmission(...) }`) BEFORE `runSigningOp` invokes the client lambda. If the breadcrumb write fails (StorageCorruptException), emit `reauth_required` per DD-4-3-1.a fail-closed semantics; do NOT proceed to wallet handshake.

**Rationale:**
- Locks the AC-2 invariant cleanly ("when clientlib.transact begins, then SecureTokenStore contains the entry") — write completes on the calling thread before the wallet round-trip is even initiated
- Parallels the `withStorageOrReauthRequired` wrapper pattern from Story 4-3 T2 — same fail-closed contract for the disk-touch site
- Cheaper failure mode: if the breadcrumb write fails, no wallet round-trip is wasted (the user must reauth before the next op anyway)

**Forbidden alternative:**
- DO NOT use call-then-write (invoke `clientlib.signAndSendTransactions` first, write breadcrumb in the lambda's first await) — leaves a window where the wallet has started but the breadcrumb is missing, breaking AC-2's "when clientlib.transact begins" timing invariant

### DD-3-3-C — Cleanup semantics across all 4 terminal signal paths [LOCKED]

**Decision:** Hybrid — cleanup helper invoked from 3 distinct call sites:

1. **Success path:** `handleSignAndSendSuccess(requestId, result: SignAndSendResult)` — calls `removePendingSubmission(requestId)` BEFORE emitting `sign_and_send_completed` (preserves AC-3's "before sign_and_send_completed fires" ordering).
2. **Error/timeout path:** `cleanupBreadcrumb(requestId)` private helper invoked from the `runSigningOp` Failure branch. Wraps `removePendingSubmission` in a try/catch — on cleanup failure, increment `diagnostics.cleanupFailedCount` + `SdkLog.w` and continue (DD-4-1-3 wipe-crashed-flag pattern: a cleanup failure on disk corruption MUST NOT abandon the user's already-emitted error signal; the breadcrumb survives to be cleaned up on next launch via AC-5 scan).
3. **Lifecycle cancellation path:** Story 5-3's lifecycle observer (when it lands) iterates inflight requests on shutdown and invokes `cleanupBreadcrumb(requestId)` synchronously per inflight slot. This is OUT OF SCOPE for Story 3-3 — Story 3-3 lands the cleanup helper signature; Story 5-3 wires the lifecycle invocation. The androidTest in T5 covers the success + error paths only; the cancellation path is exercised by Story 5-3's androidTests.

**Forbidden alternative:**
- DO NOT use try/finally around the entire `mwaSignAndSendTransactions` body — the lifecycle-cancellation path (mwa_cancelled_lifecycle, fired from Story 5-3's lifecycle observer) runs OUTSIDE this method's call stack; finally would not catch it. The hybrid approach with an explicit cleanup helper is the right tier-decomposition.

### DD-3-3-D — `pending_submission_found` scan trigger [LOCKED]

**Decision:** Dedicated `scanPendingSubmissions(identityUri: String)` private helper invoked from BOTH `handleConnectSuccess` AND `handleReauthSuccess` AFTER the success signal fires. Identity-scoped: filter `listAllPendingSubmissions()` by matching `breadcrumbDictJson.identity_uri == identityUri`. Emit `pending_submission_found` for each matching entry, THEN call `removePendingSubmission(requestId)` per entry (one-shot per AC-5).

**Rationale:**
- Identity-scoped avoids cross-identity bleed (a stale breadcrumb from identity A doesn't surface on identity B's connect)
- Post-signal ordering preserves the connect/reauth happy path's terminal-signal invariant (`connect_completed` / `reauthorize_completed` fires first; `pending_submission_found` is a SEPARATE 1-param Dictionary signal, not blocking the auth flow)
- One-shot semantics (remove after emit) prevents duplicate notifications across reconnects

**Forbidden alternatives:**
- DO NOT inline the scan into `handleConnectSuccess` / `handleReauthSuccess` — duplicates code and breaks the "one helper, two call sites" parallelism
- DO NOT scan ALL pending entries (cross-identity) — risks bleeding breadcrumbs across the identity-isolation boundary that Story 2-1 + Story 2-2's identity-scoped key-derivation enforces

### DD-3-3-E — `pending_submission_found` signal arity + payload shape [LOCKED]

### DD-3-3-F — `tx_preview_hashes` shape [LOCKED]

**Decision:** Lowercase hex string per transaction, full 64-char (32-byte / 256-bit) SHA-256 digest. Computed in Kotlin as:

```kotlin
val digest = MessageDigest.getInstance("SHA-256").digest(tx)
val hex = digest.joinToString("") { "%02x".format(it) }
```

The plan-spec language `sha256(tx)[:32]` is Python-slice notation; `[:32]` over a `bytes(32)` value is the full digest. Lowercase hex format is the canonical Solana convention (matches Phantom / Solflare debug-log output for transaction hashes).

**NOT base64**, **NOT base58**, **NOT truncated below 64 chars**.

### DD-3-3-G — DD-4-3-1.a fail-closed wrapper inheritance — write-vs-cleanup asymmetry [LOCKED]

**Decision (resolves CR-3-2-W reconciliation for 3-3):** Wrap the breadcrumb-WRITE site with `withStorageOrReauthRequired`; do NOT wrap the breadcrumb-CLEANUP sites.

Specifically:
- **Wrap (fail-closed):** `putPendingSubmission(...)` invocation in `mwaSignAndSendTransactions` body BEFORE `runSigningOp`. On `StorageCorruptException`, the wrapper emits `reauth_required` (5-key payload per DD-4-3-1.b) and returns — the wallet round-trip is abandoned.
- **Wrap (fail-closed):** `listAllPendingSubmissions()` invocation in `scanPendingSubmissions(identityUri)` invoked from `handleConnectSuccess` / `handleReauthSuccess`. On `StorageCorruptException`, emit `reauth_required` and abandon the scan; the connect/reauth success signal HAS ALREADY FIRED (per DD-3-3-D ordering), so the user observes a successful auth followed by a `reauth_required` lifecycle warning — acceptable per AC-5's best-effort scan semantics.
- **Do NOT wrap:** `removePendingSubmission(...)` invocations in `handleSignAndSendSuccess` (success path), `cleanupBreadcrumb` helper (error/timeout path), and the per-entry one-shot cleanup inside `scanPendingSubmissions`. On `StorageCorruptException` here, we follow the DD-4-1-3 wipe-crashed-flag pattern: increment `diagnostics.cleanupFailedCount`, log via `SdkLog.w`, and continue. The breadcrumb survives to be cleaned up on the next launch's scan; the user's terminal signal (sign_and_send_completed / mwa_error / mwa_timeout) HAS ALREADY FIRED or will fire next.

**Rationale:**
- Symmetric defensive branches need symmetric tests (universal retrospect rule, 2026-04-27): write-failure and read-failure both need the fail-closed treatment because both gate user-visible op outcomes. Cleanup failure does NOT gate user-visible outcomes — the op already terminated.
- Inverts CR-3-2-W (Story 3-2's "signing-path is in-memory-only" tradeoff) ONLY for the disk-touch sites of `mwaSignAndSendTransactions`; preserves CR-3-2-W's rationale for `mwaSignMessages` / `mwaSignTransactions` which remain in-memory-only.
- Three explicit asymmetric branches (write/scan/cleanup) each need their own test case at the unit tier per the symmetric-defensive-branches rule. T1 covers all three.

### Story-spec ambiguity resolutions

- **AC-1 `submitted_at: int`** — clarified at story-creation time as Long (epoch milliseconds). The plan-spec calls it `int`; in Kotlin/JVM, the proper type is `Long` (epoch-ms exceeds Int.MAX_VALUE for any timestamp post-1970+24days). The dictionary value at the GDScript boundary is a Godot `int` (which is 64-bit). T2's `buildSignAndSendSuccessJson` uses `Long`; the JSON serialization handles the conversion.
- **AC-1 `confirmation_status: "submitted"`** — confirmed as a string literal in T2's `buildSignAndSendSuccessJson(... confirmationStatus: String)` arg. Future Story 5-x may extend with `"confirmed"` / `"finalized"` / `"failed"` if a confirmation-tracking surface is added. Today, always `"submitted"`.
- **AC-2 `tx_count` vs `transactions.size`** — same value; the breadcrumb's `tx_count` is `transactions.size` at the call site. Logged as a sanity-check in T1 case 2.
