# Story 1-5: MobileWalletAdapter C++ Node Skeleton + Thread-Hop Unit Test

## User Story
As an SDK contributor, I want a `MobileWalletAdapter` C++ GDExtension node that registers all 11 signals and delegates every method to `MwaAndroidBridge` via `GodotMainDispatcher`, so that the Godot engine sees the full signal surface from day one and our thread-hop rule is testable.

## Context
- Wave 2 closed with the Kotlin plugin surface (Story 1-6) and the C++ bridge + dispatcher (Story 1-4). Both landed with ZERO script-exposed surface — GDScript cannot yet see MWA. Story 1-5 is the only story in Wave 3 and wires the Godot-facing node (arch §2.2 component B). Op methods delegate to the no-op bridge (non-Android) or JNI stub (Android) — real flows arrive in Story 2-1 (`docs/plan.md:160` declares 1-5 a prerequisite).
- **First C++ test-tier story.** Per A-8, local C++ tests have been inspection-only through 1-2, 1-3, 1-4, 1-6. Task 4 lights up `scons tests` and imports Story 1-2's deferred `secret_string_test.cpp` (closes CR-8).

## Acceptance Criteria
*(verbatim from `docs/plan.md` §Story 1-5 lines 121-124 — do not rephrase)*

- **AC-1 (happy):** *Given* `MobileWalletAdapter` GDExtension node is registered, *when* the engine loads on Android, *then* all 11 signals (`connect_completed`, `reauthorize_completed`, `disconnect_completed`, `deauthorize_completed`, `sign_messages_completed`, `sign_transactions_completed`, `sign_and_send_completed`, `mwa_error`, `mwa_timeout`, `mwa_cancelled_lifecycle`, `reauth_required`) and 13 methods (7 ops + 4 state getters + 2 utility) bind successfully; calling any op method returns within ≤16 ms.
- **AC-2 (thread-hop test):** *Given* a host-platform unit test (Linux/Windows/macOS) with `MockMwaAndroidBridge`, *when* the test invokes the mock's callback from a `std::thread` (not Godot main), *then* the corresponding signal fires on Godot main thread within 1 frame AND the count of direct `emit_signal` calls not routed through `GodotMainDispatcher` is 0 (verified by source grep).

## Architecture Guardrails

### §2.2 Component B + DD-22 [LOCKED] — ownership & thread-marshalling
> Ref: `docs/architecture.md#2.2 Container View` — component B: bind GDScript calls, marshal off-thread callbacks via `call_deferred`, delegate to `MwaAndroidBridge`.
> Ref: `docs/architecture.md#7.1 Thread Marshalling Primitive` — every GDScript signal crosses to Godot main via `call_deferred` in `GodotMainDispatcher`. No direct `emit_signal` outside `src/mwa/godot_main_dispatcher.cpp`; Task 6 adds the grep ban.

Node OWNS one `GodotMainDispatcher` (ctor target = `this`) and one `std::unique_ptr<MwaAndroidBridge>` heap-allocated via `MwaAndroidBridge::create(&dispatcher_)`. Declaration order: dispatcher FIRST, bridge SECOND — destruction reverses so the bridge's dtor cannot post to a freed dispatcher. **`<memory>` in the header is permitted here.** DD-26's `no-STL` rule is scoped to the ABI-boundary headers under `src/mwa/include/` (enforced by the `clang_tools.yml` `header-lint-mwa` allow-list); `include/mwa/mobile_wallet_adapter.hpp` is the broader public script-exposed tier (same as `include/wallet_adapter/wallet_adapter.hpp`, `include/keypair.hpp`, etc.) where `<string>`, `<vector>`, `<memory>` are already in routine use by existing headers.

### D-2 [LOCKED] — test-mode deferred-drain harness
- `scons tests` target passes `-DMWA_TESTING=1` to the compiler (CCFLAGS). Production builds (`scons platform=android target=template_release`, etc.) do NOT define `MWA_TESTING`; runtime behavior is unchanged.
- `GodotMainDispatcher` gets a new public method `void drain_for_testing()` gated by `#ifdef MWA_TESTING`. Also gated: internal `std::vector<Pending> pending_` queue + `std::mutex drain_mutex_` field (`struct Pending { godot::String signal_name; godot::Dictionary payload; };`).
- Under `MWA_TESTING`, `GodotMainDispatcher::post(signal_name, payload)` appends `{signal_name, payload}` to `pending_` (mutex-guarded) instead of calling `target->call_deferred("emit_signal", ...)`. The `call_deferred` path is skipped in test mode — the engine is not present to drain it.
- `drain_for_testing()` takes the mutex, pops each queued `Pending`, and calls `target->emit_signal(name, payload)` directly. MUST be invoked from the test's main thread (which is the Godot "main" for this test binary).
- AC-2 verification flow: test invokes `mock.drive_signal_from_thread(...)`, joins the worker, calls `dispatcher_.drain_for_testing()`, asserts signal observed within 16 ms wall-clock across the full `drive_signal_from_thread + join + drain` sequence (`std::chrono::steady_clock`).
- In production, `post()` still uses `call_deferred` — the Godot engine drains on the next `_process` tick. No behavioral change for runtime code paths.

### D-4 [LOCKED] — `request_id` format & source
- **Format:** exactly 8 lowercase hex characters. Matches Story 1-6's Kotlin `corrId` (`UUID.randomUUID().toString().take(8).lowercase()`), so correlation across C++↔Kotlin logs lines up.
- **Source preference:** (1) `godot::Crypto` → `crypto->generate_random_bytes(4)` → convert to lowercase hex via `String::num_int64(..., 16)` or `PackedByteArray::hex_encode()`. (2) Fallback: `std::random_device` → 4 `uint8_t` bytes → hex string. T2 picks whichever is reachable in the gdext host + extension context and documents the pick inline.
- **Collision budget acceptance:** 8 hex chars = 2^32 space. Adequate for per-session log correlation (the usage). Future upgrade to UUIDv4 / 16-hex is a CROSS-BOUNDARY change (must land on both Kotlin and C++ simultaneously) — out of scope for 1-5.

### Test-only API gating [LOCKED]
All test-only API surface in 1-5 is gated by `#ifdef MWA_TESTING`. The only symbols that exist in production builds are the public 13 methods + 11 signals on `MobileWalletAdapter` and the production subset of `GodotMainDispatcher`. `set_bridge_for_testing` (D-3), `drain_for_testing` (D-2), and the mutex+queue fields exist ONLY when `MWA_TESTING` is defined. ### Signal payload semantic — `developer_details` vs `message`
AC-3 prose reads `message="Kotlin plugin not loaded"`. The canonical `mwa_error` payload (arch §3.2) has `user_message` + `developer_details`, not `message`. The implementer MUST place `"Kotlin plugin not loaded"` in `developer_details` (T5 asserts on this) and populate `user_message` with a human-facing equivalent. Full payload: `{request_id: "<8-hex>" (D-4), code="UNSUPPORTED_PLATFORM", user_message, developer_details="Kotlin plugin not loaded", recoverable=false, retry_hint="none" (A-1), layer="cpp", cause="", source_method=<op>}`.
