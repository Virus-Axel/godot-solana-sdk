# Story 4-2: forget_all + MasterKey Rotation + Mutex + Cancel In-Flight

## User Story
*(verbatim from `docs/plan.md:313` — do not rephrase)*

As a Godot game developer, I want `MWA.forget_all()` to wipe every cached wallet token, rotate the encryption key, and cancel any in-flight ops, so that I can implement a compliant "Sign out everywhere" flow for privacy regulation.

## Context

- **First consumer of `mwa_cancelled_lifecycle`.** Prior stories (Story 5-3 precursor scaffolding rolled into 2-1 + 3-3) landed the full lifecycle-cancellation seam: `NativeBridge.postMwaCancelledLifecycle(cancelledDictJson)` (1-param per A-12) at `NativeBridge.kt:185-194`; `JNIEXPORT postMwaCancelledLifecycleNative` at `src/mwa/mwa_android_bridge_jni.cpp:919` (delegates to `post_arity1("mwa_cancelled_lifecycle", ...)`); `ADD_SIGNAL("mwa_cancelled_lifecycle", DICTIONARY)` at `mobile_wallet_adapter.cpp:106`; `signal mwa_cancelled_lifecycle` at `MWA.gd:36` + relay at `:93`. **No new signal surface to land.** Story 4-2 is the FIRST emitter of this signal, and CR-3-3-B (Story 5-3 lifecycle observer) will be the SECOND emitter (lifecycle teardown path).
- **Inherits Story 4-1's multi-key wipe pattern** (DD-4-1-6 LOCKED). `mwaDeauthorize` already loops `store.listAllKeys().filter { it.identityUri == identityUriSnapshot }.forEach { store.deleteToken(it) }`. Story 4-2 generalizes the filter — `forget_all` deletes EVERY key (no identity_uri filter), then ROTATES the MasterKey on top.
- **`MasterKey` alias is a hard-coded literal** at `SecureTokenStore.kt:194`: `MASTER_KEY_ALIAS = "godot-sdk-mwa-master-key-v1"`. AC-1 references this exact alias. Rotation = delete from `KeyStore.getInstance("AndroidKeyStore")` + let lazy init recreate on next `MasterKey.Builder(ctx, alias)` access.
- **`MwaSessionState.clear()` (NOT `clearOnLogout()`).** Per Story 4-1 DD-4-1-2 LOCKED: full `clear()` wipes auth_token + connectedKey + clusterName + walletLabel + walletIconUri + identity URIs (the 8-field surface set by handleSuccess/handleReauthSuccess). `clearOnLogout()` (line 270) is the disconnect-path partial wipe that PRESERVES identity fields for re-connect ergonomics. forget_all is the "wipe everything" semantics — clear() is the right primitive.
- **`InflightMap` does NOT track `source_method` today.** `InflightMap.register(requestId, startedAtMs): Boolean` is 2-arg per `InflightMap.kt`. AC-2 requires emitting `mwa_cancelled_lifecycle{request_id, source_method:"sign_messages", reason:"forget_all_invoked"}` per in-flight slot. **DD-4-2-1 LOCKS the extension strategy** — the canonical fix is to extend `InflightMap.register` to a 3-arg `(requestId, sourceMethod, startedAtMs)` signature with a default `sourceMethod: String = "unknown"` so the existing 6 callers (mwaAuthorize, mwaReauthorize, mwaDisconnect, mwaSignMessages, mwaSignTransactions, mwaSignAndSendTransactions, mwaDeauthorize) continue to compile while T2 back-fills each call site explicitly with the correct value. Rejected alternative: parallel `ConcurrentHashMap<String, String>` registry duplicates state and creates a TOCTOU window between the two maps.
- **`forget_all` does NOT fire `reauth_required`** on `StorageCorruptException` during the wipe. Story 4-3's `withStorageOrReauthRequired` wrapper exists for INVOLUNTARY corruption (Tink AEAD fail at runtime). `forget_all` is the user's INTENTIONAL wipe — if the prefs file is corrupt, that's literally the goal of the next steps. **DD-4-2-6 LOCKS** the wrapper-bypass — the wipe path catches `StorageCorruptException` quietly and proceeds with the keystore-side rotation (which IS what corruption recovery would do anyway).
- **AC-1 ≤10s budget for 3 wallets.** Best-effort per-wallet `mwaClient.deauthorize(...)` is up to 3 attempts. Total ≤10s implies ≤3.3s/wallet on the worst path; **DD-4-2-7 LOCKS per-attempt timeout to 2s** so a single offline wallet doesn't push the budget over.

## Architecture Guardrails

9 LOCKED design decisions covering: InflightMap source_method extension, mutex primitive choice, cancel-in-flight ordering, MasterKey rotation strategy, forget_all-vs-corruption-wrapper interaction, per-deauth timeout budget, no-completion-signal contract, T7 close-out + R8 smoke parity, and post-rotation SecureTokenStore instance teardown.

- **DD-4-2-1 (LOCKED) — InflightMap.register signature extension to track source_method.** Add a 3rd arg: `register(requestId: String, sourceMethod: String, startedAtMs: Long): Boolean`. The 6 existing call sites (mwaAuthorize at GDExtensionAndroidPlugin.kt:~530; mwaReauthorize at ~615; mwaDisconnect at ~1100; mwaSignMessages at ~960 via runSigningOp; mwaSignTransactions at ~997 via runSigningOp; mwaSignAndSendTransactions at ~1058 via runSigningOp + the new sync preflight register at ~1085 from D-3-3-T7-1; mwaDeauthorize at ~1160) are back-filled with explicit `sourceMethod` values: `"authorize"`, `"reauthorize"`, `"disconnect"`, `op.sourceMethod` (sign_messages / sign_transactions / sign_and_send), `"deauthorize"`. The signing-path register in `runSigningOp` already has `op.sourceMethod` in scope. **Rejected alternative:** parallel `ConcurrentHashMap<String, String>` registry — duplicates state and creates a TOCTOU window between register/unregister of the two maps; lookup also requires reading both maps, an extra read per terminal signal.
  - InflightMap also gains a `snapshot(): Map<String, String>` method returning `(requestId → sourceMethod)` for the cancel-loop. Read-only iteration; the wipe-loop calls tryTerminate per slot to atomically claim the terminal-signal CAS.
  - This is a **Rule 2 deviation** that propagates to 6 prior stories' call sites. Logged as D-4-2-1 in T7 close-out. No amendment required (the change is purely additive — the signature extension is backward-incompatible only at the InflightMap layer, and the call sites all live in 1 file).

- **DD-4-2-2 (LOCKED) — Mutex via `kotlinx.coroutines.sync.Mutex`** (NOT `synchronized` / `ReentrantLock`). The Mutex wraps the entire `mwaForgetAll` body. Concurrent `mwaConnect` / `mwaSignMessages` / etc. block on the Mutex from their `scope.launch { withLock { ... } }` site (or fail with `UNSUPPORTED_PLATFORM` on the disconnect-platform path before they reach the Mutex). The Mutex is plugin-instance-scoped — cross-plugin-instance concurrency is out of scope per the Story 2-1 plugin-instance design.
  - Mutex is held on `Dispatchers.Default` (where scope.launch runs); the wipe blocks the JNI thread only briefly during the synchronous register-the-result + signal-emit. Most of the work is suspending I/O (clientlib.deauthorize per wallet) which yields the dispatcher.
  - Concurrent `forget_all + forget_all` is serialized by the Mutex. The second caller observes empty state (already-wiped) and is a no-op (vacuous success path per the empty `listAllKeys().isEmpty()` check).
  - **Rejected alternative:** `Mutex` placed AROUND each individual op-method's body — too coarse + breaks the terminal-signal CAS contract for ops that are designed to be concurrent (e.g., concurrent sign_messages on different requestIds). The forget_all-only Mutex is the minimum sufficient surface.

- **DD-4-2-3 (LOCKED) — Cancel-in-flight ordering.** Inside the Mutex's `withLock`:
  1. `inflightMap.snapshot()` — copy `(requestId → sourceMethod)` (read-only iteration; original map can mutate but the snapshot is stable for the cancel-loop).
  2. For each `(requestId, sourceMethod)` in snapshot:
     - `if (inflightMap.tryTerminate(requestId)) { nativeBridge.postMwaCancelledLifecycle(buildCancelledLifecycleJson(requestId, sourceMethod, "forget_all_invoked")) }`
     - On CAS loss (natural completion already terminal), increment `diagnostics.lateResult` + skip emit (terminal-signal invariant per arch §7.3 inheritance).
  3. THEN proceed to the wipe (per AC-2 ordering: "BEFORE the wipe starts").
  - Each emit is a single fire-and-forget seam call; the loop is fully synchronous in the snapshot's iteration order.

- **DD-4-2-4 (LOCKED) — MasterKey rotation strategy: delete-then-let-lazy-init-recreate.** Inside the wipe:
  1. `store.deleteAll()` — wipes ciphertext from EncryptedSharedPreferences (the file remains, just empty).
  2. `context.deleteSharedPreferences(SecureTokenStore.PREFS_FILE_NAME)` — kills the on-disk file (defensive; if step 1's `clear().apply()` left any residual entries, this guarantees a clean slate).
  3. `KeyStore.getInstance("AndroidKeyStore").deleteEntry(SecureTokenStore.MASTER_KEY_ALIAS)` — kills the MasterKey alias. The next time any code accesses `SecureTokenStore`'s lazy `masterKey`, `MasterKey.Builder(ctx, alias).setKeyScheme(AES256_GCM).build()` regenerates a fresh key.
  4. **Rebuild the `SecureTokenStore` instance.** The existing instance's `masterKey` and `prefs` are `by lazy` — they cached the OLD key reference. The next `storeProvider(requireContext())` call returns a fresh instance with fresh lazy fields. **DD-4-2-9 (LOCKED)** mandates that `forget_all` does NOT cache a `SecureTokenStore` instance across the rotation — every access goes through the provider, which produces a fresh instance.
  5. Sanity check: `KeyStore.getInstance("AndroidKeyStore").containsAlias(MASTER_KEY_ALIAS) == false` after step 3 (T1 unit test asserts this with a real Android Keystore mock or reflection-based fake).

- **DD-4-2-5 (LOCKED) — Best-effort per-wallet deauthorize loop.** Inside the wipe (BEFORE step 1 of DD-4-2-4):
  ```kotlin
  val store = storeProvider(requireContext())
  val keys = try { store.listAllKeys() } catch (ex: StorageCorruptException) {
    SdkLog.w(TAG, requestId) { "forget_all_listAllKeys_failed cause=${ex.cause?.javaClass?.simpleName}; proceeding with keystore-side wipe only" }
    emptyList<CacheKey>()  // fall through to MasterKey rotation; the corrupt prefs file will be deleted in DD-4-2-4 step 2
  }
  for (key in keys) {
    val cached = try { store.getToken(key) } catch (_: StorageCorruptException) { null }
    if (cached == null) continue
    val authToken = cached.authToken
    val identity = ConnectionIdentity(android.net.Uri.parse(cached.identityUri), android.net.Uri.parse(cached.walletIconUri), cached.walletLabel)
    try {
      withTimeoutOrNull(2_000L) { client.deauthorize(senderProvider(), identity, authToken) }
    } catch (ex: Throwable) {
      SdkLog.w(TAG, requestId) { "forget_all_deauthorize_failed walletPackage=${key.walletPackage} cause=${ex.javaClass.simpleName}" }
    }
  }
  ```
  - Per-attempt timeout: 2s per DD-4-2-7. AC-1 budget: 3 wallets × 2s = 6s for the deauth phase + ~2s for keystore + sessionState clear = 8s, comfortably under the 10s AC-1 ceiling.
  - **Continue-on-error policy** (AC-4): every failure is logged + skipped; the loop never aborts. The local wipe (steps 1-5 of DD-4-2-4) runs unconditionally after the deauth loop.
  - **Rejected alternative:** `Promise.all`-style parallel deauth with `kotlinx.coroutines.async` per wallet + `awaitAll()` — speeds up the per-wallet deauth phase but introduces a watchdog-vs-tail-of-slowest-wallet race that's hard to bound deterministically; the sequential loop's worst-case 2s × N is predictable and fits the 10s budget for N≤3.

- **DD-4-2-6 (LOCKED) — `forget_all` bypasses `withStorageOrReauthRequired`** (Story 4-3's fail-closed wrapper). Rationale: `forget_all` is the user's INTENTIONAL wipe; corruption during the wipe is irrelevant because the corrupt state is about to be removed anyway. The deauth loop's `try { listAllKeys() } catch (StorageCorruptException) { emptyList() }` (DD-4-2-5) handles the "can't enumerate cached wallets" case gracefully — we simply skip the deauth phase and proceed to MasterKey rotation, which also wipes the corrupt prefs file as a side-effect (DD-4-2-4 step 2).
  - **Rejected alternative:** wrap the `listAllKeys()` call with `withStorageOrReauthRequired` — would emit `reauth_required{reason:"keystore_corrupt"}` during forget_all, which is exactly the OPPOSITE of what the user requested ("wipe everything", not "tell me to reauth").

- **DD-4-2-7 (LOCKED) — Per-deauth timeout = 2s.** Bounded by AC-1's 10s ceiling for 3 wallets. The internal watchdog (`effectiveWatchdog` from Story 3-1 DD-23) is NOT used here — `forget_all` has its own per-attempt budget that's tighter than the default `INTERNAL_WATCHDOG_DEFAULT_MS` (60s). Caller-provided `timeoutMs` is NOT applicable to `forget_all` (the facade method takes no `opts: Dictionary` arg per the no-completion-signal contract); the 2s constant is the only timeout knob.

- **DD-4-2-8 (LOCKED) — No completion signal.** AC-1's evidence path is post-condition state assertions (`listAllKeys().isEmpty()`, `KeyStore.containsAlias(...) == false`, `sessionState` empty, `is_connected() == false`). The `MWA.gd forget_all` facade returns `void` per D-3-1-12 LOCKED inheritance. **Rejected alternative:** add a `forget_all_completed{request_id}` signal (1-param) — the AC explicitly does NOT require this; adding a signal would expand the public surface unnecessarily and create a new test vector (signal arity, A-12 row update, JNIEXPORT mirror, MWA.gd relay, R8 keep). The state-only AC is sufficient.
  - The user observes completion via `is_connected() == false` after a short wait OR by checking `MwaSessionState.getAuthToken() == null`. This is the same UX as `disconnect()` (Story 2-3) where no `disconnect_complete` is observed by the user (a 2-param `disconnect_completed` signal exists but the user typically doesn't await it).

- **DD-4-2-9 (LOCKED) — Post-rotation SecureTokenStore instance teardown.** The plugin holds `storeProvider: (Context) -> SecureTokenStore` — a function, not a cached instance. Every storage access calls `storeProvider(requireContext())` which (in production) returns a fresh `SecureTokenStore(ctx)` per call. **In tests**, `PluginTestHarness.freshPlugin(storeOverride = ...)` injects a fixed instance; tests that exercise rotation MUST inject a `MutableStoreProvider` that lets the rotation step swap the inner instance.
  - For the integration tier (T5 androidTest), the harness uses a real `SecureTokenStore(ctx)` per call — the rotation step's `KeyStore.deleteEntry` invalidates the previous instance's lazy MasterKey reference, but the NEXT `storeProvider(ctx)` call gets a fresh instance whose lazy init regenerates the key. This is the production behavior.
  - **Rejected alternative:** plugin holds a `var secureTokenStore: SecureTokenStore` field that gets reassigned post-rotation — adds a mutable field to a class that's mostly immutable, and creates a TOCTOU window where a concurrent op reads the old reference. The provider-lambda is cleaner.

## Concerns

- **CR-4-2-A (LOW, candidate for Story 5-2 metrics):** `forget_all` invocation count + per-call wallet count + total wallclock are NOT logged via `diagnostics`. Useful operational signal for "how often do users invoke the GDPR/CCPA wipe" but not a 4-2 deliverable. Story 5-2 should add `diagnostics.forgetAllInvocations` + `diagnostics.forgetAllAvgMs` if a histogram surface lands.
- **CR-4-2-B (LOW, MasterKey rotation atomicity):** Steps 1-3 of DD-4-2-4 (deleteAll → deleteSharedPreferences → KeyStore.deleteEntry) are NOT atomic. A process kill between steps 1 and 3 leaves the prefs file deleted but the MasterKey alias intact (or vice-versa). Recovery on next launch: the lazy `prefs` init catches the AEAD failure (Tink) and emits `reauth_required` per Story 4-3's wrapper — which is the right behavior (the user re-authenticates, the new key generates fresh, the orphaned prefs file is recreated). Acceptable design tradeoff; the partial-state recovery path IS exercised by Story 4-3's existing tests (transitively).
- **CR-4-2-C (LOW, plugin-instance-scoped Mutex):** DD-4-2-2's `forgetAllMutex` is plugin-instance-scoped. Cross-plugin-instance concurrency (e.g., user spawns two plugin instances in parallel via reflection) is NOT serialized. Out of scope per Story 2-1's plugin-instance design. Production callers always use the single Godot-autoload-instance pattern; the cross-instance concern is theoretical.
