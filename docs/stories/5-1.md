# Story 5-1: Non-Android Platform Behavior + 4-Platform GDExtension Load CI

## User Story
*(verbatim from `docs/plan.md:356` — do not rephrase)*

As a Godot game developer testing on desktop before deploying to Android, I want MWA methods to return a typed `UNSUPPORTED_PLATFORM` error (not crash the editor) when I'm on non-Android, so that my cross-platform code compiles and runs with a clear error signal.

## Context

- **All 7 op-methods already wire the null-bridge fallback (Story 1-5 + 1-4 inheritance).** Each op-method on `MobileWalletAdapter` (mwa_connect / reauthorize / mwa_disconnect / deauthorize / sign_messages / sign_transactions / sign_and_send / forget_all) executes `if (bridge_ == nullptr) { dispatcher_.post("mwa_error", build_unsupported_platform_payload(request_id, "<source>")); return; }` BEFORE bridge delegation. Verified at `src/mwa/mobile_wallet_adapter.cpp:150,162,172,182,192,202,212,272`. **Story 5-1 is verification-first** — there is no new production code in the C++ layer.
- **Two paths produce identical-shape `mwa_error{UNSUPPORTED_PLATFORM}`** — the platform-selector returns `NoOpMwaAndroidBridge` on non-Android (per `src/mwa/platform_selector.cpp:17-22` AC-A-4 LOCKED single TU), so on a real Linux/Windows/macOS/Web build the bridge_ pointer is **non-null** but each op delegates to `NoOpMwaAndroidBridge::emit_unsupported(...)` (`no_op_mwa_android_bridge.cpp:21-43`). The `if (bridge_ == nullptr)` short-circuit branch is reachable ONLY in MWA_TESTING builds (DD-D-3 / 1-5 D-3). On real non-Android builds the NoOp bridge path fires; the payload shape is the SAME 10-key A-14 envelope either way (verified by inspection — both `build_unsupported_platform_payload` and `NoOpMwaAndroidBridge::emit_unsupported` produce all 10 keys: request_id / code / message / user_message / developer_details / recoverable / retry_hint / layer / cause / source_method).
- **`MWA.is_supported()` does NOT exist on the GDScript facade today** (verified by grep — 0 hits in `addons/SolanaSDK/mwa/MWA.gd`). AC-1 mandates it. Story 5-1 T2 lands a 1-LOC GDScript getter (no C++ binding required — `OS.get_name() == "Android"` is a pure GDScript runtime check).
- **CR-32 backstop already exists** at `addons/SolanaSDK/mwa/tests/test_unsupported_platform.gd` (Story 2-1 T10 + Story 2-3 T6 expansion). Today it covers 2 scenarios (connect + disconnect) on linux-release ONLY via `.github/workflows/gdscript_tests.yml` (the `cr32-desktop` job). Story 5-1 EXTENDS this — adds the remaining 5 user-facing ops (reauthorize / deauthorize / sign_messages / sign_transactions / sign_and_send / forget_all) AND extends the workflow matrix to all 4 non-Android platforms (linux/windows/macos/web).
- **AC-A-4 LOCKED — single platform-abstraction TU.** Story 1-4's `platform_selector.cpp` is the ONE-AND-ONLY `#ifdef __ANDROID__` translation unit; CI grep ban (`ci/grep_bans.sh` pattern 5) enforces this. Story 5-1 does NOT introduce any new `#ifdef __ANDROID__` — the existing NoOp-bridge path produces `UNSUPPORTED_PLATFORM` on every non-Android platform automatically.
- **A-13 retired host-mode C++ tests; CR-35 tracks future headless-Godot tier.** Story 5-1 is the FIRST story to systematically validate non-Android behavior across all 4 target platforms via headless Godot — closes the AC-1..5 coverage gap that A-13 left as CR-31 / CR-32. After 5-1, the CR-32 backstop is no longer "MEDIUM partial" — it's a 7-op × 4-platform full grid. CR-31 (no automated host-mode binding-smoke) remains MEDIUM (Kotlin-side parity tests carry partial coverage); CR-35 stays open for future C++ assertion tier.

## Architecture Guardrails

5 LOCKED design decisions covering: workflow choice (`gdscript_tests.yml` extension vs `build.yml` integration), Web platform handling, `MWA.is_supported()` impl strategy, headless harness extension scope, signal-capture timeout budget.

- **DD-5-1-1 (LOCKED) — Extend `gdscript_tests.yml`, NOT `build.yml`.** AC-3's prose says "Existing `.github/workflows/build.yml` matrix is extended", but the **logical home for runtime-load assertions is the existing `gdscript_tests.yml`** (Story 2-1 T10 / CR-32 backstop). `build.yml` is a compile-only matrix (16 entries spanning android-arm64 / android-x86_64 / windows-debug/release / linux-debug/release / macos / ios / web — 4-mode); adding a runtime-load step there would couple compilation jobs to Godot installation + addon staging. `gdscript_tests.yml` already does the right thing for linux-release (`cr32-desktop` job: scons build → stage addon → headless Godot → grep stdout); 5-1 extends THIS workflow's matrix to 4 platforms. **AC-3 prose intent satisfied** by the cross-reference: `gdscript_tests.yml` is the runtime-tier sibling of `build.yml`, both run on every PR.
  - **Rejected alternative:** add a new `gdextension_load.yml` workflow — duplicates the build-and-stage prologue; CR-32's harness already proves the pattern. Reuse > duplicate.
  - **Rejected alternative:** add the runtime step to `build.yml`'s `package` job — that job runs ONCE on ubuntu-latest after the matrix merges artifacts; can't validate per-platform load.

- **DD-5-1-2 (LOCKED) — Web platform: skip runtime headless, assert symbol-resolve only.** Web export does not run under `godot --headless` the way native platforms do (the engine binary IS the wasm32 module; there is no host runtime). For the Web matrix entry, the assertion is **build-time and symbol-table only**: (a) `scons platform=web target=template_release arch=wasm32 threads=False -j2` builds clean (already covered by `build.yml`'s 4 web entries); (b) the resulting `bin/libgodot-solana-sdk.web.template_release.wasm32.wasm` (or equivalent) loads in `wasm-objdump`-style inspection without unresolved imports for any MWA symbol. Practically: T3 adds a `web-symbol-resolve` matrix entry that runs `scons platform=web ...` + verifies the `.wasm` artifact exists and is non-empty; the runtime UNSUPPORTED_PLATFORM emission for Web is implicit via the same NoOp bridge path (the C++ TU compiles for wasm32 and the NoOp branch is selected via `#ifdef !__ANDROID__` in `platform_selector.cpp:11-13`).
  - **Rejected alternative:** spin up Node.js + jsdom + a headless wasm runner — high CI cost, fragile across emsdk versions, blocks for hours of debugging if the addon's `.gdextension` manifest doesn't match the wasm export shape. Symbol-resolve is the minimum sufficient evidence for AC-2 (no crash) on Web.
  - **AC-1 partial coverage on Web:** AC-1 says "in headless Godot on Linux/Windows/macOS/Web" — for Web we satisfy AC-1 transitively via the linux-release run (the C++ NoOp-bridge code path is platform-conditional ONLY at `platform_selector.cpp:17-22`; the SAME branch fires on linux-release as on web-release). This is logged as a 5-1-specific coverage caveat in T6 close-out (D-5-1-1 Rule 1 — informational, no amendment).

- **DD-5-1-3 (LOCKED) — `MWA.is_supported()` is a 1-LOC GDScript getter, NOT a C++ binding.** Implementation: `func is_supported() -> bool: return OS.get_name() == "Android"`. Rationale: (a) the value is a pure runtime constant of the Godot platform — `OS.get_name()` is a stable godot-cpp API; (b) avoiding the C++ binding side-steps the `_bind_methods` + `ClassDB::bind_method` ceremony for a 1-line check that has no Android-side behavior; (c) consistent with `_SUGGESTED_WALLETS` (already a pure-GDScript constant on `MWA.gd:61`).
  - **Rejected alternative:** add `bool MobileWalletAdapter::is_supported() const` C++ method that returns `bridge_ != nullptr` (or branches on `#ifdef __ANDROID__`) — adds a binding row, requires a new T3-style C++ task, conflates "supported" with "bridge null state" (the bridge is NEVER null in production builds — the NoOp instance is non-null on non-Android per AC-A-4). The GDScript-side check is semantically tighter.
  - **AC-1 evidence assertion:** T2's harness adds `assert(not MWA.is_supported())` AFTER the `OS.get_name()` check confirms a non-Android platform. On Android (production), `MWA.is_supported() == true`; this is asserted in androidTest tier (FUTURE, optional — not in 5-1 scope per the size budget).

- **DD-5-1-4 (LOCKED) — Harness extension scope: 7 ops, sequential, single test scene.** T2 extends `test_unsupported_platform.gd` from 2 scenarios (connect + disconnect) to **8 scenarios** covering all 7 user-facing ops + 1 `is_supported()` getter check:
  1. `is_supported() == false` (BEFORE any op invocation — pure-GDScript path)
  2. `mwa.connect(identity, cluster)` → mwa_error{UNSUPPORTED_PLATFORM, source_method:"connect"} (existing — preserved)
  3. `mwa.disconnect()` → mwa_error{UNSUPPORTED_PLATFORM, source_method:"disconnect"} (existing — preserved)
  4. `mwa.reauthorize({})` → mwa_error{source_method:"reauthorize"} (NEW)
  5. `mwa.deauthorize()` → mwa_error{source_method:"deauthorize"} (NEW)
  6. `mwa.sign_messages([PackedByteArray([1,2,3])])` → mwa_error{source_method:"sign_messages"} (NEW)
  7. `mwa.sign_transactions([PackedByteArray([1,2,3])])` → mwa_error{source_method:"sign_transactions"} (NEW)
  8. `mwa.sign_and_send([PackedByteArray([1,2,3])])` → mwa_error{source_method:"sign_and_send"} (NEW)
  9. `mwa.forget_all()` → mwa_error{source_method:"forget_all"} (NEW)
  - **Sequential** (NOT parallel) — each scenario calls `_reset_capture()` between phases; the existing `_run_connect_scenario` + `_run_disconnect_scenario` pattern is replicated for the 6 new scenarios. Each scenario waits up to `TIMEOUT_FRAMES = 10` for the deferred mwa_error (1 frame per AC-1 spec; 10 for CI slack — preserved from CR-32).
  - **No `pending_submission_found` / `mwa_cancelled_lifecycle` checks** — those are post-success lifecycle signals that don't fire on non-Android (the NoOp bridge path produces `mwa_error` only). Out of scope for AC-1.
  - **Each new scenario asserts the 10-key A-14 envelope** via the existing `_assert_unsupported_platform_payload(source_method)` helper at line 84 — the helper is already correct and reusable; new scenarios just feed their `source_method` literal.
  - **Rejected alternative:** spin up a separate test scene per op (8 .tscn files) — bloats the addon directory and the workflow has to grep multiple log files. Single-scene single-grep is simpler and matches the existing CR-32 pattern.

- **DD-5-1-5 (LOCKED) — CI matrix: 4 platforms, fail-fast=false, identical recipe modulo platform/runner/threads.** T3's matrix in `gdscript_tests.yml`:
  | id | runner | platform | arch | threads | runtime check |
  |----|--------|----------|------|---------|---------------|
  | linux-desktop | ubuntu-latest | linux | x86_64 | True | `godot --headless --path .` runs harness |
  | windows-desktop | windows-latest | windows | x86_64 | True | `godot.exe --headless --path .` runs harness |
  | macos-desktop | macos-latest | macos | universal | True | `godot --headless --path .` runs harness |
  | web-symbol-resolve | ubuntu-latest | web | wasm32 | True | scons builds; `.wasm` artifact exists + non-empty (per DD-5-1-2) |
  - **fail-fast: false** — one platform failing should NOT mask the others; T4 needs evidence from all 4.
  - **godot binary install per OS:** Linux + macOS use `chickensoft-games/setup-godot@v1` (already used by CR-32). Windows is the new platform — install via the same action (action supports windows-latest per its docs). Web uses `mymindstorm/setup-emsdk@v14` (matches `build.yml`).
  - **Each runtime job** stages the addon (cp gdextension; symlink addons + bin into `example/WalletAdapterAndroid/`), then runs `godot --headless --path . res://addons/SolanaSDK/mwa/tests/test_unsupported_platform.tscn --quit-after 10000 --log-file <id>-log.txt`. Success gate: `grep -q 'ALL TESTS PASSED' <id>-log.txt`. Same recipe as the existing `cr32-desktop` job; just parameterized over the matrix.
  - **The existing `cr32-desktop` job is RENAMED to `linux-desktop`** in T3 (Rule 1 deviation D-5-1-2 — purely cosmetic; the GitHub Actions cache key changes but rebuild cost is bounded by the godot-cpp commit SHA pin which doesn't change).

Future optimization: gate windows/macos to `pull_request` only (skip on `push to feature/**` per `gdscript_tests.yml:13`'s current trigger config) — not applied in 5-1 to avoid `on:` trigger churn mid-implementation.
- **CR-5-1-C (LOW, `MWA.is_supported()` lacks an Android-positive unit test):** DD-5-1-3 LOCKS the 1-LOC GDScript getter as `OS.get_name() == "Android"`. On non-Android the harness asserts `not MWA.is_supported()` (T2).
