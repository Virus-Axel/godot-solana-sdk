# Story 2-3: disconnect (client-side session wipe with token retention)

## User Story
As a Godot game developer, I want to call `MWA.disconnect()` to close the session while keeping the auth token cached, so that a later `reauthorize()` still works silently.

## Context
- **Small-S slice dependent only on Story 2-1.** All scaffolding is already in place from the 2-1 vertical slice: `MwaSessionState.clearOnLogout()` (established at Story 2-1 T4, `MwaSessionState.kt:261-280` — wipes auth-only state, preserves identity), `InflightMap` (2-1 T4), `SecureTokenStore` (2-1 T1-T3), dispatcher `Callable+Array` arity-2 plumbing (2-1 T6, CR-18 closure), `MWA.gd` facade `disconnect()` delegation (2-1 T7, `MWA.gd:116-118`), JNI `mwaDisconnectFromJni` companion wrapper (2-1 T5, `GDExtensionAndroidPlugin.kt:166-169`), and C++ `mwa_disconnect()` bridge-delegation body (2-1 T6, `mobile_wallet_adapter.cpp:164-172`). Story 2-3 replaces **only** the `emitNotImplemented("disconnect")` stub on the Kotlin side with real wipe+emit logic.
- **The key invariant differentiating disconnect from deauthorize (Story 4-1)**: after `disconnect`, the `SecureTokenStore` CacheRecord is RETAINED. A subsequent `reauthorize` (Story 2-2) must succeed silently. `deauthorize` (Story 4-1) is the path that revokes + removes the token. AC-1 tests the retention mechanically. This is the single design choice most likely to be miscoded — if an implementer "cleans everything up" on disconnect, AC-1 fails and Story 2-2's reauth-after-disconnect contract breaks in the field.
- **AC-2 is a partial contract with not-yet-implemented Story 2-2.** 2-2 (reauthorize) lands in Wave 5 alongside 2-3 but depends on 2-1 only (not on 2-3). To keep AC-2 testable inside this story without creating a 2-3↔2-2 cycle, we assert the **preconditions for silent reauth** (cache record present + fingerprint retained on disk + `MwaSessionState.authToken == null` + `getPublicKeyBase58 == ""`) rather than the full reauth round-trip. The full E2E lands in 2-2's AC-1 coverage.
- **Pure client-side wipe — DD-2-3-1 [LOCKED]**: disconnect is a CLIENT-SIDE operation. The MWA protocol (`mobile-wallet-adapter-protocol/specification.md` and clientlib-ktx 2.0.3's `MwaClient.disconnect()` contract at `MwaClient.kt:70-88`) states explicitly: "Client-layer no-op. At this layer there is no persistent session to tear down — clientlib-ktx binds each wallet interaction to an ActivityResultSender for the duration of the transact/connect call, so there is nothing to close when a caller is done using the client." There is no remote "disconnect" RPC to invoke. Arch §4 UC-5 alt-flow confirms: the Android Intent association simply ends when the wallet app stops expecting more calls.
- **All 7 Wave 1-4 stories are CLOSED.** Story 2-1 Gate 5 passed 2026-04-24 (`post-mwa-2-1` tag applied; CI fixups rounds 1-2 + round-3 all green on emulator + R8 smoke). Kotlin unit baseline: 109/109 green. Pytest: 83/83 green. Total: 192 unit tests passing.

## Architecture Guardrails

### §4 UC-5 Alt-Flow (disconnect) [LOCKED — inlined from arch.md]
> Ref: `docs/architecture.md#4 Workflows` — UC-5 alt-flow.

Disconnect is a client-side wipe of in-memory session state (`MwaSessionState.authToken` → `null`, `publicKeyBase58` / `clusterName` / `walletLabel` / `authTokenFingerprint` → `""`, `walletIconUri` → `""`, `connectedKey` → `null`). The on-disk `SecureTokenStore` CacheRecord is PRESERVED. The per-install fingerprint salt is PRESERVED. No remote RPC; no wallet-app Activity transition; no network IO.

### DD-2-3-1 [LOCKED] — Pure client-side wipe; no JNI→clientlib-ktx hop
Disconnect does NOT invoke `MwaClientImpl.disconnect()`. Rationale: `MwaClientImpl.disconnect()` is a documented client-layer no-op (`MwaClient.kt:70-88`) — calling it would add a coroutine-launch + clientlib-close overhead for zero semantic work and make AC-1's ≤200ms bound less predictable. The `MwaClient` interface method stays available for future use (e.g. if clientlib-ktx adds a remote session teardown in a future version), but 2-3's code path does not route through it. If a later story needs remote disconnect, file an amendment and change this DD.

**Implementation:** `GDExtensionAndroidPlugin.mwaDisconnect(requestId)` runs synchronously on the JNI worker thread, calls `sessionState.clearOnLogout()`, and posts `disconnect_completed` via `nativeBridge.postDisconnectCompleted(requestId, resultDictJson)`. No `scope.launch {}`; no `withTimeoutOrNull`; no watchdog.

### DD-2-3-2 [LOCKED] — Idempotent emission via CAS-first-then-wipe-then-emit (AC-3)
AC-3 says "disconnect_completed still fires" when already disconnected. Two options were considered:
- **(a) Hard no-op** — second call returns immediately without emitting. Violates AC-3 verbatim wording.
- **(b) Always emit** — second call runs `clearOnLogout()` (which is itself idempotent on already-cleared state: `authToken?.clear()` is null-safe, and re-assigning empty strings is a no-op) and always posts `disconnect_completed`. Satisfies AC-3; cheap.

**Choose (b).** Every `mwaDisconnect(requestId)` call produces exactly one `disconnect_completed` signal.

**Ordering discipline (concern C3 resolution):** Inside the `scope.launch` body (see DD-2-3-3), the op sequence is **register → CAS (tryTerminate) → wipe → emit** — NOT register → wipe → CAS → emit. The CAS gates both the state mutation and the emission atomically, keeping the state-machine discipline uniform with Story 2-1's authorize path even though disconnect has no competing failure/timeout path. If `tryTerminate(requestId)` loses the CAS (can't happen in this sync-ish flow, but defense in depth), no wipe and no emit occur — caller sees nothing, which is acceptable because (i) the second concurrent disconnect with the SAME request_id is a caller contract violation caught at `register` via the duplicate_request_id branch, and (ii) two DIFFERENT request_ids each get their own register slot + their own CAS and proceed independently. The `requestId` changes per call (see DD-2-3-4), so InflightMap tracking works naturally.

### DD-2-3-3 [LOCKED] — Dispatcher-marshalled (Godot main), no watchdog
Because disconnect is a pure in-memory wipe (DD-2-3-1), no timeout is needed; AC-1's ≤200ms bound is trivially satisfied (typical wipe: <1ms). **Concern C1 resolution — thread specification:** the `mwaDisconnect(requestId)` call enters on the JNI worker thread (the thread that invoked `mwaDisconnectFromJni` via JNI), then immediately hops to the Kotlin main dispatcher via `scope.launch { withContext(mainDispatcher) { ... } }` — exactly parallel to Story 2-1's `mwaAuthorize` / `handleSuccess` pattern (which established DD-22 dispatcher marshalling in B1's resolution). All `MwaSessionState` writes, `InflightMap.register` / `tryTerminate`, JSON build, and `nativeBridge.postDisconnectCompleted(...)` run on the main dispatcher. Although `MwaSessionState` is fully `@Synchronized` and would tolerate an off-main write, matching the B1 discipline keeps the convention uniform and avoids a "why is this one op different" reader tax.

**No watchdog:** there is NO `withTimeoutOrNull` wrapper because there is no await-point — the entire body is synchronous Kotlin (state assignments + JSON build + one `postDisconnectCompleted` call). Skipping `withTimeoutOrNull` also avoids the late-result-drop logic — a disconnect that's mid-flight during plugin teardown cannot be "late" because there's no suspension point. The CR-41 teardown-lease guard from Story 2-1 still covers the C++ `postDisconnectCompletedNative` JNIEXPORT callback (T3) in the dispatcher-teardown race.

InflightMap is still used (see DD-2-3-4) for terminal-signal-invariant consistency; the `tryTerminate` CAS can't realistically lose the race in this flow, but it's still called because it's the canonical gate (see DD-2-3-2).

### DD-2-3-4 [LOCKED] — Fresh UUID `request_id` per call + A-12 2-param signal shape
`request_id` generation is unchanged from the authorize path: C++ `MobileWalletAdapter::mwa_disconnect()` calls the existing `generate_request_id()` helper producing 8 lowercase hex chars (D-4, `mobile_wallet_adapter.cpp`). The JNI bridge's `MwaAndroidBridgeJni::disconnect(request_id, opts)` forwards that id to `mwaDisconnectFromJni(reqId: String)` in Kotlin.

**No request_id collapsing (concern C5 resolution):** Each `MWA.disconnect()` GDScript call MUST mint its own `request_id` via `generate_request_id()` and MUST NOT be coalesced with a concurrent or already-in-flight disconnect. The C++ facade does not dedupe or short-circuit back-to-back calls — two `MWA.disconnect()` calls → two `generate_request_id()` invocations → two distinct 8-hex `request_id`s → two `disconnect_completed` emissions, each carrying its originating `request_id` (AC-3 semantics). Collapsing would break caller-side correlation tracking (e.g., a caller storing `request_id → ui_state_handle` mapping). T3's AC explicitly asserts this: grep `mobile_wallet_adapter.cpp::mwa_disconnect` for any early-return or dedupe branch → 0 matches.

**Signal shape per A-12 (2-param `*_completed` family)**: `disconnect_completed(request_id: String, result: Dictionary)`. The `result` Dictionary carries a minimal two-key payload: `{source_method: "disconnect", request_id: <same>}` — `request_id` is duplicated inside the Dictionary for consumer ergonomics (some consumers key off the dict; some off the typed arg), matching the `connect_completed` shape from 2-1. No additional fields (no `success: true`, no `retained_token: true` — keep the payload minimal; AC-1 assertions are on STATE, not on payload content).

Even though no failure path exists in DD-2-3-1 + DD-2-3-3, the InflightMap pattern is preserved for consistency with the 2-1 authorize path: `register(requestId, nowMs)` → wipe → `tryTerminate(requestId)` → `postDisconnectCompleted(requestId, resultJson)`. If a duplicate `requestId` somehow arrives (caller contract violation), we log+drop per 2-1 T4's `duplicate requestId` branch pattern.

### §7.1 / DD-22 Thread-marshalling [LOCKED]
> Ref: `docs/architecture.md#7.1 Thread Marshalling Primitive`.

`disconnect_completed` emission crosses to Godot main through the C++ `GodotMainDispatcher`. The Kotlin JNI callback `postDisconnectCompletedNative(reqId, resultDictJson)` (new `external fun` in T2) reaches the C++ `Java_..._postDisconnectCompletedNative` JNIEXPORT (T3) which calls `dispatcher->post("disconnect_completed", godot::Array::make(reqId, result))` — exactly parallel to the 2-1 `postConnectCompletedNative` path. Grep-ban `emit-signal-outside-dispatcher` stays protective.

### DD-2-3-5 [LOCKED] — CacheKey is derived from caller-provided fields, NOT from MwaSessionState
**Concern C2 resolution.** The `CacheKey` 4-tuple `(cluster: String, chainId: String, walletPackage: String, identityUri: String)` is built at the call site from arguments passed by the GDScript caller through the facade — see `GDExtensionAndroidPlugin.kt:407` (authorize path) and `SecureTokenStore.kt:102` (key enumeration). `MwaSessionState` is NOT a CacheKey source. In particular:

| MwaSessionState field | Role | Wiped by `clearOnLogout()`? | Is a `CacheKey` input? |
|-----------------------|------|-----------------------------|-----------------------|
| `identityUri` | session snapshot | NO — preserved | NO (the CacheKey's `identityUri` comes from the caller's arg) |
| `identityName`, `iconUri` | session snapshot | NO — preserved | NO |
| `cluster: Int` | session snapshot | NO — preserved | NO (CacheKey uses `cluster: String`, a different field) |
| `clusterName: String` | display-only snapshot for `get_cluster()` GDScript getter | YES — wiped | NO (despite the name, this is NOT the CacheKey `cluster` — it's a display copy) |
| `walletLabel`, `walletIconUri`, `publicKeyBase58`, `authToken`, `authTokenFingerprint`, `connectedKey`, `lastResult`, `signingStatus` | auth-only state | YES — wiped | NO |

Silent reauth in Story 2-2 will rebuild the `CacheKey` from the caller's fresh `MWA.reauthorize(cluster, chain_id, ...)` arguments and look up the retained `CacheRecord` on disk. `MwaSessionState` serves the session-display + diagnostics surface; `SecureTokenStore` serves the auth-token cache; `CacheKey` is the bridge between them but it depends on the CALLER, not on session state. If a future story is tempted to read `CacheKey.cluster` from `MwaSessionState.clusterName` (or any sibling), that would be a Rule 3 amendment-requiring change — but Story 2-3 does not make that change and does not require it.

### §8.3 SecureTokenStore — retention invariant [LOCKED]
> Ref: `docs/architecture.md#8.3 Secure Storage Implementation`.

`mwaDisconnect` DOES NOT call `SecureTokenStore.deleteToken(key)` or `deleteAll()`. Those are reserved for Story 4-1 (`deauthorize`) and Story 4-2 (`forget_all`). Reaching for either from the disconnect path is a Rule 3 deviation (amendment required) because it breaks AC-1's retention invariant + Story 2-2's silent-reauth contract.

### A-14 10-key `mwa_error` — not used on happy path
Disconnect has no happy-path error. If `sessionState` itself throws during `clearOnLogout()` (it can't — the impl is pure assignment under `@Synchronized`), a defensive `try/catch` wraps the wipe and emits `mwa_error` with the 10-key A-14 shape (`code=PROTOCOL_ERROR`, `source_method="disconnect"`, `layer="kotlin"`). This defensive branch is not expected to fire in practice but keeps the terminal-signal invariant (AT MOST ONE terminal signal per `requestId`) watertight per Story 2-1's established convention.

### D-1 method rename — already wired
> Ref: `docs/stories/1-5.md#D-1` + `docs/stories/2-1.md#D-10`.

C++ node binding is `mwa_disconnect` (not `disconnect`) to avoid `godot::Object::disconnect(signal, callable, flags)` arg-count collision. `MWA.gd`'s facade method is `disconnect()` with `@warning_ignore("shadowed_global_identifier")` forwarding to `_node.mwa_disconnect()`. Both already land from 2-1 T6+T7 — Story 2-3 does NOT edit this wiring.

## Deviations Encountered

- **D-2-3-1 (Rule 1, implementation time):** T1 and T2 merged into a single atomic commit. Reason: Kotlin's type system requires `nativeBridge.postDisconnectCompleted(...)` to resolve at compile time, so a test-only T1 commit referencing the not-yet-added interface method would fail to compile and break the `:plugin:testDebugUnitTest` baseline. Merging preserves Gate 5 criterion 5 (one atomic per-task commit) AND TDD discipline (tests written first, verified red against empty impl stub locally, body filled in, verified green — all in one commit). Story's T1/T2 split was aspirational.
- **D-2-3-2 (Rule 1, implementation time):** Test method names use `AC-1 / AC-2 / AC-3` prefixes (e.g., `` `AC-1 disconnect wipes session state but retains SecureTokenStore entry` ``) rather than the bare names spec'd in T1. Reason: the AC cross-reference is useful when reading `./gradlew` output and JUnit reports. `must_haves.artifacts.contains` patterns are substring-matched so downstream automation still resolves. Fourth test name extended to `` `duplicate requestId emits PROTOCOL_ERROR and does not wipe state` `` to reflect the expanded assertion scope.
