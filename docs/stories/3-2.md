### DD-3-2-1 [LOCKED] — `mwaSignTransactions` is a near-mirror of `mwaSignMessages`; do NOT extract a shared parent
The two methods share ~80 % of their bodies (identity reconstruction, auth-token read, scope.launch + runSigningOp invocation). DO NOT introduce an abstract parent or higher-order helper to deduplicate them. Reasons:
1. **AC-2 ≤20 LOC budget** is enforced PER METHOD — extracting a parent would make `mwaSignTransactions`'s "body" ambiguous (does the LOC counter follow the inheritance chain?). Test-side complexity not worth the ~5-LOC saving.
2. **Stories 3-3 (`sign_and_send`) needs an additional `cluster` argument** — its lambda body shape differs (`signAndSendTransactions(... transactions, cluster)`) and its success-path emits a different signal (`sign_and_send_completed` with a different result shape). A parent would have to abstract over (a) signature shape, (b) lambda receiver-type, (c) success builder. The complexity outweighs the savings.
3. **Codebase precedent.** `mwaReauthorize` (`GDExtensionAndroidPlugin.kt:507-619`) and the (future) `mwaConnect` body both share the same auth-flow scaffolding without an extracted parent — the project's convention is "near-clone with deltas commented" over "abstract parent + concrete subclasses". Story 3-1's drift discipline (per its Dev Notes "runSigningOp is the consumed-by-3-2/3-3 contract") explicitly chose the lambda-closure pattern over a parent class.

### DD-3-2-2 [LOCKED] — Extract `countMethodLines(...)` to a shared top-level `internal fun` in `LocCountUtil.kt` (test source set)
The LOC-counter helper landed in Story 3-1 T1 as `private fun countMethodLines(...)` **inside** the `MwaAndroidPluginSignMessagesTest` class body (verified at `MwaAndroidPluginSignMessagesTest.kt:255` — `private fun` is class-scoped per Kotlin spec, NOT file-scoped). It is therefore NOT visible from another test class even in the same package + same test source-set. The earlier draft of DD-3-2-2 incorrectly claimed "file-level visibility allows direct cross-file reference" — that's Kotlin's behavior for `private fun` declared at file top-level, which the helper is NOT.

**Two options were considered (per C-3-2-A code-review fix on rev1):**
1. **Refactor (chosen).** Extract `countMethodLines(...)` from the test class body to a top-level `internal fun` in a NEW file `android/plugin/src/test/kotlin/com/godotengine/godot_solana_sdk/mwa/plugin/LocCountUtil.kt`. Both 3-1's `MwaAndroidPluginSignMessagesTest` and 3-2's `MwaAndroidPluginSignTransactionsTest` import + call the same shared helper. Story 3-3 (`sign_and_send` AC-2 LOC budget) inherits the same shared helper without further refactor. Refactor cost amortizes across two future stories.
2. **Duplicate.** Copy the helper verbatim into `MwaAndroidPluginSignTransactionsTest.kt`. Cleaner story scope (no 3-1 source touch) but introduces duplication that 3-3 inherits + a future deduplication CR.

**Decision: Option 1.** The Story 3-1 source touch is bounded (T1 task scope, NOT a Story 3-1 amendment) and is logged as **D-3-2-3 (Rule 1 — extract `countMethodLines` to shared `LocCountUtil.kt` for cross-file reuse)** at T7 close-out. The 3-1 test class loses 1 helper definition + gains 1 import; behavioral semantics unchanged. T1 must execute the extract BEFORE writing the 3-2 AC-2 test (otherwise `MwaAndroidPluginSignTransactionsTest.kt` fails to compile against the unresolved symbol).

### DD-3-2-3 [LOCKED] — Success-payload builder: parameterize `buildSignSuccessJson` with payload-key string
The 3-1-introduced `buildSignSuccessJson(requestId, signedPayloads): JSONObject` (`GDExtensionAndroidPlugin.kt:1721-1736`) hardcodes the JSON key `"signed_payloads"`. AC-1 requires the 3-2 wire-format key be `"signed_transactions"` (per the verbatim AC text). Three options were considered:

1. **Parameterize the existing helper** (selected) — change signature to `buildSignSuccessJson(requestId, signedPayloads, payloadKey: String = "signed_payloads"): JSONObject` and call with `payloadKey = "signed_transactions"` from the new `handleSignTransactionsSuccess` helper. ≤3 LOC delta to the helper; preserves the 3-1 default for `mwaSignMessages` (no change at 3-1's call site).
2. Add a sibling `buildSignTransactionsSuccessJson(requestId, signedPayloads): JSONObject`. REJECTED: violates the 3-1 DD-3-1-5 directive "do NOT introduce a sibling builder".
3. Introduce a `SignSchema` enum to pick the key. REJECTED: over-abstracted for one-line case-discrimination.

**DD-3-2-3 LOCKS option 1.** T2 changes the helper signature with a default parameter (Kotlin idiom; no breaking call-site impact). Story 3-3 will inherit the parameterized form (its key is `"signatures"` per arch §3 sign_and_send_completed signal — different from both `signed_payloads` and `signed_transactions`). Logged as **D-3-2-1 (Rule 1 — helper parameterized for downstream-compatible reuse)** at T7.

### DD-3-2-4 [LOCKED] — DD-4-3-1.a fail-closed wrapper does NOT apply to the signing family
DD-4-3-1.a (Story 4-3) wraps SecureTokenStore disk reads with `withStorageOrReauthRequired`. The signing family does NOT touch disk on the happy path:
- `sessionState.getAuthToken()` — in-memory `SecretString` (set at connect/reauth via `setAuthToken`).
- `sessionState.getConnectedKey()` — in-memory `ByteArray?`.
- `sessionState.getIdentity*()` — in-memory `String?`.

The wrapper is therefore inapplicable. `mwaSignMessages` confirms this with NO wrapper at `GDExtensionAndroidPlugin.kt:897-911`. `mwaSignTransactions` MUST mirror — do NOT introduce the wrapper for the signing family in 3-2. CR-3-2-W (§Concerns) flags the broader question "should the signing family ever defensively touch the store for corruption-detection?" for a future hardening story; NOT a 3-2 deliverable.

### DD-3-2-5 [LOCKED] — All Story 3-1 LOCKED DDs (DD-3-1-1 .. DD-3-1-10) inherited unchanged
> Ref: `docs/stories/3-1.md#Architecture Guardrails` — DD-3-1-1 (`runSigningOp` private member), DD-3-1-2 (block receiver = `MwaClient`), DD-3-1-3 (`timeoutMs: Long` param), DD-3-1-4 (`SigningOp` enum), DD-3-1-5 (parameterized via DD-3-2-3), DD-3-1-6 (NOT_CONNECTED preflight BEFORE register), DD-3-1-7 (contract test = unit test, not separate tier), DD-3-1-8 (ConnectionIdentity from sessionState; auth-token from sessionState — NO disk I/O), DD-3-1-9 (call-site does sessionState reads; helper does CAS+watchdog+errors), DD-3-1-10 (kdoc cites DD-3-2-* + plan.md line numbers + arch §X).
